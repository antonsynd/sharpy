## Identified Issues

### Issue 1: Private Field Handling in GenerateField()

**Location:** `RoslynEmitter.cs:1192-1235`

**Problem:** The `GenerateField` method does NOT check for private fields (underscore prefix). It:
1. Always uses `ToPascalCase()` (line 1195)
2. Always applies `public` modifier (line 1226)

**Expected Behavior per Spec:**
- `_snake_case` fields → `private` modifier + `_camelCase` name
- `user_count` → `public int UserCount`
- `_user_count` → `private int _userCount`

**Current Behavior:**
- `_user_count` → `public int _UserCount` (WRONG!)

### Issue 2: NameContext.Field Uses camelCase Not PascalCase

**Location:** `NameMangler.cs:197`

The `Transform()` method maps `NameContext.Field` to `ToCamelCase()`:
```csharp
NameContext.Field => ToCamelCase(name),
```

But the task spec and test cases show:
- Public fields should use PascalCase: `user_count` → `UserCount`
- Private fields should use _camelCase: `_user_count` → `_userCount`

**However**, `RoslynEmitter.GenerateField()` uses `ToPascalCase()` directly, which is correct for public fields but doesn't handle the private/public distinction.

### Issue 3: Private Method Name Mangling

**Location:** `NameMangler.cs:106-127`

`ToPascalCase()` handles private prefix correctly for methods:
```csharp
// _private_method → _PrivateMethod
```

But the emitter needs to also set the access modifier based on the prefix.

## Potential Risks and Questions

### Risks

1. **Breaking change potential:** Changing field generation could affect existing compiled code that relies on current behavior. However, current behavior appears incorrect per spec.

2. **Decorator conflict:** If a field has both `_` prefix AND a `@public` decorator, which wins? (Need to check decorator handling)

3. **Consistency with methods:** Private methods (`_method`) should also get `private` modifier - verify this is handled.

### Questions for Clarification

1. **Constant fields:** Should `_PRIVATE_CONST` become `private const int _PrivateConst` or preserve CAPS?
   - Current spec suggests: `CAPS_SNAKE_CASE` → `PascalCase` for constants
   - But private prefix should still trigger `private` modifier

2. **Double underscore fields:** Python-style name mangling (`__field`) - is this supported?
   - Current code: `__name` without trailing `__` is kept as private prefix
   - May need semantic error if user tries `__field` (not a dunder)

3. **Type name preservation:** The spec says class names are preserved (`UserProfile` stays `UserProfile`), but the task table shows `user_profile` → `UserProfile`. Which is correct?
   - Looking at `ToTypeName()`: It preserves the name as-is (only keyword escaping)
   - This means `user_profile` class would stay `user_profile` in C# (lowercase)
   - This may be intentional to allow Python-style class names

The NameMangler implementation is **mostly complete and correct**. The main issue is in `RoslynEmitter.GenerateField()` which:
1. Does not detect private fields via underscore prefix
2. Does not set `private` access modifier for `_` prefixed fields
3. Uses PascalCase for all fields instead of camelCase for private fields
