## Summary: Fixed Multiple Entry Points Issue ✓

### Problem
When compiling multiple Sharpy files, each file with a `main()` function would generate a C# `Main()` method, causing the C# compiler to fail with "multiple entry points" error.

### Root Cause
In `src/Sharpy.Compiler/CodeGen/RoslynEmitter.cs:582`, the `GenerateFunctionDeclaration` method was unconditionally converting all Sharpy `main` functions to C# `Main` methods using `NameMangler.Transform(func.Name, NameContext.Method)`, which applies PascalCase conversion (`main` → `Main`).

This happened regardless of whether the file was designated as the entry point.

### Solution Implemented

**File Modified:** `src/Sharpy.Compiler/CodeGen/RoslynEmitter.cs:583-585`

Changed the name mangling logic to check `_context.IsEntryPoint` before converting `main` to `Main`:

```csharp
// Transform name using NameMangler
// Special case: only convert "main" to "Main" if this is the entry point file
var mangledName = func.Name == "main" && !_context.IsEntryPoint
    ? "MainFunc"  // Rename to avoid C# entry point conflict in non-entry files
    : NameMangler.Transform(func.Name, NameContext.Method);
```

**How it works:**
- **Entry point file** (`_context.IsEntryPoint == true`): `main()` → `Main()` ✓
- **Non-entry files** (`_context.IsEntryPoint == false`): `main()` → `MainFunc()` ✓

### Entry Point Determination Logic

The entry point is determined in `src/Sharpy.Compiler/Project/ProjectCompiler.cs:401-420`:

1. If `ProjectConfig.EntryPoint` is specified, that file is the entry point
2. Otherwise, defaults to `main.spy` for executable projects

This context is passed to `RoslynEmitter` via `CodeGenContext.IsEntryPoint` (line 427).
