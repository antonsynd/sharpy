# Sharpy v0.5 Implementation Verification Guide

This document provides detailed implementation verification instructions for Sharpy v0.5 features. Each feature section describes exactly what syntax and semantics are allowed, what restrictions apply, and how to verify correctness through the entire compilation pipeline (lexing → parsing → semantic analysis → code generation → testing).

---

## Feature 1.1: UTF-8 Encoding Support

### Feature Description
Sharpy source files must support UTF-8 encoding as specified in the language reference. This is the foundational encoding requirement for all `.spy` files.

### Allowed Syntax/Semantics

**Valid UTF-8 encoded files:**
```python
# ASCII characters (subset of UTF-8)
x = 42
name = "Alice"

# UTF-8 multibyte characters in strings
message = "Hello, 世界"
greeting = "Привет, мир"

# UTF-8 multibyte characters in comments
# This is a comment with emoji: 🚀 📝 ✨
# Unicode characters: ñ, é, ü, ö

# UTF-8 identifiers (allowed but discouraged for interop)
переменная = 100
変数 = 200
```

**Optional BOM handling:**
```
Files may start with UTF-8 BOM (EF BB BF), but it should be optional
Files without BOM must be accepted
```

### Disallowed Syntax/Semantics

**Invalid encodings:**
- UTF-16 encoded files (must reject or error)
- UTF-32 encoded files (must reject or error)
- Latin-1 / ISO-8859-1 encoded files (must reject or error)
- Files with invalid UTF-8 byte sequences (must error)

**BOM requirements:**
- BOM must NOT be required (files without BOM are valid)
- Non-UTF-8 BOMs (UTF-16 LE/BE, UTF-32 LE/BE) must be rejected

### Lexer Verification

**Test cases:**
1. **ASCII-only file** - Verify tokens generated correctly
2. **UTF-8 multibyte in strings** - Verify string literals preserve UTF-8 content
3. **UTF-8 multibyte in comments** - Verify comments handled correctly
4. **UTF-8 identifiers** - Verify identifiers with Unicode letters tokenize
5. **File with UTF-8 BOM** - Verify BOM is ignored/stripped
6. **File without BOM** - Verify file processes normally
7. **Invalid UTF-8 sequences** - Verify lexer errors appropriately
8. **Mixed ASCII and multibyte** - Verify correct tokenization

**What to check:**
- Lexer reads bytes as UTF-8 and produces correct character sequences
- BOM bytes (EF BB BF) at start are skipped/ignored
- Invalid UTF-8 byte sequences produce lexer error with clear message
- Column/position tracking accounts for multibyte characters correctly
- String literals preserve exact UTF-8 byte sequences

### Parser Verification

**Test cases:**
1. **Parse file with UTF-8 strings** - Verify AST contains correct string values
2. **Parse file with UTF-8 comments** - Verify comments don't affect AST
3. **Parse file with UTF-8 identifiers** - Verify identifiers in AST match source
4. **Parse file with BOM** - Verify AST identical to file without BOM

**What to check:**
- AST nodes contain correct string values (UTF-8 preserved)
- Identifier nodes contain correct Unicode identifiers
- Position/location information is accurate for multibyte characters
- No differences in AST between BOM/no-BOM files (except metadata)

### Semantic Analysis Verification

**Test cases:**
1. **Type check with UTF-8 strings** - Verify string type inference
2. **UTF-8 identifiers in scope** - Verify variable resolution works
3. **UTF-8 in error messages** - Verify errors display correctly

**What to check:**
- String literals with UTF-8 inferred as `str` type
- Variable names with Unicode characters resolve correctly in scope
- Error messages display UTF-8 characters correctly (no mojibake)
- Symbol table handles UTF-8 identifiers correctly

### Code Generation Verification

**Test cases:**
1. **Generate C# with UTF-8 strings** - Verify string literals in output
2. **Generate C# with UTF-8 identifiers** - Verify identifier mapping
3. **Round-trip UTF-8 content** - Compile and run, verify output

**What to check:**
- Generated C# file is UTF-8 encoded
- String literals preserve UTF-8 content exactly
- Unicode identifiers transformed appropriately (snake_case → PascalCase)
- Generated code compiles with C# compiler
- Runtime string values match source literals

### Test Requirements

**Minimum test coverage:**
1. Unit test: Lexer with ASCII-only input
2. Unit test: Lexer with UTF-8 multibyte in strings
3. Unit test: Lexer with UTF-8 in comments
4. Unit test: Lexer with UTF-8 identifiers
5. Unit test: Lexer with BOM
6. Unit test: Lexer with invalid UTF-8 (expect error)
7. Integration test: End-to-end compilation with UTF-8 strings
8. Integration test: Run compiled code, verify UTF-8 output matches

**Test data examples:**
```python
# Test file: test_utf8_basic.spy
message = "Hello, 世界"
assert message == "Hello, 世界"

# Test file: test_utf8_identifiers.spy
переменная = 42
assert переменная == 42

# Test file: test_utf8_bom.spy
# (Same content as test_utf8_basic.spy but with BOM)

# Test file: test_utf8_invalid.spy
# (File with invalid UTF-8 byte sequence 0xFF 0xFE)
```

### Common Issues to Check

**Lexer level:**
- Byte Order Mark not properly skipped
- Invalid UTF-8 sequences cause crash instead of error
- Column numbers incorrect for multibyte characters
- String escapes interact incorrectly with multibyte

**Parser level:**
- String literal AST nodes contain wrong encoding
- Position tracking breaks with multibyte characters

**Semantic level:**
- Error messages garbled when displaying UTF-8
- Symbol table hash collisions with Unicode identifiers

**Codegen level:**
- Generated C# file not UTF-8 encoded
- String literals incorrectly escaped or encoded
- Unicode identifiers cause invalid C# identifiers

## Feature 1.2.1: Exactly 4 Spaces Per Indentation Level (Enforced)

### Feature Description
Sharpy uses indentation to denote code blocks. Each indentation level must be exactly 4 spaces. This is strictly enforced - no other indentation size is allowed.

### Allowed Syntax/Semantics

**Valid indentation (4 spaces per level):**
```python
# Level 0 (no indentation)
def function():
    # Level 1 (4 spaces)
    if condition:
        # Level 2 (8 spaces)
        statement1
        statement2
        if nested:
            # Level 3 (12 spaces)
            statement3
    else:
        # Level 2 (8 spaces)
        statement4

# Level 0
class MyClass:
    # Level 1 (4 spaces)
    field: int

    # Level 1 (4 spaces)
    def method(self):
        # Level 2 (8 spaces)
        for i in range(10):
            # Level 3 (12 spaces)
            print(i)

# Level 0
if x > 0:
    # Level 1 (4 spaces)
    y = x * 2
```

**Blank lines (any amount of whitespace, or none):**
```python
def function():
    statement1

    statement2
```

**Comments at any indentation:**
```python
# Comment at level 0
def function():
    # Comment at level 1
    statement
        # Comment at arbitrary indentation (allowed in comments)
```

### Disallowed Syntax/Semantics

**Wrong number of spaces:**
```python
# ERROR: 2 spaces (must be 4)
def function():
  statement

# ERROR: 3 spaces (must be 4)
def function():
   statement

# ERROR: 5 spaces (must be 4)
def function():
     statement

# ERROR: 8 spaces for first level (must be 4)
def function():
        statement
```

**Tabs for indentation:**
```python
# ERROR: Tab character for indentation
def function():
	statement  # Tab character here

# ERROR: Mixed tabs and spaces
def function():
    if condition:
	statement  # Tab character here
```

**Mixed spaces and tabs:**
```python
# ERROR: Some lines with spaces, some with tabs
def function():
    statement1  # 4 spaces
	statement2  # 1 tab
```

**Inconsistent indentation decrease:**
```python
# ERROR: Indentation decreases to non-previous level
def function():
    if condition:
        statement1
      statement2  # ERROR: 6 spaces, not previous level (4 or 8)
```

**Indentation not multiple of 4:**
```python
# ERROR: Indentation is 6 spaces (not multiple of 4)
def function():
    if condition:
      statement  # Only 2 additional spaces
```

### Lexer Verification

**Test cases:**
1. **Valid 4-space indentation** - Generates INDENT token
2. **Valid 8-space indentation (level 2)** - Generates 2 INDENT tokens
3. **Valid dedentation to level 0** - Generates DEDENT token(s)
4. **Tab character in indentation** - Lexer error
5. **2-space indentation** - Lexer error
6. **3-space indentation** - Lexer error
7. **Mixed spaces/tabs** - Lexer error
8. **Indentation to non-previous level** - Lexer error
9. **Blank lines** - No INDENT/DEDENT generated
10. **EOF with indentation** - Generates DEDENT tokens to level 0

**What to check:**
- INDENT token generated when indentation increases by exactly 4 spaces
- DEDENT token(s) generated when indentation decreases
- Indentation level tracked correctly (must be multiple of 4)
- Tab characters in leading whitespace cause immediate error
- Mixed tabs/spaces cause immediate error
- Non-4-space increases cause error with clear message
- Dedent to non-previous level causes error
- Indentation stack maintained correctly
- End-of-file generates enough DEDENT tokens to reach level 0

**Token sequence example:**
```python
# Source:
def f():
    x = 1

# Expected tokens:
DEF, IDENTIFIER("f"), LPAREN, RPAREN, COLON, NEWLINE,
INDENT, IDENTIFIER("x"), EQUALS, INTEGER(1), NEWLINE,
DEDENT, EOF
```

### Parser Verification

**Test cases:**
1. **Parse nested blocks** - Verify correct AST structure
2. **Parse multiple dedent levels** - Verify AST closes blocks correctly
3. **Parse blank lines in blocks** - Verify blank lines ignored
4. **Parse indentation error** - Verify parser reports or propagates error

**What to check:**
- Indented blocks create correct nested structure in AST
- INDENT increases block nesting depth
- DEDENT closes blocks at correct nesting level
- Multiple consecutive DEDENT tokens close multiple blocks
- Blank lines (no INDENT/DEDENT) don't affect block structure
- Parser doesn't crash on indentation errors from lexer

**AST structure example:**
```python
# Source:
if x:
    y = 1
    if z:
        w = 2
    v = 3

# AST (simplified):
IfStatement(
    condition=Identifier("x"),
    body=[
        Assignment(Identifier("y"), Integer(1)),
        IfStatement(
            condition=Identifier("z"),
            body=[
                Assignment(Identifier("w"), Integer(2))
            ]
        ),
        Assignment(Identifier("v"), Integer(3))
    ]
)
```

### Semantic Analysis Verification

**Test cases:**
1. **Scope nesting from indentation** - Verify scopes created correctly
2. **Variable shadowing in nested blocks** - Verify scope resolution
3. **Variable access across indentation levels** - Verify scope lookup

**What to check:**
- Each indented block creates new scope
- Variables declared in outer block accessible in inner block
- Variables declared in inner block not accessible in outer block
- Scope hierarchy matches indentation structure
- No semantic analysis performed on syntactically invalid indentation

### Code Generation Verification

**Test cases:**
1. **Generate C# braces for indentation** - Verify `{` and `}` placement
2. **Nested blocks** - Verify correct brace nesting
3. **Empty blocks** - Verify C# doesn't generate empty blocks (or uses valid syntax)

**What to check:**
- Each Sharpy indented block becomes `{ ... }` in C#
- Brace matching is correct
- Indentation in generated C# is consistent (cosmetic but helpful)
- Generated code compiles with C# compiler
- Runtime behavior preserves block semantics

**Code generation example:**
```python
# Sharpy:
if x:
    y = 1

# Generated C# (simplified):
if (x)
{
    y = 1;
}
```

### Test Requirements

**Minimum test coverage:**
1. Unit test: Lexer with valid 4-space indent
2. Unit test: Lexer with valid 8-space indent (level 2)
3. Unit test: Lexer with valid dedent
4. Unit test: Lexer with tab indent (expect error)
5. Unit test: Lexer with 2-space indent (expect error)
6. Unit test: Lexer with mixed tabs/spaces (expect error)
7. Unit test: Lexer with dedent to non-previous level (expect error)
8. Unit test: Parser with nested blocks
9. Unit test: Parser with multiple dedent levels
10. Integration test: Compile and run program with nested blocks

**Test data examples:**
```python
# Test file: test_indent_valid.spy
def function():
    if True:
        x = 1
        if True:
            y = 2
        z = 3
    return x

# Test file: test_indent_tab_error.spy
def function():
	x = 1  # Tab character - should error

# Test file: test_indent_2space_error.spy
def function():
  x = 1  # 2 spaces - should error

# Test file: test_indent_nonprevious_error.spy
def function():
    if True:
        x = 1
      y = 2  # 6 spaces, not previous level - should error
```

### Common Issues to Check

**Lexer level:**
- Indentation stack not maintained correctly
- Tab characters not detected (checking only spaces)
- Blank lines incorrectly generate INDENT/DEDENT
- EOF doesn't generate DEDENT tokens to level 0
- Error messages unclear about indentation requirements
- Indentation inside strings or comments affects indent tracking

**Parser level:**
- INDENT token doesn't start new block
- DEDENT token doesn't close block correctly
- Multiple DEDENT tokens handled incorrectly
- Parser crashes instead of reporting indentation errors

**Semantic level:**
- Scopes not created for indented blocks
- Scope nesting doesn't match indentation structure

**Codegen level:**
- Missing braces for indented blocks
- Mismatched braces causing C# compilation error
- Empty blocks generate invalid C# syntax

## Feature 1.2.2: Tabs Rejected for Indentation

### Feature Description
Tab characters (`\t`) are not allowed for indentation in Sharpy. Any leading tab character must cause a lexical error. This enforces consistency and prevents mixed indentation issues.

### Allowed Syntax/Semantics

**Spaces for indentation (valid):**
```python
def function():
    statement  # 4 spaces

if condition:
    if nested:
        statement  # 8 spaces
```

**Tabs in string literals (valid):**
```python
# Tab inside string is allowed
message = "Name:\tValue"
multiline = """
Line 1
\tIndented line
"""
```

**Tabs in comments (valid, though discouraged):**
```python
# Comment with	tab character (allowed but discouraged)
def function():
    statement
    # Another	comment	with	tabs
```

**Tabs as non-leading whitespace (valid):**
```python
x = 1	# Tab after statement (allowed but discouraged)
y = 2   	  # Trailing whitespace with tab (allowed)
```

### Disallowed Syntax/Semantics

**Any tab in leading whitespace:**
```python
# ERROR: Tab at start of line
	statement

# ERROR: Tab for indentation
def function():
	statement

# ERROR: Tab mixed with spaces for indentation
def function():
    	statement  # Spaces then tab

# ERROR: Multiple tabs for indentation
def function():
		statement

# ERROR: Tab then spaces
def function():
	    statement  # Tab then spaces
```

**Mixed tabs and spaces for indentation:**
```python
# ERROR: Different lines use different indentation
def function():
    line1  # 4 spaces
	line2  # 1 tab
```

### Lexer Verification

**Test cases:**
1. **Leading tab character** - Lexer error
2. **Tab in middle of indentation** - Lexer error
3. **Tab after spaces in indentation** - Lexer error
4. **Tab in string literal** - Valid, no error
5. **Tab in comment** - Valid, no error
6. **Tab after statement (trailing)** - Valid, no error
7. **Mixed tabs/spaces across lines** - Lexer error on first tab line
8. **Blank line with only tab** - Should be treated as blank line or error

**What to check:**
- Lexer scans leading whitespace character by character
- Any tab character in leading whitespace triggers immediate error
- Error message clearly indicates tab character not allowed
- Error message shows line and column of tab character
- Tabs in strings don't trigger indentation error
- Tabs in comments don't trigger indentation error
- Tabs after code on same line don't trigger error (trailing whitespace)

**Error message example:**
```
Error: Tab character not allowed for indentation
  at line 3, column 1
  def function():
      statement
  ^
  Use 4 spaces for indentation instead of tabs
```

### Parser Verification

**Test cases:**
1. **Parse after tab error** - Verify parser handles lexer error gracefully
2. **No AST generation on tab error** - Verify parsing stops or skips

**What to check:**
- Parser receives error from lexer when tab encountered
- Parser doesn't attempt to continue parsing after tab error (or gracefully recovers)
- No partial AST generated for invalid indentation
- Error propagated to user clearly

### Semantic Analysis Verification

**Test cases:**
1. **No semantic analysis on tab error** - Verify analysis skipped

**What to check:**
- Semantic analysis doesn't run on files with tab errors
- Or, if error recovery is implemented, semantic analysis handles missing AST nodes

### Code Generation Verification

**Test cases:**
1. **No code generation on tab error** - Verify codegen skipped

**What to check:**
- Code generation doesn't run on files with tab errors
- No C# output produced for invalid input

### Test Requirements

**Minimum test coverage:**
1. Unit test: Lexer with leading tab (expect error)
2. Unit test: Lexer with tab in indentation (expect error)
3. Unit test: Lexer with spaces then tab in indentation (expect error)
4. Unit test: Lexer with tab in string literal (valid)
5. Unit test: Lexer with tab in comment (valid)
6. Unit test: Lexer with trailing tab (valid)
7. Unit test: Parser with tab error (verify graceful handling)
8. Integration test: Attempt to compile file with tab (expect error, no output)

**Test data examples:**
```python
# Test file: test_tab_leading_error.spy
	x = 1  # Tab at start

# Test file: test_tab_indent_error.spy
def function():
	statement  # Tab for indentation

# Test file: test_tab_string_valid.spy
message = "Value:\tData"  # Tab in string is valid

# Test file: test_tab_comment_valid.spy
# Comment	with	tabs  # Valid (though discouraged)
x = 1

# Test file: test_tab_trailing_valid.spy
x = 1	# Tab after code (valid)
```

### Common Issues to Check

**Lexer level:**
- Tab characters not detected (only checking for spaces count)
- Tabs in strings incorrectly trigger indentation error
- Tabs in comments incorrectly trigger indentation error
- Error message doesn't clearly indicate tab vs. spaces issue
- Column number in error incorrect for tab character
- Tab detection only happens at certain positions (not all leading whitespace)

**Parser level:**
- Parser crashes on tab error instead of graceful handling
- Parser tries to continue with invalid indentation state

**Error reporting:**
- Error message doesn't show which line has tab
- Error message doesn't suggest using spaces instead
- Error doesn't show visible indication of tab character

## Feature 1.2.3: Mixed Spaces and Tabs Cause Lexical Error

### Feature Description
If a source file mixes spaces and tabs for indentation across different lines, this must cause a lexical error. Consistency is enforced: all indentation must use spaces only (no tabs anywhere in indentation).

### Allowed Syntax/Semantics

**Consistent spacing (all spaces):**
```python
# All lines use spaces for indentation - valid
def function():
    line1
    line2
    if condition:
        line3
        line4
    line5
```

**No indentation:**
```python
# No indentation at all - valid
x = 1
y = 2
z = 3
```

### Disallowed Syntax/Semantics

**Mixed tabs and spaces across lines:**
```python
# ERROR: Line 2 uses spaces, line 3 uses tab
def function():
    line1  # 4 spaces
	line2  # 1 tab - ERROR

# ERROR: First indented line uses tab, second uses spaces
if condition:
	line1  # 1 tab
    line2  # 4 spaces - ERROR (inconsistent with previous tab)
```

**Mixed within same indentation block:**
```python
# ERROR: Function body mixes tabs and spaces
def function():
    if True:
	statement1  # Tab
    statement2  # Spaces - ERROR
```

**Spaces then tabs on same line:**
```python
# ERROR: Spaces followed by tab in leading whitespace
def function():
    	statement  # Spaces + tab in indentation
```

### Lexer Verification

**Test cases:**
1. **All spaces (consistent)** - Valid, no error
2. **All tabs** - Error on first tab (tabs rejected entirely)
3. **First line spaces, second line tab** - Error on second line
4. **First line tab, second line spaces** - Error on first line (tab rejected)
5. **Spaces then tab in same line indentation** - Error
6. **Multiple files mixed (if batch processing)** - Each file independent

**What to check:**
- Lexer tracks indentation method (spaces only allowed)
- Any tab in indentation triggers error immediately
- Error message indicates mixing/tab usage
- Indentation tracking is per-file (not global state)
- First tab encountered causes error (fail-fast)

**Error detection flow:**
```
1. Lexer encounters leading whitespace
2. For each character in leading whitespace:
   a. If space: increment space count
   b. If tab: IMMEDIATE ERROR - tab not allowed
3. If only spaces: validate count is multiple of 4
4. Generate INDENT/DEDENT as appropriate
```

### Parser Verification

**Test cases:**
1. **Parse file with mixed spaces/tabs error** - Verify error propagated
2. **No AST on mixed indentation** - Verify parsing halted

**What to check:**
- Parser receives error from lexer
- Parsing stops or gracefully handles error
- No AST generated for invalid indentation

### Semantic Analysis Verification

**Test cases:**
1. **No analysis on mixed indentation error** - Verify skipped

**What to check:**
- Semantic analysis doesn't run on files with indentation errors

### Code Generation Verification

**Test cases:**
1. **No codegen on mixed indentation error** - Verify skipped

**What to check:**
- No C# code generated for files with indentation errors

### Test Requirements

**Minimum test coverage:**
1. Unit test: Lexer with all spaces (valid)
2. Unit test: Lexer with spaces on line 1, tab on line 2 (error)
3. Unit test: Lexer with tab on line 1 (error immediately)
4. Unit test: Lexer with spaces+tab in same indentation (error)
5. Unit test: Multiple files processed independently
6. Integration test: Compile attempt with mixed indentation (expect error)

**Test data examples:**
```python
# Test file: test_mixed_1_error.spy
def function():
    line1  # Spaces
	line2  # Tab - ERROR

# Test file: test_mixed_2_error.spy
def function():
	line1  # Tab - ERROR (tabs not allowed at all)
    line2  # Spaces

# Test file: test_all_spaces_valid.spy
def function():
    line1
    line2
    line3
```

### Common Issues to Check

**Lexer level:**
- Mixed indentation not detected (only checking individual lines)
- Tab allowed if it happens to align with space count
- Error not triggered on first tab occurrence
- Indentation state leaks between files in batch processing
- Error message doesn't indicate which line uses tab vs. spaces

**Parser level:**
- Parser attempts to continue after mixed indentation error
- Inconsistent error recovery

**Error reporting:**
- Error doesn't show both the tab line and space line for comparison
- Error message unclear about tabs vs. spaces vs. mixing

## Feature 1.3.1: Single-line Comments Starting with `#`

### Feature Description
Sharpy supports single-line comments that begin with the `#` character and continue to the end of the line. Comments are ignored by the lexer and do not produce tokens.

### Allowed Syntax/Semantics

**Comment at start of line:**
```python
# This is a comment
# This is another comment
```

**Comment after code:**
```python
x = 42  # Inline comment after statement
y = 10 + 20  # Comment after expression
```

**Comment with various content:**
```python
# Comment with numbers 123456
# Comment with symbols !@#$%^&*()
# Comment with unicode 世界 🚀
# Comment with URL https://example.com
```

**Multiple consecutive comment lines:**
```python
# Comment line 1
# Comment line 2
# Comment line 3
```

**Empty comment (just `#`):**
```python
#
# Also valid
x = 1  #
```

**Comments at any indentation:**
```python
def function():
    # Comment at level 1
    if True:
        # Comment at level 2
        pass
# Comment at level 0
```

**Comment in various contexts:**
```python
# Before import
import math

# Before function
def function():
    # Inside function
    pass

# Before class
class MyClass:
    # Inside class
    field: int
```

### Disallowed Syntax/Semantics

**`#` inside string is NOT a comment:**
```python
# This IS a comment
message = "This # is not a comment"  # But this IS a comment
text = 'String with # character'  # Comment here
```

**Multi-line comments with `#` (not a single feature):**
```python
# This is line 1 of separate comments
# This is line 2 of separate comments
# Each # starts a NEW single-line comment
```

**Block comments (not supported in v0.5):**
```python
# C-style block comments not supported:
/* This is not valid in Sharpy */

# There is no multi-line comment syntax
# You must use # on each line
```

### Lexer Verification

**Test cases:**
1. **Comment at line start** - No tokens generated, comment ignored
2. **Comment after code** - Code tokens generated, comment ignored
3. **Empty comment** - No tokens, no error
4. **Comment with special chars** - No tokens, no error
5. **Comment with unicode** - No tokens, no error
6. **`#` in string literal** - String token contains `#` character
7. **`#` in f-string** - String token contains `#` character
8. **Consecutive comment lines** - Each line ignored independently

**What to check:**
- Lexer detects `#` character outside strings
- Everything from `#` to end of line is ignored
- Comment content doesn't produce any tokens
- `#` inside string literals is preserved as part of string
- Comments removed before tokenization (don't appear in token stream)
- Line numbers tracked correctly (comments don't affect line count)
- Column numbers tracked correctly (comments affect current position)

**Token stream example:**
```python
# Source:
x = 42  # This is a comment
y = 10

# Expected tokens:
IDENTIFIER("x"), EQUALS, INTEGER(42), NEWLINE,
IDENTIFIER("y"), EQUALS, INTEGER(10), NEWLINE
# No tokens for comments
```

### Parser Verification

**Test cases:**
1. **Parse file with comments** - Verify AST identical to file without comments
2. **Comments between statements** - Verify no AST nodes for comments
3. **Comments in various locations** - Verify AST structure unaffected

**What to check:**
- AST contains no comment nodes (comments completely stripped)
- AST structure identical whether comments present or not
- Docstrings (triple-quoted strings) are NOT comments (they are string literals)
- Parser never sees comment tokens (already removed by lexer)

### Semantic Analysis Verification

**Test cases:**
1. **Analysis with comments present** - Verify same results as without comments

**What to check:**
- Comments don't affect semantic analysis (already stripped)
- Type inference works same with or without comments
- Error messages correct line numbers despite comments

### Code Generation Verification

**Test cases:**
1. **Generate code from file with comments** - Comments not in output
2. **Optionally preserve comments as C# comments** - If desired, map to `//`

**What to check:**
- Sharpy comments don't appear in generated C# (or optionally converted to `//`)
- Generated code compiles correctly
- Comments don't affect runtime behavior

### Test Requirements

**Minimum test coverage:**
1. Unit test: Lexer with comment at line start
2. Unit test: Lexer with comment after code
3. Unit test: Lexer with empty comment
4. Unit test: Lexer with `#` in string (not comment)
5. Unit test: Lexer with `#` in f-string (not comment)
6. Unit test: Parser with comments (verify AST)
7. Integration test: Compile file with comments, verify output
