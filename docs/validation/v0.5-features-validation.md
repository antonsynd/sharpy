# Sharpy v0.5 Feature Validation Checklist

**Purpose:** This document provides a detailed breakdown of all v0.5 features for systematic validation. Each feature is designed to be small, semantically atomic, and complete. Engineers can use this to verify implementation correctness from lexing → parsing → codegen → testing.

**How to use this document:**
1. For each feature, check that the lexer correctly tokenizes the syntax
2. Verify the parser builds correct AST nodes
3. Ensure codegen produces correct C# output
4. Confirm sufficient test coverage exists

---

## 1. Lexical Structure

### 1.1. Source File Encoding

**What is allowed:**
- File extension: `.spy` (required)
- Encoding: UTF-8 (required)
- Line endings: LF (`\n`) or CRLF (`\r\n`)
- BOM: Optional (but not recommended)

**What is NOT allowed:**
- Non-UTF-8 encodings
- Other file extensions for Sharpy source

**Validation points:**
- [ ] Lexer accepts UTF-8 files with `.spy` extension (file handling at compiler level, not lexer)
- ✅ Lexer handles both LF and CRLF line endings (test: `Tokenize_WindowsLineEndings_HandledCorrectly`)
- [ ] Lexer accepts (but doesn't require) BOM (file handling at compiler level, not lexer)
- [ ] Lexer rejects non-UTF-8 files with clear error (file handling at compiler level, not lexer)

### 1.2. Indentation

**What is allowed:**
- Exactly 4 spaces per indentation level (enforced)
- Consistent indentation throughout file
- Indentation increases to start new block
- Indentation decreases to previous level to end block(s)

**What is NOT allowed:**
- Tabs for indentation
- Mixed spaces and tabs
- Inconsistent spacing (not multiples of 4)
- Decreasing indentation to arbitrary levels (must match a previous level)

**Validation points:**
- ✅ Lexer generates INDENT token when indentation increases by 4 spaces (test: `Tokenize_SimpleIndent_GeneratesIndentToken`)
- ✅ Lexer generates DEDENT token(s) when indentation decreases (test: `Tokenize_SimpleDedent_GeneratesDedentToken`)
- ✅ Lexer enforces exactly 4 spaces per level (test: `Tokenize_NonMultipleOf4Indentation_ThrowsLexerError`)
- ✅ Lexer rejects tabs with clear error (test: `Tokenize_TabIndentation_ThrowsLexerError`)
- ✅ Lexer rejects inconsistent indentation with clear error (test: `Tokenize_IndentationMismatch_ThrowsLexerError`)
- ✅ Lexer handles multiple DEDENT tokens at once correctly (test: `Tokenize_MultiLevelDedent_GeneratesMultipleDedentTokens`)
- ✅ Parser correctly builds nested block structures from INDENT/DEDENT (verified through complex parser tests)

**Test cases:**
```python
# Valid: proper 4-space indentation
if x:
    y = 1
    if z:
        w = 2
    v = 3

# Invalid: tabs (should error)
if x:
	y = 1  # Tab character - ERROR

# Invalid: 2 spaces (should error)
if x:
  y = 1  # 2 spaces - ERROR

# Invalid: arbitrary dedent level (should error)
if x:
    if y:
        z = 1
  w = 2  # Dedent to 2 spaces - ERROR (not a previous level)
```

### 1.3. Comments

**What is allowed:**
- Single-line comments starting with `#` to end of line
- `#` at any position on a line (including after code)
- Multiple consecutive `#` lines for multi-line comments
- Triple-quoted strings as docstrings (not comments, but serve similar purpose)

**What is NOT allowed:**
- Multi-line comment syntax like `/* */` (C-style)
- `#` inside string literals treated as comment

**Validation points:**
- ✅ Lexer ignores everything from `#` to end of line (test: `Tokenize_SingleLineComment_IsSkipped`)
- ✅ Lexer preserves `#` inside string literals as literal text (verified through string tests)
- ✅ Parser handles comments at any position (test: `Tokenize_CommentAtEndOfLine_IsSkipped`)
- [ ] Codegen has option to omit comments or emit them as C# comments (`//`) (comments are not preserved in AST)
- ✅ Triple-quoted strings are parsed as string literals, not comments (test: `Tokenize_TripleQuotedString_NotTreatedAsComment`)

**Test cases:**
```python
# This is a comment
x = 42  # Comment after code

```

### 1.4. Identifiers

**What is allowed:**
- Start with letter (a-z, A-Z) or underscore (_)
- Contain letters, digits (0-9), and underscores
- Case-sensitive
- Unicode letters allowed (but discouraged for interop)

**What is NOT allowed:**
- Starting with a digit
- Containing hyphens or other special characters
- Using keywords as identifiers (unless backtick-escaped)

**Validation points:**
- ✅ Lexer accepts valid identifiers (test: `Tokenize_ValidIdentifier_ReturnsIdentifierToken`)
- ✅ Lexer rejects identifiers starting with digits (test: `Tokenize_IdentifierStartingWithDigit_ThrowsLexerError`)
- ✅ Lexer rejects identifiers with invalid characters (test: `Tokenize_IdentifierWithHyphen_TokenizesAsThreeSeparateTokens` - hyphen treated as operator)
- ✅ Lexer treats keywords as keywords, not identifiers (test: `Tokenize_Keyword_ReturnsCorrectToken`)
- ✅ Case sensitivity is preserved (verified through identifier tests)
