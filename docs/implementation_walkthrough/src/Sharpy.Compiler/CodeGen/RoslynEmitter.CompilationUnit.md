# Walkthrough: RoslynEmitter.CompilationUnit.cs

**Source File**: `src/Sharpy.Compiler/CodeGen/RoslynEmitter.CompilationUnit.cs`

---

## 1. Overview

`RoslynEmitter.CompilationUnit.cs` is a **partial class file** that handles the outermost layer of C# code generation—transforming a Sharpy module into a complete C# compilation unit with namespaces, using directives, and the module wrapper class. This file orchestrates the "big picture" structure of generated C# files.

### Position in the Pipeline

```
Source (.spy) → Lexer → Parser (AST) → Semantic Analysis → RoslynEmitter → C#
                                                               ↓
                                                    RoslynEmitter.CompilationUnit.cs
                                                    (Module → CompilationUnit transformation)
```

**Upstream Input:**
- `Module` (from Parser) — contains import statements and module body
- `CodeGenContext` — provides project namespace, source file path, entry point flag
- `SemanticInfo` — symbol resolution for imports

**Downstream Output:**
- `CompilationUnitSyntax` — a complete C# source file with:
  - `#nullable enable` directive
  - `using` directives (System namespaces + Sharpy runtime + user imports)
  - Namespace declaration (computed from project structure)
  - Module wrapper class (generated by other partial class files)

### Key Responsibilities

- **Namespace Generation:** Compute C# namespace from file path and project structure
- **Import Translation:** Convert Python-style imports to C# `using` directives
- **Module Export Handling:** Generate re-export members for library modules
- **Symbol Tracking:** Maintain registries of imported symbols for proper casing
- **Null Safety:** Add `#nullable enable` to align with Sharpy's null-safe semantics
- **Deduplication:** Ensure using directives don't conflict or duplicate

## 3. Key Functions/Methods

### 3.1 `GenerateCompilationUnit` — Top-Level Orchestrator

```csharp
public CompilationUnitSyntax GenerateCompilationUnit(Module module)
```

**What it does:** Transforms a Sharpy module into a complete C# compilation unit with all structural components.

**Execution Flow:**

```
1. Pre-process from-import statements → Build _fromImportSymbols registry
2. Generate using directives → Convert imports to C# using statements
3. Separate imports from code → Filter module body
4. Collect re-export statements → For library modules (not entry points)
5. Generate module class → Wrap code in Exports class
6. Generate namespace → Compute from project structure
7. Build compilation unit → Assemble all parts
8. Add #nullable enable → Inject as leading trivia
```

**Example Transformation:**

```python
# Sharpy: src/myproject/utils/helpers.spy
from config import MAX_SIZE
import system.io

def process_file(path: str) -> str:
    return path.upper()
```

```csharp
// Generated C#
#nullable enable

using System;
using System.Collections.Generic;
using System.Linq;
using global::Sharpy.Core;
using System.IO;
using static MyProject.Config.Exports;

namespace MyProject.Utils.Helpers
{
    public static class Exports
    {
        public static string ProcessFile(string path)
        {
            return path.ToUpper();
        }
    }
}
```

**Key Implementation Details:**

1. **From-Import Pre-Processing (Lines 18-38):**
   ```csharp
   _fromImportSymbols.Clear();
   _importAliasToOriginal.Clear();
   foreach (var stmt in module.Body)
   {
       if (stmt is FromImportStatement fromImport)
       {
           foreach (var importedName in fromImport.Names)
           {
               // Track: from config import MAX_SIZE as MAX
               _fromImportSymbols.Add(importedName.Name);  // MAX_SIZE
               if (importedName.AsName != null)
               {
                   _fromImportSymbols.Add(importedName.AsName);  // MAX
                   _importAliasToOriginal[importedName.AsName] = importedName.Name;
                   // MAX → MAX_SIZE mapping for code generation
               }
           }
       }
   }
   ```

   **Why?** When generating code that references `MAX`, we need to know:
   - It came from a `from-import` (use exact casing, not camelCase)
   - Its original name was `MAX_SIZE` (for constant casing rules)

2. **Re-Export Detection (Lines 49-58):**
   ```csharp
   List<FromImportStatement>? fromImports = null;
   if (!_context.IsEntryPoint)  // Only library modules re-export
   {
       fromImports = module.Body
           .OfType<FromImportStatement>()
           .Where(f => f.ReExportedSymbols != null && f.ReExportedSymbols.Count > 0)
           .ToList();
   }
   ```

   **Scenario:** Package `__init__.spy` files re-export symbols from submodules:
   ```python
   # mypackage/__init__.spy
   from .helpers import process_file  # Re-exported to users of mypackage
   ```

   Generates delegating members in `mypackage.Exports` class.

3. **Null Safety Directive (Lines 75-80):**
   ```csharp
   var nullablePragma = ParseLeadingTrivia("#nullable enable\n\n");
   return compilationUnit.WithLeadingTrivia(nullablePragma);
   ```

   **Critical Detail:** Must be added **AFTER** `NormalizeWhitespace()` to preserve leading position. If added before, normalization would strip it.

   **Rationale:** Aligns with Sharpy Axiom 3 ("null-safe by default") — all C# reference types are nullable-aware.

### 3.2 `GenerateNamespaceName` — Namespace Strategy Selection

```csharp
private NameSyntax GenerateNamespaceName()
```

**What it does:** Chooses the appropriate namespace generation strategy based on available context information.

**Decision Tree:**

```
Is ProjectNamespace + ProjectRootPath + SourceFilePath all set?
  ├─ YES → GenerateProjectNamespace() (multi-file project)
  └─ NO
      ├─ Is ProjectNamespace set?
      │   ├─ YES → "{ProjectNamespace}.{FileName}"
      │   └─ NO
      │       ├─ Is SourceFilePath set?
      │       │   ├─ YES → "Sharpy.{FileName}"
      │       │   └─ NO → "SharpyGenerated"
```

**Example Outputs:**

| Context | Input | Output Namespace |
|---------|-------|------------------|
| Multi-file project | `src/myproject/lib/math/stats.spy` | `MyProject.Lib.Math.Stats` |
| Single-file with namespace | `script.spy` (ProjectNamespace = "Tools") | `Tools.Script` |
| Single-file, no namespace | `script.spy` | `Sharpy.Script` |
| REPL mode | (no file path) | `SharpyGenerated` |

**Why Simple Fallback for Single-File?**

Comment on lines 109-119 explains:
> "For single-file compilation without a project, use a simple namespace based on just the file name to avoid problematic paths (numeric directories, etc.)"

**Problem:** File paths like `/tmp/123/test.spy` would generate `Sharpy._123.Test` (invalid C# identifier).

**Solution:** Use only the file name for single-file compilation.

### 3.3 `GenerateProjectNamespace` — Multi-File Project Namespaces

```csharp
private NameSyntax GenerateProjectNamespace()
```

**What it does:** Generates hierarchical namespaces for multi-file projects based on directory structure.

**Algorithm:**

```
1. Start with ProjectNamespace (e.g., "MyProject")
2. Get relative path from ProjectRootPath to SourceFilePath
3. Extract directory components (excluding filename)
4. Convert each directory to PascalCase
5. Add filename as final component (UNLESS it's __init__.spy)
6. Join with '.'
```

**Example:**

```
ProjectNamespace: "MyCompany.MyProduct"
ProjectRootPath:  "/code/myproduct/src"
SourceFilePath:   "/code/myproduct/src/lib/data/parsers.spy"

Computation:
1. relativePath = "lib/data/parsers.spy"
2. relativeDir = "lib/data"
3. dirParts = ["lib", "data"] → ["Lib", "Data"]
4. fileName = "parsers" → "Parsers"
5. Result: "MyCompany.MyProduct.Lib.Data.Parsers"
```

**Special Case: `__init__.spy` Files (Lines 150-154):**

```csharp
if (!string.IsNullOrEmpty(fileName) && fileName != "__init__")
{
    namespaceParts.Add(SimpleToPascalCase(fileName));
}
```

**Why?** Python packages use `__init__.py` to represent the package itself, not a submodule.

```
Directory structure:     Sharpy namespace:
lib/
  data/
    __init__.spy    →   MyProject.Lib.Data (not MyProject.Lib.Data.Init)
    parsers.spy     →   MyProject.Lib.Data.Parsers
```

**Name Sanitization:**

Uses `SimpleToPascalCase()` (from NameMangler) to handle:
- `snake_case` → `PascalCase`
- `.NET` acronyms (io → IO, xml → XML)

### 3.4 `GenerateUsingDirectives` — Import Orchestration

```csharp
private List<UsingDirectiveSyntax> GenerateUsingDirectives(Module module)
```

**What it does:** Collects all using directives from default namespaces and import statements, then deduplicates.

**Default Usings (Always Added):**

```csharp
usings.Add(UsingDirective(ParseName("System")));
usings.Add(UsingDirective(ParseName("System.Collections.Generic")));
usings.Add(UsingDirective(ParseName("System.Linq")));
usings.Add(UsingDirective(ParseName("global::Sharpy.Core")));
```

**Why `global::Sharpy.Core`?** (Line 168 comment)

> "Use global:: to avoid conflicts when output namespace contains 'Sharpy'"

**Problem Scenario:**
```csharp
namespace Sharpy.MyModule  // Collision!
{
    using Sharpy.Core;  // ERROR: Ambiguous reference
}
```

**Solution:**
```csharp
namespace Sharpy.MyModule
{
    using global::Sharpy.Core;  // OK: Absolute reference
}
```

**Deduplication (Lines 184-195):**

```csharp
var seen = new HashSet<string>();
var dedupedUsings = new List<UsingDirectiveSyntax>();
foreach (var u in usings)
{
    var key = u.NormalizeWhitespace().ToFullString();
    if (seen.Add(key))  // Returns true if newly added
    {
        dedupedUsings.Add(u);
    }
}
```

**Why Normalize for Key?** Different whitespace/trivia can produce semantically identical usings:
```csharp
using System.IO;      // Key: "using System.IO;"
using   System.IO  ;  // Key: "using System.IO;" (same after normalization)
```

### 3.5 `GenerateImportUsings` — Standard Import Translation


**What it does:** Converts Python-style `import` statements to C# `using` directives.

**Import Styles and Their Translations:**

| Sharpy Import | C# Using | Notes |
|---------------|----------|-------|
| `import system.io` | `using System.IO;` | .NET framework namespace |
| `import system.io as io` | `using io = System.IO;` | Aliased .NET namespace |
| `import config` | `using config = MyProject.Config.Exports;` | Sharpy module (static alias) |
| `import lib.math.stats` | `using lib_math_stats = MyProject.Lib.Math.Stats.Exports;` | Nested Sharpy module |

**Key Implementation Logic:**

1. **Module Name Transformation (Line 204):**
   ```csharp
   var namespaceName = ConvertModuleNameToNamespace(alias.Name);
   // "system.io" → "System.IO"
   // "mymodule" → "Mymodule" (or computed from project structure)
   ```

2. **.NET Framework Detection (Line 205):**
   ```csharp
   var isNetFramework = IsNetFrameworkNamespace(alias.Name);
   ```

   Uses prefix matching against: `system`, `microsoft`, `windows`, `xamarin`, `mono`, `netstandard`

3. **Sharpy Module Handling (Lines 217-238):**

   **Why `Exports` Class?**

   Sharpy modules expose their members through a static class named `Exports`:
   ```csharp

       public static class Exports
       {
           public static int MAX_SIZE = 100;
           public static void Initialize() { }
       }

   ```

   To access members via `config.MAX_SIZE`, we need:

### 3.6 `GenerateFromImportUsings` — From-Import Translation

```csharp
private IEnumerable<UsingDirectiveSyntax> GenerateFromImportUsings(FromImportStatement fromImport)
```

**What it does:** Converts Python-style `from X import Y` statements to C# `using static` directives.

**From-Import Styles and Their Translations:**

| Sharpy Import | C# Using | Access Pattern |
|---------------|----------|----------------|
| `from system.io import File` | `using System.IO;` | `File.ReadAllText(...)` |
| `from config import MAX_SIZE` | `using static MyProject.Config.Exports;` | `MAX_SIZE` (direct access) |
| `from .helpers import process` | `using static MyProject.Utils.Helpers.Exports;` | `process(...)` |

**Critical Design Decision: `using static` (Lines 292-327)**

**Why `using static` for Sharpy Modules?**

Enables direct access to exported symbols without qualifying:

```python
# Sharpy
from config import MAX_SIZE, MIN_SIZE

def validate(value: int) -> bool:
    return MIN_SIZE <= value <= MAX_SIZE
```

```csharp
// Generated C#
using static MyProject.Config.Exports;

public static bool Validate(int value)
{
    return MIN_SIZE <= value && value <= MAX_SIZE;
    // Not: Exports.MIN_SIZE (verbose)
}
```

**Relative Import Resolution (Lines 302-304):**

```csharp
var moduleName = fromImport.ResolvedModulePath ?? fromImport.Module;
```

**Why Two Fields?**

| Field | Example | Usage |
|-------|---------|-------|
| `Module` | `".helpers"` | Original source text |
| `ResolvedModulePath` | `"mypackage.utils.helpers"` | Computed by semantic analyzer |

For relative imports (`.helpers`, `..config`), the semantic analyzer resolves them based on the current module's package location.

**Namespace Path Construction (Lines 305-323):**

```csharp
// "lib.math.operations" → "Lib.Math.Operations.Exports"
var moduleNamespacePath = ConvertModuleNameToNamespace(moduleName);
const string moduleClassName = "Exports";

string fullModuleClass;
if (!string.IsNullOrEmpty(_context.ProjectNamespace))
{
    fullModuleClass = $"{_context.ProjectNamespace}.{moduleNamespacePath}.{moduleClassName}";
}
else
{
    fullModuleClass = $"{moduleNamespacePath}.{moduleClassName}";
}
```

**Important:** The module class is **always** named `Exports`, not the module name. This is a Sharpy convention (see `GetModuleClassName()` method).

### 3.7 `IsNetFrameworkNamespace` — .NET Detection

```csharp
private static bool IsNetFrameworkNamespace(string moduleName)
```

**What it does:** Determines if a module name refers to a .NET framework namespace (not a Sharpy module).

**Detection Strategy:**

```csharp
var netPrefixes = new[]
{
    "system", "microsoft", "windows", "xamarin", "mono", "netstandard"
};
var firstPart = moduleName.Split('.')[0].ToLowerInvariant();
return netPrefixes.Contains(firstPart);
```

**Examples:**

| Module Name | Is .NET? | Reason |
|-------------|----------|--------|
| `system.io` | ✓ | Starts with "system" |
| `microsoft.extensions.logging` | ✓ | Starts with "microsoft" |
| `config` | ✗ | Not in prefix list |
| `System.Custom.MyNamespace` | ✓ | Starts with "system" (case-insensitive) |

**Why This Matters:**

.NET namespaces don't have an `Exports` class:
```csharp
// .NET namespace
using System.IO;  // NOT: using System.IO.Exports;

// Sharpy module
using config = MyProject.Config.Exports;  // Requires .Exports
```

### 3.8 `IsConstantCaseName` — Constant Detection

```csharp
private static bool IsConstantCaseName(string name)
```

**What it does:** Checks if a name appears to be in `CONSTANT_CASE` (all uppercase with underscores).

**Algorithm:**

```csharp
bool hasLetter = false;
foreach (char c in name)
{
    if (char.IsLetter(c))
    {
        if (!char.IsUpper(c))
            return false;  // Lowercase letter → not constant case
        hasLetter = true;
    }
    else if (!char.IsDigit(c) && c != '_')
    {
        return false;  // Invalid character
    }
}
return hasLetter;
```

**Examples:**

| Name | Is Constant? | Reason |
|------|--------------|--------|
| `MAX_SIZE` | ✓ | All uppercase letters |
| `MAX_SIZE_2` | ✓ | Uppercase with digits |
| `maxSize` | ✗ | Contains lowercase |
| `MAX_size` | ✗ | Mixed case |
| `123` | ✗ | No letters |
| `___` | ✗ | No letters |

**Usage Context:**

Used in `GetMangledVariableName()` (in main `RoslynEmitter.cs`) to decide casing for imported symbols:

```csharp
if (IsConstantCaseName(actualName))
{
    return NameMangler.ToConstantCase(actualName);  // MAX_SIZE → MAX_SIZE
}
else
{
    return NameMangler.ToPascalCase(actualName);  // process_file → ProcessFile
}
```

### 3.9 `EscapeCSharpKeyword` — Keyword Safety

```csharp
private static string EscapeCSharpKeyword(string name)
```

**What it does:** Escapes C# reserved keywords by prefixing with `@` symbol.

**Example Transformations:**

| Input | Output | Scenario |
|-------|--------|----------|
| `base` | `@base` | Sharpy module named "base" |
| `class` | `@class` | Variable named "class" |
| `default` | `@default` | Import alias "default" |
| `config` | `config` | Not a keyword, unchanged |

**Full Keyword List (Lines 382-396):**

```csharp
private static readonly HashSet<string> CSharpKeywords = new(StringComparer.OrdinalIgnoreCase)
{
    "abstract", "as", "base", "bool", "break", "byte", "case", "catch",
    "char", "checked", "class", "const", "continue", "decimal", "default",

```

**Case-Insensitive Matching:**

The `StringComparer.OrdinalIgnoreCase` ensures that `Base`, `BASE`, and `base` all get escaped.

**Usage Example:**

```python

```

```csharp
// Generated C#
using @base = MyProject.Base.Exports;  // Escaped to avoid C# keyword conflict
```

## 5. Patterns and Design Decisions

### Pattern: Import Symbol Pre-Registration

**Problem:** When generating code that references imported symbols, we need to know they came from a `from-import` to apply correct casing rules.

**Solution:** Pre-process all `from-import` statements before code generation:

```csharp
// Phase 1: Registration (in GenerateCompilationUnit)
foreach (var fromImport in module.Body.OfType<FromImportStatement>())
{
    foreach (var importedName in fromImport.Names)
    {
        _fromImportSymbols.Add(importedName.Name);
        if (importedName.AsName != null)
        {
            _fromImportSymbols.Add(importedName.AsName);
            _importAliasToOriginal[importedName.AsName] = importedName.Name;
        }
    }
}

// Phase 2: Usage (in GetMangledVariableName, later in code generation)
if (_fromImportSymbols.Contains(name))
{
    // Apply special casing rules
}
```

**Benefits:**
- Symbol name resolution is consistent across all references
- Alias mapping is available for all code generation phases
- No need to pass import context through every method

---

### Design Decision: Block-Scoped Namespaces (Unity Compatibility)

**Context (Line 64 comment):**
> "Use block-scoped namespace for C# 9.0 compatibility (Unity)"

**Modern C# (File-Scoped Namespace):**
```csharp
namespace MyProject.MyModule;

public class Exports { }
```

**Generated C# (Block-Scoped Namespace):**
```csharp
namespace MyProject.MyModule
{
    public class Exports { }
}
```

**Rationale:** Unity's Mono runtime (as of writing) supports C# 9.0 but not C# 10's file-scoped namespaces. Sharpy targets Unity as a key platform, so we use the older syntax.

---

### Design Decision: Entry Point vs. Library Module Distinction

**Code (Lines 49-58):**
```csharp
List<FromImportStatement>? fromImports = null;
if (!_context.IsEntryPoint)
{
    fromImports = module.Body
        .OfType<FromImportStatement>()
        .Where(f => f.ReExportedSymbols != null && f.ReExportedSymbols.Count > 0)
        .ToList();
}
```

**Why?**

| Module Type | Re-Export Behavior | Rationale |
|-------------|-------------------|-----------|
| Library module | Generates delegating members | Other modules import from this |
| Entry point | Does NOT re-export | This is the application entry, not a library |

**Example:**

```python
# mypackage/__init__.spy (library module, IsEntryPoint = false)
from .helpers import process_file  # Re-exported

# Generated: MyPackage.Exports class gets a ProcessFile member
```

```python
# main.spy (entry point, IsEntryPoint = true)
from config import MAX_SIZE  # NOT re-exported

# Generated: Main.Exports class does NOT get a MaxSize member
```

This prevents unnecessary member generation in application entry points.

---

### Design Decision: `#nullable enable` Post-Normalization

**Code (Lines 75-80):**
```csharp
var compilationUnit = CompilationUnit()
    .WithUsings(List(usingDirectives))
    .WithMembers(SingletonList<MemberDeclarationSyntax>(namespaceDecl))
    .NormalizeWhitespace();  // FIRST normalize

var nullablePragma = ParseLeadingTrivia("#nullable enable\n\n");
return compilationUnit.WithLeadingTrivia(nullablePragma);  // THEN add pragma
```

**Why This Order?**

Roslyn's `NormalizeWhitespace()` strips leading trivia. If we added the pragma before normalization:

```csharp
// WRONG ORDER:
var nullablePragma = ParseLeadingTrivia("#nullable enable\n\n");
var compilationUnit = CompilationUnit()
    .WithLeadingTrivia(nullablePragma)  // Added here
    .WithUsings(...)
    .WithMembers(...)
    .NormalizeWhitespace();  // STRIPPED! Pragma disappears

// Result: No #nullable enable in output
```

**Correct Order:** Normalize first, **then** add the pragma as leading trivia on the final syntax tree.

---

### Design Decision: Namespace Fallback Hierarchy

**Three-Level Fallback Strategy (in `GenerateNamespaceName`):**

```
1. Project-based (preferred):
   - Requires: ProjectNamespace + ProjectRootPath + SourceFilePath
   - Example: "MyProject.Lib.Utils.Helpers"
   - Use: Multi-file projects with organized structure

2. Explicit namespace + file:
   - Requires: ProjectNamespace + SourceFilePath
   - Example: "MyTools.ScriptName"
   - Use: Single-file scripts with manual namespace

3. Simple file-based:
   - Requires: SourceFilePath
   - Example: "Sharpy.ScriptName"
   - Use: Quick scripts, testing

4. Generic fallback:
   - Requires: Nothing
   - Example: "SharpyGenerated"
   - Use: REPL, in-memory compilation
```

**Why Hierarchical?**

Provides maximum flexibility:
- Large projects get organized namespaces
- Quick scripts get simple namespaces
- REPL mode still works

### Common Issues and Solutions

| Issue | Symptom | Cause | Fix |
|-------|---------|-------|-----|
| Wrong namespace | `MyProject.Src.MyModule` instead of `MyProject.MyModule` | ProjectRootPath not set to `src/` | Verify context setup |
| Missing `using static` | `MAX_SIZE` not found | `from-import` generated regular using | Check `IsNetFrameworkNamespace()` logic |
| Duplicate using directives | `using System;` appears twice | Deduplication failed | Check normalization in dedup loop |
| `#nullable enable` missing | Nullable warnings in generated code | Added before `NormalizeWhitespace()` | Move pragma to after normalization |
| Keyword collision | `using base = ...` compile error | Missing keyword escaping | Verify `EscapeCSharpKeyword()` called |
| Import alias not working | Alias name not recognized | `_importAliasToOriginal` not populated | Check pre-processing loop |

### Changing Namespace Generation Strategy

To switch to file-scoped namespaces (C# 10+):

1. **Replace namespace declaration (Line 66):**
   ```csharp
   // Old (block-scoped):
   var namespaceDecl = NamespaceDeclaration(namespaceName)
       .WithMembers(SingletonList<MemberDeclarationSyntax>(moduleClass));

   // New (file-scoped):
   var namespaceDecl = FileScopedNamespaceDeclaration(namespaceName);
   ```

2. **Add module class to compilation unit instead:**
   ```csharp
   var compilationUnit = CompilationUnit()
       .WithUsings(List(usingDirectives))
       .WithMembers(SingletonList<MemberDeclarationSyntax>(namespaceDecl))
       .AddMembers(moduleClass);  // Add directly
   ```

3. **Update Unity compatibility note**

### Extending Import Alias Support

To support wildcard imports (`from module import *`):

1. **Detect wildcard in `GenerateFromImportUsings()`:**

   ```csharp
   if (fromImport.Names.Any(n => n.Name == "*"))
   ```

## Summary

`RoslynEmitter.CompilationUnit.cs` is the **architectural scaffolding** of Sharpy's C# code generation. It transforms a Sharpy module into a complete, compilable C# source file with proper structure:

| Sharpy Concept | C# Output |
|----------------|-----------|
| `import system.io` | `using System.IO;` |
| `import config` | `using config = MyProject.Config.Exports;` |
| `from config import MAX` | `using static MyProject.Config.Exports;` |
| File: `src/lib/utils.spy` | `namespace MyProject.Lib.Utils` |
| Package: `pkg/__init__.spy` | `namespace MyProject.Pkg` (not `Pkg.Init`) |
| Module body | `public static class Exports { ... }` |

**Key Design Insights:**

1. **Pre-Registration Pattern:** Import symbols are registered before code generation for consistent casing
2. **Hierarchical Namespace Fallback:** Provides flexibility from large projects to REPL mode
3. **Entry Point Distinction:** Library modules re-export, entry points don't
4. **Post-Normalization Pragmas:** `#nullable enable` added after whitespace normalization
5. **Exports Class Convention:** All Sharpy modules expose members through a static `Exports` class

**Critical for Debugging:**
- Import symbol registration (`_fromImportSymbols`, `_importAliasToOriginal`)
- Namespace generation strategy selection (check context properties)
- Using directive deduplication (normalized string comparison)
