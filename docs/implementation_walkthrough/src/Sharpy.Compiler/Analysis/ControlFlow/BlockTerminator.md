# Walkthrough: BlockTerminator.cs

**Source File**: `src/Sharpy.Compiler/Analysis/ControlFlow/BlockTerminator.cs`

---

## Overview

`BlockTerminator.cs` defines the **control flow terminator types** for the Sharpy compiler's Control Flow Graph (CFG) infrastructure. Terminators describe **how control flow exits a basic block**—whether it's branching, returning, throwing, or looping.

This file is a pure type definition file with no logic—it's a collection of **immutable record types** that represent the different ways a block can end. Every basic block (except the exit block) has exactly one terminator, making this a foundational abstraction for control flow analysis.

### Role in the Compiler Pipeline

```
Source (.spy) → Lexer → Parser (AST) → Semantic Analysis → [CFG CONSTRUCTION] → RoslynEmitter → C#
                                                                    ↑
                                                            BlockTerminator used here
```

Terminators are created during **CFG construction** by the `ControlFlowGraphBuilder` when analyzing function bodies. They're then used by:
- **Return path analysis**: Identifying all paths through a function to ensure proper returns
- **Unreachable code detection**: Finding code after terminal statements (return/raise/break/continue)
- **Loop validation**: Ensuring break/continue only appear inside loops
- **Future async analysis**: Identifying control flow boundaries for state machine generation

## Architecture and Design

### Base Type: `BlockTerminator`

```csharp
public abstract record BlockTerminator
{
    public Statement? SourceStatement { get; init; }
}
```

All terminators inherit from this abstract base record. Key design decisions:

1. **Immutable Records**: Terminators are immutable because they represent computed control flow that doesn't change after CFG construction. Using `record` gives us structural equality and concise syntax.

2. **Abstract Base**: Forces all terminators to be one of the specific types defined below—no generic terminators exist.

3. **Source Statement Tracking**: The `SourceStatement` property links back to the AST node that created this terminator. This is crucial for **error diagnostics**—when reporting unreachable code or missing returns, we need to point to the source location.

### Design Pattern: Discriminated Union

This file implements a **discriminated union pattern** (also called a sum type or tagged union) in C#. Each terminator type is a sealed record that cannot be further subclassed. This gives exhaustive pattern matching capabilities:

```csharp
var message = terminator switch
{
    ReturnTerminator => "Function returns",
    ThrowTerminator => "Function throws",
    BranchTerminator => "Unconditional branch",
    // Compiler ensures all cases are handled
    _ => throw new UnreachableException()
};
```

## Terminator Types Reference

### 1. `BranchTerminator` — Unconditional Jump

```csharp
public sealed record BranchTerminator(BasicBlock Target) : BlockTerminator;
```

**Purpose**: Represents an unconditional jump to a single target block.

**Generated By**:
- Fall-through from one statement to the next
- Unconditional `goto` (if Sharpy adds them)
- The implicit flow at the end of an `if` branch that rejoins the main path

**Example Scenario**:
```python
x = 5      # Block A
y = 10     # Still Block A
# BranchTerminator(Target = Block B) ← fall-through
z = x + y  # Block B
```

**Key Properties**:
- `Target`: The single block to jump to

---

### 2. `ConditionalBranchTerminator` — If/While Conditions

```csharp
public sealed record ConditionalBranchTerminator(
    Expression Condition,
    BasicBlock TrueTarget,
    BasicBlock FalseTarget
) : BlockTerminator;
```

**Purpose**: Represents a binary decision point based on a boolean expression.

**Generated By**:
- `if` statements
- `while` loop conditions
- `for` loop conditions (after transformation)
- Ternary expressions (potentially, if they generate control flow)

**Example Scenario**:
```python
if x > 10:  # ConditionalBranchTerminator(Condition=x>10, TrueTarget=then_block, FalseTarget=else_block)
    print("big")   # then_block
else:
    print("small") # else_block
```

**Key Properties**:
- `Condition`: The expression to evaluate (must be boolean)
- `TrueTarget`: Block to execute if condition is true
- `FalseTarget`: Block to execute if condition is false

**Note**: Both targets must be specified, even if one is empty. An `if` without `else` has a `FalseTarget` that jumps to the continuation block.

---

### 3. `SwitchTerminator` — Multi-Way Branch (Future)

```csharp
public sealed record SwitchTerminator(
    Expression Value,
    ImmutableArray<SwitchCase> Cases,
    BasicBlock? DefaultTarget
) : BlockTerminator;

public sealed record SwitchCase(
    Pattern? Pattern,  // null for default case
    Expression? Guard,
    BasicBlock Target
);
```

**Purpose**: Represents a multi-way branch based on pattern matching (for future `match`/`switch` statements in v0.2.x).

**Generated By**:
- Python `match` statements (PEP 634, when Sharpy adds support)

**Example Scenario** (hypothetical):
```python
match status:
    case 200:
        return "OK"
    case 404:
        return "Not Found"
    case _:
        return "Other"
```

**Key Properties**:
- `Value`: The expression being matched against
- `Cases`: Array of pattern-target pairs
- `DefaultTarget`: Fallback block if no patterns match (nullable for exhaustive matches)

**SwitchCase Details**:
- `Pattern`: The pattern to match (e.g., literal value, type check, destructuring)—`null` represents the default case
- `Guard`: Optional `when` clause for additional filtering
- `Target`: Block to jump to when this case matches

**Status**: This is forward-looking infrastructure. As of the current implementation, Sharpy doesn't generate switch terminators yet.

### 4. `ReturnTerminator` — Function Exit

```csharp
public sealed record ReturnTerminator(Expression? Value) : BlockTerminator;
```

**Purpose**: Represents a `return` statement that exits the function.

**Generated By**:
- Explicit `return` statements
- Implicit return at the end of functions returning `None`

**Example Scenario**:
```python
def calculate(x: int) -> int:
    if x < 0:
        return 0  # ReturnTerminator(Value=IntLiteral(0))
    return x * 2  # ReturnTerminator(Value=BinaryOp(...))
```

**Key Properties**:
- `Value`: The expression to return (nullable for `None` or `void` functions)

**Important for Analysis**:
- Return path analysis checks that all non-`None` functions have return terminators on all paths
- Blocks after a return terminator are unreachable

---

### 5. `ThrowTerminator` — Exception Exit

```csharp
public sealed record ThrowTerminator(Expression? Exception) : BlockTerminator;
```

**Purpose**: Represents a `raise` statement that throws an exception.

**Generated By**:
- `raise SomeException(...)` — throws new exception
- Bare `raise` (re-raise) — re-throws current exception (but see `RethrowTerminator` below)

**Example Scenario**:
```python
def validate(x: int) -> None:
    if x < 0:
        raise ValueError("Negative value")  # ThrowTerminator(Exception=CallExpression(...))
```

**Key Properties**:
- `Exception`: The exception expression to throw (nullable for bare `raise`)

**Special Case**: Bare `raise` with `Exception=null` differs from `RethrowTerminator` (see below).

---

### 6. `RethrowTerminator` — Explicit Re-raise

```csharp
public sealed record RethrowTerminator() : BlockTerminator;
```

**Purpose**: Represents a bare `raise` statement with no expression inside an exception handler.

**Generated By**:
- Bare `raise` inside `except` blocks

**Example Scenario**:
```python
try:
    risky_operation()
except Exception:
    log_error()
    raise  # RethrowTerminator()
```

**Why Separate from ThrowTerminator?**
- Explicit semantic distinction: re-throwing preserves the original stack trace
- In the future, may emit different IL (C# has `throw;` vs `throw ex;`)
- Makes validation easier: `RethrowTerminator` can only appear in exception handlers

---

### 7. `BreakTerminator` — Loop Exit

```csharp
public sealed record BreakTerminator(BasicBlock Target) : BlockTerminator;
```

**Purpose**: Breaks out of a loop to a specific target block.

**Generated By**:
- `break` statements
- `BreakWithFlagStatement` (internal node for Python's `else` clause on loops)

**Example Scenario**:
```python
while True:
    if condition:
        break  # BreakTerminator(Target=after_loop_block)
    process()
# after_loop_block
```

**Key Properties**:
- `Target`: The block immediately after the loop

**Special Note on Loop-Else**:
Python's `for...else` and `while...else` execute the `else` block only if the loop completes without a `break`. Sharpy uses `BreakWithFlagStatement` (an internal AST node) that:
1. Sets a flag before the break
2. Creates a `BreakTerminator` to the post-loop block
3. The post-loop block checks the flag to decide whether to run the `else` block

The flag assignment is added as a regular statement in the block; the terminator itself is just a plain `BreakTerminator`.

---

### 8. `ContinueTerminator` — Loop Iteration

```csharp
public sealed record ContinueTerminator(BasicBlock Target) : BlockTerminator;
```

**Purpose**: Continues to the next iteration of a loop.

**Generated By**:
- `continue` statements

**Example Scenario**:
```python
for item in items:
    if item.skip:
        continue  # ContinueTerminator(Target=loop_condition_block)
    process(item)
```

**Key Properties**:
- `Target`: The block that checks the loop condition (for `while`) or advances the iterator (for `for`)

**Note**: Unlike break, continue doesn't exit the loop—it jumps back to the top.

---

### 9. `UnreachableTerminator` — Error Marker

```csharp
public sealed record UnreachableTerminator() : BlockTerminator;
```

**Purpose**: Marks blocks that don't naturally terminate. This **should not appear in a well-formed CFG**.

**Usage**:
- Temporary placeholder during CFG construction
- Indicates a bug in the CFG builder if it persists

**Example Scenario**:
If the CFG builder encounters a malformed AST or has a logic error, it might temporarily create a block with `UnreachableTerminator`. This should be replaced with a proper terminator before the CFG is finalized.

**Debugging Tip**: If you see an `UnreachableTerminator` in a completed CFG, there's a bug in `ControlFlowGraphBuilder`.

## Patterns and Design Decisions

### Why Records Instead of Classes?

1. **Immutability**: Terminators represent computed facts about control flow—they shouldn't change
2. **Value Semantics**: Two `ReturnTerminator(null)` instances are the same conceptually
3. **Pattern Matching**: C# records get excellent switch expression support
4. **Concise Syntax**: Primary constructors reduce boilerplate

### Why Sealed?

All concrete terminator types are `sealed` to prevent further subclassing. This ensures the compiler can:
- Exhaustively enumerate all terminator types
- Optimize switch expressions
- Reason about control flow without unknown extensions

### Separation of Concerns

Notice this file contains **zero logic**—it's pure data types. The logic for:
- **Creating** terminators: Lives in `ControlFlowGraphBuilder.cs`
- **Analyzing** terminators: Lives in `ControlFlowAnalysis.cs`
- **Storing** terminators: Lives in `BasicBlock.cs`

This separation keeps the codebase maintainable.

## Debugging Tips

### 1. Visualizing Terminators

When debugging CFG issues, add a `ToString()` override or use a debugger visualizer:

```csharp
var description = terminator switch
{
    BranchTerminator b => $"Branch → {b.Target}",
    ConditionalBranchTerminator c => $"If {c.Condition} then {c.TrueTarget} else {c.FalseTarget}",
    ReturnTerminator r => $"Return {r.Value?.ToString() ?? "None"}",
    BreakTerminator b => $"Break → {b.Target}",
    ContinueTerminator c => $"Continue → {c.Target}",
    // ... etc
};
```

### 2. Checking Source Statements

If a diagnostic is wrong, check if `SourceStatement` is properly set:
- Is it null when it shouldn't be?
- Does it point to the right AST node?
- Does the span match the expected source location?

### 3. Common Mistakes

**Missing Terminator**: Every block except the exit block must have a terminator. If you see null terminators, the CFG builder has a bug.

**Wrong Target**: Branch/break/continue terminators must point to valid blocks in the same CFG. Check that:
- The target block exists
- The target is reachable
- The target has the expected predecessors

**Mismatched Types**: If you see a `ThrowTerminator` where you expected `RethrowTerminator` (or vice versa), check the AST node type and the CFG builder's logic for `raise` statements.

### 4. Testing Terminators

When writing tests for CFG construction:

```csharp
var cfg = builder.Build(functionDef);
var firstBlock = cfg.Blocks[0];

// Check terminator type
Assert.IsType<ConditionalBranchTerminator>(firstBlock.Terminator);

// Check specific properties
var conditional = (ConditionalBranchTerminator)firstBlock.Terminator!;
Assert.Equal("x > 10", conditional.Condition.ToString());
Assert.Equal(cfg.Blocks[1], conditional.TrueTarget);
```

## Contribution Guidelines

### When to Modify This File

You should modify `BlockTerminator.cs` when:

1. **Adding New Control Flow Constructs**:
   - Example: Adding `async`/`await` might require an `AwaitTerminator`
   - Example: Adding exception filters (`except E as e if condition`) might extend `ThrowTerminator`

2. **Refining Existing Terminators**:
   - Example: Adding source spans for better diagnostics
   - Example: Splitting `ThrowTerminator` into `ThrowNewTerminator` and `ThrowCaughtTerminator`

3. **Implementing Planned Features**:
   - The `SwitchTerminator` is ready for pattern matching implementation
   - When implementing `match` statements, ensure `SwitchCase` aligns with the pattern AST types

### What NOT to Change

**Do not**:
- Add mutable properties (violates immutability contract)
- Add logic or methods (terminators are pure data)
- Remove `sealed` keyword (breaks exhaustiveness guarantees)
- Change primary constructor parameters (breaks CFG builder)

### Testing New Terminators

When adding a new terminator type:

1. **Update `ControlFlowGraphBuilder.cs`** to generate the new terminator from AST
2. **Update `ControlFlowAnalysis.cs`** to handle the new terminator in analysis passes
3. **Add integration tests** in `Sharpy.Compiler.Tests/Integration/ControlFlow/`
4. **Document the terminator** with XML comments and usage examples

### Code Style

Follow existing patterns:
- Use XML doc comments with `<summary>` and `<remarks>`
- Document what AST nodes generate each terminator
- Use sealed records with primary constructors
- Keep properties nullable only when semantic meaning requires it (e.g., `ReturnTerminator.Value` can be null)

## Example: Complete Control Flow Pattern

Here's how all the pieces fit together for a simple function:

```python
def process(x: int) -> str:
    if x < 0:
        raise ValueError("Negative")
    elif x == 0:
        return "zero"
    else:
        return "positive"
```

**CFG Structure**:

```
Entry Block
  ↓ BranchTerminator
Block 1: [empty]
  ↓ ConditionalBranchTerminator(x < 0)
  ├─ True → Block 2
  └─ False → Block 3

Block 2: [empty]
  ↓ ThrowTerminator(ValueError("Negative"))
  (no successors)

Block 3: [empty]
  ↓ ConditionalBranchTerminator(x == 0)
  ├─ True → Block 4
  └─ False → Block 5

Block 4: [empty]
  ↓ ReturnTerminator("zero")
  (no successors)

Block 5: [empty]
  ↓ ReturnTerminator("positive")
  (no successors)

Exit Block
  (no terminator)
```

Notice:
- Every block has exactly one terminator
- `ThrowTerminator` and `ReturnTerminator` have no successors (they exit the function)
- `ConditionalBranchTerminator` has two successors
- The exit block is never reached (all paths return or throw)

## Summary

`BlockTerminator.cs` is a foundational type definition file that models **how control flow exits basic blocks**. Its clean discriminated union design makes control flow analysis both type-safe and exhaustive. As a pure data type file with no logic, it's stable and rarely needs changes—but when adding new control flow features to Sharpy, this is where you start.
