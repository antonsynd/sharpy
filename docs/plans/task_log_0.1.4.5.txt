- [ ] **Section: loop_else.md**: Loop else clause not validated
  - **Spec says:** "For both `for` and `while` loops, an `else` clause can be added to execute if the loop completes without a break."
  - **Implementation:** The AST (`WhileStatement` and `ForStatement` at Statement.cs:127-141) has no `ElseBody` property. The `ControlFlowValidator` does not handle loop else clauses.
  - **Impact:** Medium - Loop else is a documented language feature that isn't supported at the AST level.

- [ ] **Section: match_statement.md**: Match statement not handled
  - **Spec says:** Match statements support exhaustiveness checking and case-by-case control flow.
  - **Implementation:** No `MatchStatement` case exists in `ValidateStatement()` (lines 88-139). There's no `MatchStatement` type in the AST.
  - **Impact:** Medium - Match statement is a documented feature that isn't parsed or validated yet.

- [ ] **Section: exception_handling.md**: Try `else` clause not handled
  - **Spec says:** `"else: # Executed if no exception"` - try statements can have an else clause.
  - **Implementation:** `TryStatement` AST (Statement.cs:146-151) has no `ElseBody` property. `ValidateTry()` doesn't account for else clause in return path analysis.
  - **Impact:** Low-Medium - Try else is documented but not supported at AST level.

- [ ] **Section: async_programming.md**: Async function control flow not validated
  - **Spec says:** Async functions use `async def` and can have generators with `yield`.
  - **Implementation:** No async-specific control flow validation. No handling of `yield` as a control flow element.
  - **Impact:** Medium - Async functions are documented but control flow validation doesn't account for their semantics.

- [ ] **Undocumented Behavior**: `pass`, `assert`, and `ellipsis` statements
  - **Spec says:** `pass` is a no-op; `...` (ellipsis) compiles to `throw new NotImplementedException()`.

  - **Impact:** Low - Correct behavior, but ellipsis (`...`) should arguably be treated like `raise` for control flow since it always throws.

2. **Add match statement support**:

   - For return analysis: all cases must return for match to "always return"

   - Try else runs only if no exception, contributing to return paths

4. **Handle ellipsis as control flow exit**:
   - Add case for `EllipsisStatement` (if it exists as statement) that returns `(false, true)` since it always throws

5. **Add async/generator validation**:
