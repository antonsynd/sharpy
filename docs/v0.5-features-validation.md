# Sharpy v0.5 Feature Validation Checklist

**Purpose:** This document provides a detailed breakdown of all v0.5 features for systematic validation. Each feature is designed to be small, semantically atomic, and complete. Engineers can use this to verify implementation correctness from lexing → parsing → codegen → testing.

**How to use this document:**
1. For each feature, check that the lexer correctly tokenizes the syntax
2. Verify the parser builds correct AST nodes
3. Ensure codegen produces correct C# output
4. Confirm sufficient test coverage exists

---

## 1. Lexical Structure

### 1.1. Source File Encoding

**What is allowed:**
- File extension: `.spy` (required)
- Encoding: UTF-8 (required)
- Line endings: LF (`\n`) or CRLF (`\r\n`)
- BOM: Optional (but not recommended)

**What is NOT allowed:**
- Non-UTF-8 encodings
- Other file extensions for Sharpy source

**Validation points:**
- [ ] Lexer accepts UTF-8 files with `.spy` extension
- [ ] Lexer handles both LF and CRLF line endings
- [ ] Lexer accepts (but doesn't require) BOM
- [ ] Lexer rejects non-UTF-8 files with clear error

---

### 1.2. Indentation

**What is allowed:**
- Exactly 4 spaces per indentation level (enforced)
- Consistent indentation throughout file
- Indentation increases to start new block
- Indentation decreases to previous level to end block(s)

**What is NOT allowed:**
- Tabs for indentation
- Mixed spaces and tabs
- Inconsistent spacing (not multiples of 4)
- Decreasing indentation to arbitrary levels (must match a previous level)

**Validation points:**
- [ ] Lexer generates INDENT token when indentation increases by 4 spaces
- [ ] Lexer generates DEDENT token(s) when indentation decreases
- [ ] Lexer enforces exactly 4 spaces per level
- [ ] Lexer rejects tabs with clear error
- [ ] Lexer rejects inconsistent indentation with clear error
- [ ] Lexer handles multiple DEDENT tokens at once correctly
- [ ] Parser correctly builds nested block structures from INDENT/DEDENT

**Test cases:**
```python
# Valid: proper 4-space indentation
if x:
    y = 1
    if z:
        w = 2
    v = 3

# Invalid: tabs (should error)
if x:
	y = 1  # Tab character - ERROR

# Invalid: 2 spaces (should error)
if x:
  y = 1  # 2 spaces - ERROR

# Invalid: arbitrary dedent level (should error)
if x:
    if y:
        z = 1
  w = 2  # Dedent to 2 spaces - ERROR (not a previous level)
```

### 1.3. Comments

**What is allowed:**
- Single-line comments starting with `#` to end of line
- `#` at any position on a line (including after code)
- Multiple consecutive `#` lines for multi-line comments
- Triple-quoted strings as docstrings (not comments, but serve similar purpose)

**What is NOT allowed:**
- Multi-line comment syntax like `/* */` (C-style)
- `#` inside string literals treated as comment

**Validation points:**
- [ ] Lexer ignores everything from `#` to end of line
- [ ] Lexer preserves `#` inside string literals as literal text
- [ ] Parser handles comments at any position
- [ ] Codegen has option to omit comments or emit them as C# comments (`//`)
- [ ] Triple-quoted strings are parsed as string literals, not comments

**Test cases:**
```python

```

**What is NOT allowed:**
- Starting with a digit
- Containing hyphens or other special characters
- Using keywords as identifiers (unless backtick-escaped)

**Validation points:**
- [ ] Lexer accepts valid identifiers
- [ ] Lexer rejects identifiers starting with digits
- [ ] Lexer rejects identifiers with invalid characters
- [ ] Lexer treats keywords as keywords, not identifiers
- [ ] Case sensitivity is preserved

**Test cases:**

stream: FileStream? = None
