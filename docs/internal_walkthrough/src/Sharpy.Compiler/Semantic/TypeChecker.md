# Walkthrough: TypeChecker.cs

**Source File**: `src/Sharpy.Compiler/Semantic/TypeChecker.cs`

---

## Overview

`TypeChecker.cs` is the **heart of Sharpy's type system**. This class is responsible for validating that all types in a Sharpy program are used correctly. It walks through the Abstract Syntax Tree (AST) generated by the parser and ensures:

- Variables are assigned compatible types
- Function calls have the correct number and types of arguments
- Return statements match function signatures
- Operators are used with valid operands
- Type narrowing works correctly in conditional branches (e.g., `x is not None`)
- Access modifiers are respected (private/public members)

The TypeChecker is the third stage in the compilation pipeline:
```
Source Code → Lexer → Parser → **Type Checker** → Code Generator → C# Assembly
```

It works in tandem with several helper components:
- **SymbolTable**: Tracks variables, functions, and types in scope
- **SemanticInfo**: Caches type information for AST nodes
- **TypeResolver**: Converts type annotations to SemanticType instances
- **ControlFlowValidator**: Ensures all code paths return values
- **AccessValidator**: Validates private/public access

## Key Methods

### 1. Entry Point: `CheckModule(Module module)`

This is the main entry point for type checking a Sharpy module.

```csharp
public void CheckModule(Module module)
{
    _logger.LogInfo("Type checking module");
    foreach (var statement in module.Body)
    {
        CheckStatement(statement);
    }
}
```

**What it does:**
- Iterates through all top-level statements in the module
- Delegates to `CheckStatement()` for each one
- Collects errors but doesn't throw exceptions (configurable)

---

### 2. Statement Dispatcher: `CheckStatement(Statement statement)`

This method is a **switch-based dispatcher** that routes different statement types to specialized handlers.

**Key patterns handled:**

| Statement Type | Handler Method | What it validates |
|---------------|---------------|-------------------|
| `FunctionDef` | `CheckFunction()` | Parameter types, return type, body |
| `ClassDef` | `CheckClass()` | Field types, method signatures |
| `Assignment` | `CheckAssignment()` | Type compatibility, tuple unpacking |
| `VariableDeclaration` | `CheckVariableDeclaration()` | Declared type matches initializer |
| `ReturnStatement` | `CheckReturn()` | Return type matches function signature |
| `IfStatement` | `CheckIf()` | Condition is boolean, type narrowing |
| `ForStatement` | `CheckFor()` | Iterator type, loop variable inference |
| `TryStatement` | `CheckTry()` | Exception handler scoping |

**Design Pattern:** Visitor Pattern (manual dispatch via switch)

---

### 3. Function Checking: `CheckFunction(FunctionDef functionDef)`

Validates function definitions including special cases for constructors and instance methods.

**Key validations:**

1. **Constructor validation** (`__init__`):
   - Must return `None` or have no return type annotation
   ```csharp
   if (functionDef.Name == "__init__" && returnType != SemanticType.Void)
   {
       AddError("Constructor '__init__' cannot have return type...");
   }
   ```

2. **Instance method validation**:
   - First parameter must be named `self`
   - Type of `self` is set to the current class type
   ```csharp
   if (_currentClass != null && functionDef.Parameters[0].Name != "self")
   {
       AddError("Instance method must have 'self' as first parameter");
   }
   ```

3. **Parameter ordering**:
   - Non-default parameters cannot follow default parameters
   ```csharp
   bool hasSeenDefault = false;
   for (int i = 0; i < functionDef.Parameters.Count; i++)
   {
       if (param.DefaultValue != null)
           hasSeenDefault = true;
       else if (hasSeenDefault)
           AddError("Non-default parameter cannot follow default parameters");
   }
   ```

4. **Control flow validation**:
   - Delegates to `ControlFlowValidator` to ensure all code paths return a value

**Scope management:**
- Enters a new scope: `_symbolTable.EnterScope("function:name")`
- Registers parameters in the scope
- Checks function body
- Exits scope: `_symbolTable.ExitScope()`

---

### 4. Assignment Checking: `CheckAssignment(Assignment assignment)`

Handles both simple assignments and tuple unpacking.

**Case 1: Tuple Unpacking**

```python
x, y = (1, 2)  # Tuple unpacking
```

Validation:
- Right-hand side must be a tuple type
- Element count must match
- Each element type must be compatible
- Creates new variable symbols for unpacked variables

**Case 2: Simple Assignment (Type Inference)**

```python
x = 42  # x is inferred as int
```

**Important:** In Sharpy, simple assignments create **new variable versions**, enabling Python-like behavior where variables can change types:

```python
x = 42      # x: int
x = "hello" # x: str (allowed!)
```

This is implemented by redefining the symbol:
```csharp
var newSymbol = new VariableSymbol
{
    Name = targetId.Name,
    Type = inferredType,  // Type from right-hand side
    // ...
};
_symbolTable.Define(newSymbol);  // Replaces existing symbol
```

**Constant protection:**
```csharp
if (existingSymbol is VariableSymbol { IsConstant: true })
{
    AddError("Cannot reassign constant variable");
}
```

### 5. Type Narrowing: `ExtractNarrowedTypes(Expression condition, bool isPositiveBranch)`

Implements **flow-sensitive type analysis** for nullable types and type checks.

**Supported patterns:**

1. **`x is not None`** (null check):
   ```python
   x: int? = get_value()
   if x is not None:
       # Here, x is narrowed from int? to int
       print(x + 1)  # Safe to use as int
   ```

2. **`x is None`** (reverse):
   ```python
   if x is None:
       return
   # Here, x is narrowed from int? to int
   ```

3. **`isinstance(x, Type)`**:
   ```python
   animal: Animal = get_animal()
   if isinstance(animal, Dog):
       # Here, animal is narrowed from Animal to Dog
       animal.bark()
   ```

4. **`A and B`** (conjunctions):
   ```python
   if x is not None and y is not None:
       # Both x and y are narrowed
   ```

**How it works:**

The narrowed types are stored in a dictionary:
```csharp
private Dictionary<string, SemanticType> _narrowedTypes = new();
```

When checking an `if` statement:
1. Extract narrowed types for the then-branch
2. Apply narrowing by adding to `_narrowedTypes`
3. Check the then-body
4. Restore original types
5. Extract narrowed types for else-branch (inverted)
6. Apply and check else-body

**Example implementation:**
```csharp
private void CheckIf(IfStatement ifStmt)
{
    var narrowedTypesInThen = ExtractNarrowedTypes(ifStmt.Test, true);
    var savedNarrowedTypes = new Dictionary<string, SemanticType>(_narrowedTypes);
    
    // Apply narrowing for then-branch
    foreach (var kvp in narrowedTypesInThen)
        _narrowedTypes[kvp.Key] = kvp.Value;
    
    // Check then-body with narrowed types
    foreach (var stmt in ifStmt.ThenBody)
        CheckStatement(stmt);
    
    // Restore and apply narrowing for else-branch
    _narrowedTypes = savedNarrowedTypes;
    // ...
}
```

### 6. Expression Type Inference: `CheckExpression(Expression expr)`

This is the **workhorse method** that determines the type of any expression.

**Caching mechanism:**
```csharp
var cached = _semanticInfo.GetExpressionType(expr);
if (cached != null)
    return cached;  // Return cached result
```

**Type inference by expression type:**

| Expression | Inferred Type | Example |
|-----------|--------------|---------|
| `IntegerLiteral` | `SemanticType.Int` | `42` |
| `StringLiteral` | `SemanticType.Str` | `"hello"` |
| `BooleanLiteral` | `SemanticType.Bool` | `True` |
| `NoneLiteral` | `SemanticType.Void` | `None` |
| `Identifier` | From symbol table | `x` |
| `BinaryOp` | Depends on operator | `1 + 2` → `int` |
| `FunctionCall` | Function's return type | `len([1,2])` → `int` |
| `ListLiteral` | `list[T]` where T is element type | `[1, 2, 3]` → `list[int]` |
| `TupleLiteral` | `tuple[T1, T2, ...]` | `(1, "hi")` → `tuple[int, str]` |

**Result is cached:**
```csharp
_semanticInfo.SetExpressionType(expr, type);
return type;
```

### 7. Function Call Validation: `CheckFunctionCall(FunctionCall call)`

One of the most complex methods, handling:
- Regular function calls
- Constructor calls (calling a type)
- Overloaded builtin functions
- Method calls (via member access)

**Special case: Constructor calls**
```csharp
if (symbol is TypeSymbol typeSymbol)
{
    // Constructor call returns instance of the type
    return new UserDefinedType { Symbol = typeSymbol };
}
```

**Overload resolution for builtins:**
```csharp
var overloads = _symbolTable.BuiltinRegistry.GetFunctionOverloads(id.Name);
if (overloads != null && overloads.Count > 1)
{
    // Filter by argument count
    var candidateOverloads = overloads.Where(o => {
        var requiredParams = o.Parameters.Count(p => !p.HasDefault);
        var totalParams = o.Parameters.Count;
        return argTypes.Count >= requiredParams && argTypes.Count <= totalParams;
    }).ToList();
    
    // Find matching overload by type compatibility
    FunctionSymbol? matchingOverload = null;
    foreach (var overload in candidateOverloads)
    {
        // Check if all argument types match
        bool typesMatch = true;
        for (int i = 0; i < argTypes.Count; i++)
        {
            if (!IsAssignable(argTypes[i], overload.Parameters[i].Type))
            {
                typesMatch = false;
                break;
            }
        }
        if (typesMatch)
        {
            matchingOverload = overload;
            break;
        }
    }
    
    if (matchingOverload != null)
        return matchingOverload.ReturnType;
}
```

**Default parameter support:**
```csharp
var requiredParamCount = funcSymbol.Parameters.Count(p => !p.HasDefault);
var totalParamCount = funcSymbol.Parameters.Count;

if (argTypes.Count < requiredParamCount || argTypes.Count > totalParamCount)
{
    AddError($"Function expects {requiredParamCount} to {totalParamCount} arguments...");
}
```

### 8. Type Assignability: `IsAssignable(SemanticType source, SemanticType target)`

Determines if a value of type `source` can be assigned to a variable of type `target`.

**Extended rules beyond basic assignability:**

1. **Nullable types:**
   ```csharp
   if (target is NullableType nullable)
   {
       return source.IsAssignableTo(nullable.UnderlyingType);
   }
   ```
   Example: `int` is assignable to `int?`

2. **Generic covariance:**
   ```csharp
   if (sourceGeneric.Name == "list" || sourceGeneric.Name == "set")
   {
       return IsAssignable(sourceGeneric.TypeArguments[0], 
                          targetGeneric.TypeArguments[0]);
   }
   ```
   Example: `list[Dog]` is assignable to `list[Animal]` if `Dog` extends `Animal`

### 9. Comprehension Checking

List/set/dict comprehensions create their own scopes to prevent variable leakage.

**Pattern:**
```csharp
private SemanticType CheckListComprehension(ListComprehension listComp)
{
    // Enter comprehension scope
    _symbolTable.EnterScope("list-comprehension");
    
    // Process clauses (for/if)
    foreach (var clause in listComp.Clauses)
    {
        if (clause is ForClause forClause)
        {
            // Infer loop variable type from iterator
            var iterType = CheckExpression(forClause.Iterator);
            var elemType = ExtractElementType(iterType);
            
            // Define loop variable in comprehension scope
            var loopVarSymbol = new VariableSymbol { ... };
            _symbolTable.Define(loopVarSymbol);
        }
        else if (clause is IfClause ifClause)
        {
            // Validate filter is boolean
            var condType = CheckExpression(ifClause.Condition);
            if (!condType.IsAssignableTo(SemanticType.Bool))
                AddError("Comprehension filter must be bool");
        }
    }
    
    // Check element expression
    var elementType = CheckExpression(listComp.Element);
    
    _symbolTable.ExitScope();
    
    return new GenericType { Name = "list", TypeArguments = [elementType] };
}
```

**Key insight:** Loop variables in comprehensions don't leak to outer scope, matching Python behavior.

## Dependencies

### Internal Dependencies

1. **SymbolTable** (`Semantic/SymbolTable.cs`):
   - `Lookup(name)`: Find symbols by name
   - `Define(symbol)`: Add new symbols
   - `EnterScope()/ExitScope()`: Manage nested scopes

2. **SemanticInfo** (`Semantic/SemanticInfo.cs`):
   - Caches type information for AST nodes
   - `GetExpressionType(expr)`/`SetExpressionType(expr, type)`
   - `GetIdentifierSymbol(id)`/`SetIdentifierSymbol(id, symbol)`

3. **TypeResolver** (`Semantic/TypeResolver.cs`):
   - Converts type annotations to `SemanticType` instances
   - `ResolveTypeAnnotation(typeAnnotation)`

4. **ControlFlowValidator** (`Semantic/ControlFlowValidator.cs`):
   - Validates all code paths return values
   - `ValidateFunction(functionDef, returnType)`

5. **AccessValidator** (`Semantic/AccessValidator.cs`):
   - Validates access to private/public members
   - `ValidateFieldAccess(field, type, line, col)`
   - `ValidateMethodAccess(method, type, line, col)`

### AST Dependencies

Depends on all AST node types from `Parser/Ast/`:
- Statements: `FunctionDef`, `ClassDef`, `Assignment`, etc.
- Expressions: `BinaryOp`, `FunctionCall`, `Identifier`, etc.
- Types: `TypeAnnotation`, `GenericTypeAnnotation`, etc.

### Semantic Type System

Uses types from `Semantic/SemanticType.cs`:
- `SemanticType`: Base type
- `PrimitiveType`: Int, String, Bool, etc.
- `GenericType`: list[T], dict[K,V], etc.
- `NullableType`: T?
- `TupleType`: tuple[T1, T2, ...]
- `FunctionType`: (T1, T2) -> T3
- `UserDefinedType`: Classes, structs, interfaces

## Patterns and Design Decisions

### 1. Visitor Pattern (Manual)

Instead of using the Visitor pattern with virtual dispatch, TypeChecker uses a **switch-based manual dispatch**:

```csharp
SemanticType type = expr switch
{
    IntegerLiteral => SemanticType.Int,
    StringLiteral => SemanticType.Str,
    Identifier id => CheckIdentifier(id),
    BinaryOp binOp => CheckBinaryOp(binOp),
    // ... more cases
    _ => SemanticType.Unknown
};
```

**Why?** 
- Simpler than double dispatch
- All type logic in one file
- Easier to add new expression types
- Pattern matching makes it readable

### 2. Error Recovery

The TypeChecker continues after errors instead of throwing exceptions immediately:

```csharp
public bool ContinueAfterError { get; set; } = true;
public int MaxErrors { get; set; } = 100;

private void AddError(string message, int? line, int? column)
{
    if (_errors.Count >= MaxErrors)
    {
        // Stop collecting errors
        return;
    }
    
    _errors.Add(new SemanticError(message, line, column));
    _logger.LogError(message, line ?? 0, column ?? 0);
}
```

**Why?**
- Better developer experience (see all errors at once)
- Matches compiler behavior (like C#, TypeScript)
- Configurable for different use cases

### 3. Type Caching

All expression types are cached in `SemanticInfo`:

```csharp
var cached = _semanticInfo.GetExpressionType(expr);
if (cached != null)
    return cached;

// ... compute type ...

_semanticInfo.SetExpressionType(expr, type);
```

**Why?**
- Expressions can appear multiple times (e.g., in error messages)
- Avoids redundant type checking
- Performance optimization

### 4. Scope-Based Type Narrowing

Type narrowing is implemented using a dictionary that's saved/restored:

```csharp
var savedNarrowedTypes = new Dictionary<string, SemanticType>(_narrowedTypes);
// Apply narrowing...
foreach (var kvp in narrowedTypesInThen)
    _narrowedTypes[kvp.Key] = kvp.Value;
// Check body...
CheckStatement(stmt);
// Restore...
_narrowedTypes = savedNarrowedTypes;
```

**Why?**
- Type narrowing is scoped to branches
- Simple to implement and reason about
- Matches how developers think about control flow

### 5. Special Symbol Table Handling

Variables can be redefined in Sharpy (Python-like), but constants cannot:

```csharp
if (existingSymbol is VariableSymbol { IsConstant: true })
{
    AddError("Cannot reassign constant variable");
    return;
}

// For non-const, allow redefinition
var newSymbol = new VariableSymbol { ... };
_symbolTable.Define(newSymbol);  // Replaces existing
```

**Why?**
- Enables Python-like dynamic typing within static bounds
- Prevents accidental const mutation
- Balances flexibility and safety

## Debugging Tips

### 1. Enable Verbose Logging

The TypeChecker uses `ICompilerLogger` extensively:

```csharp
_logger.LogDebug($"Type checking function: {functionDef.Name}");
_logger.LogInfo("Type checking module");
```

To debug, inject a logger with higher verbosity when creating the TypeChecker.

### 2. Inspect SemanticInfo Cache

If type checking produces unexpected results, inspect the cached types:

```csharp
// In debugger or test:
var exprType = _semanticInfo.GetExpressionType(someExpression);
Console.WriteLine($"Cached type: {exprType?.GetDisplayName()}");
```

### 3. Check Symbol Table State

Scope-related issues? Inspect the symbol table:

```csharp
var symbol = _symbolTable.Lookup("variableName");
Console.WriteLine($"Symbol type: {symbol?.GetType().Name}");
if (symbol is VariableSymbol varSymbol)
    Console.WriteLine($"Variable type: {varSymbol.Type.GetDisplayName()}");
```

### 4. Trace Type Narrowing

Type narrowing issues? Add debug output:

```csharp
foreach (var kvp in _narrowedTypes)
{
    Console.WriteLine($"Narrowed: {kvp.Key} -> {kvp.Value.GetDisplayName()}");
}
```

### 5. Common Error Patterns

**Problem:** "Undefined identifier" for a variable that should exist
- **Check:** Is the variable defined in the current scope? Use `_symbolTable.Lookup(name, searchParents: false)`
- **Check:** Did you enter/exit scopes correctly?

**Problem:** Type narrowing not working
- **Check:** Is `ExtractNarrowedTypes` recognizing the pattern?
- **Check:** Are narrowed types being saved/restored correctly?

**Problem:** Function overload resolution failing
- **Check:** Is the function registered with multiple overloads in `BuiltinRegistry`?
- **Check:** Do argument types match expected parameter types?

### 6. Add Assertions

When in doubt, add assertions to validate assumptions:

```csharp
System.Diagnostics.Debug.Assert(_currentFunctionReturnType != null, 
    "Return statement outside function should have been caught earlier");
```

## Contribution Guidelines

### Adding Support for a New Statement Type

1. **Add case to `CheckStatement()`:**
   ```csharp
   case NewStatement newStmt:
       CheckNewStatement(newStmt);
       break;
   ```

2. **Implement checker method:**
   ```csharp
   private void CheckNewStatement(NewStatement stmt)
   {
       // Validate statement semantics
       // Check child expressions
       // Add errors if invalid
   }
   ```

3. **Add tests** in `Sharpy.Compiler.Tests/Semantic/TypeCheckerTests.cs`

### Adding Support for a New Expression Type

1. **Add case to `CheckExpression()`:**
   ```csharp
   NewExpression newExpr => CheckNewExpression(newExpr),
   ```

2. **Implement type inference:**
   ```csharp
   private SemanticType CheckNewExpression(NewExpression expr)
   {
       // Infer the type of this expression
       return inferredType;
   }
   ```

3. **Ensure caching works** (it should automatically via the cache check at the start of `CheckExpression()`)

### Extending Type Narrowing

To support new narrowing patterns:

1. **Add pattern recognition to `ExtractNarrowedTypes()`:**
   ```csharp
   // Handle 'len(x) > 0' pattern
   if (condition is BinaryOp { Operator: BinaryOperator.GreaterThan } binOp)
   {
       if (binOp.Left is FunctionCall { Function: Identifier { Name: "len" } } call)
       {
           // Extract variable name and narrow to non-empty type
           // ...
       }
   }
   ```

2. **Test thoroughly** - type narrowing is subtle and error-prone

### Improving Error Messages

Good error messages are crucial:

```csharp
// ❌ Bad: Vague
AddError("Type error");

// ✅ Good: Specific with context
AddError($"Cannot assign type '{valueType.GetDisplayName()}' to " +
         $"variable of type '{targetType.GetDisplayName()}'",
         assignment.LineStart, assignment.ColumnStart);
```

**Tips:**
- Include actual vs. expected types
- Provide line and column numbers
- Suggest fixes when possible
- Match Python/TypeScript error message style

### Common Pitfalls

1. **Forgetting to enter/exit scopes:**
   - Always pair `EnterScope()` with `ExitScope()`
   - Use try/finally if exceptions are possible

2. **Not saving/restoring narrowed types:**
   - Always save before modifying `_narrowedTypes`
   - Always restore after checking branches

3. **Mutating shared state:**
   - TypeChecker is stateful (holds `_errors`, `_narrowedTypes`, etc.)
   - Don't reuse instances across modules

4. **Assuming expressions are cached:**
   - Always check for null from `GetExpressionType()`
   - Cache misses can happen

### Testing Guidelines

When adding new type checking logic:

1. **Write positive tests** (valid code should pass):
   ```csharp
   [Fact]
   public void TestValidAssignment()
   {
       var source = "x: int = 42";
       var errors = TypeCheck(source);
       Assert.Empty(errors);
   }
   ```

2. **Write negative tests** (invalid code should fail):
   ```csharp
   [Fact]
   public void TestInvalidAssignment()
   {
       var source = "x: int = \"not an int\"";
       var errors = TypeCheck(source);
       Assert.NotEmpty(errors);
       Assert.Contains("Cannot assign type 'str'", errors[0].Message);
   }
   ```

3. **Test edge cases:**
   - Empty collections
   - Nested scopes
   - Type narrowing interactions
   - Overload resolution ambiguities

4. **Follow existing patterns** in `Sharpy.Compiler.Tests/Semantic/TypeCheckerTests.cs`

## Summary

`TypeChecker.cs` is a critical component that ensures type safety in Sharpy programs. It:

- **Validates** all type usage in the AST
- **Infers** types for expressions
- **Narrows** types in conditional branches
- **Supports** advanced features like tuple unpacking, comprehensions, and overload resolution
- **Accumulates** errors for better developer experience

Understanding this file is essential for:
- Adding new type checking rules
- Debugging type errors in the compiler
- Extending the type system
- Contributing to Sharpy's semantic analysis

When working with this code, remember:
- Types are cached in `SemanticInfo`
- Scopes are managed by `SymbolTable`
- Errors are accumulated, not thrown immediately
- Type narrowing is scope-based and must be saved/restored
- The code is designed for clarity over cleverness

Happy type checking! 🎯
