  ---
  Critical Analysis of Dunder Method Specification

  1. The __eq__ / Equals Duality Problem

  Issue: The spec says __eq__ generates both operator== and override bool
  Equals(U rhs), with the former invoking the latter.

  Pitfalls:
  - Signature mismatch with System.Object.Equals(object): The spec shows
  Equals(U rhs) but .NET expects Equals(object other). This creates ambiguity ‚ÄĒ
  are you generating an overload or the override?
  - GetHashCode contract violation: If someone defines __eq__ without __hash__,
  you violate the .NET contract that equal objects must have equal hash codes.
  The compiler should either:
    - Error if __eq__ is defined without __hash__
    - Auto-generate a warning
    - Synthesize a default __hash__ (dangerous)
  - Reference equality vs value equality: When __eq__ is missing, what does ==
  do? In Python it's identity; in C# for reference types it's also reference
  equality, but for record types it's value equality. This needs explicit
  documentation.

  Axiom tension: This is Axiom 1 (.NET) vs Axiom 2 (Python). Python's __eq__
  returning NotImplemented for incompatible types doesn't exist in C# ‚ÄĒ you must
   return bool. The spec shows return False for type mismatches, which is
  correct but differs from Python semantics where the reflected __eq__ of the
  other type would be tried.

  ---
  2. The Reflected Operator Conundrum

  Issue: The spec defines __radd__, __rsub__, etc., which generate operators
  with swapped operand positions.

  Pitfalls:
  - Conflict resolution is undefined: What happens if both types define
  conflicting operators?
  class A:
      def __add__(self, other: B) -> A: ...

  class B:
      def __radd__(self, other: A) -> B: ...  # Now A + B is ambiguous
  - Python has a precedence rule (subclass wins, then __radd__ if __add__
  returns NotImplemented). C# uses static overload resolution. These are
  fundamentally incompatible.
  - NotImplemented doesn't exist: Python's reflected operators are a fallback
  mechanism when the primary returns NotImplemented. C# has no such concept ‚ÄĒ
  the compiler picks one overload at compile time. This means Sharpy's reflected
   operators are really just "operators defined on the right-hand type" with no
  fallback semantics.

  Recommendation: Be very explicit that __radd__ is not Python's reflected
  operator semantics but simply "define an operator where this type is the RHS."
   Pythonic developers will expect fallback behavior they won't get.

  ---
  3. The __str__ vs __string__ Split

  Issue: The spec defines:
  - __str__ ‚Üí explicit operator Str(T self) (Sharpy's string type)
  - __string__ ‚Üí ToString() + explicit operator string(T self)

  Pitfalls:
  - Cognitive overhead for Python developers: Python has one __str__. Now they
  must learn when to use which.
  - Implicit conversion chains: If Str is implicitly convertible to string (it
  probably is for .NET interop), you get T ‚Üí Str ‚Üí string and also T ‚Üí string.
  This can cause "ambiguous conversion" errors.
  - str() built-in behavior: Which does str(x) call? If it calls __str__, then
  string(x) should call __string__. But the spec doesn't define a string()
  built-in ‚ÄĒ that's C# System.String.

  Axiom tension: This is a necessary compromise for Axiom 1 (.NET interop) but
  hurts Axiom 2 (Pythonic). The split is defensible but the naming is confusing
  ‚ÄĒ __string__ looks like a dunder but it's really a .NET-ism.

  ---
  4. The __bool__ / true/false Operator Problem

  Issue: __bool__ generates both operator true and operator false.

  Pitfalls:
  - C#'s true/false operators are weird: They're used for short-circuit && and
  || on types that don't have & and | operators. Most C# developers don't use
  them.
  - if statement behavior: In C#, if (x) where x is a custom type with operator
  true works. But what about if not x? You need both operators, and the spec
  says false invokes true and negates. That's fine.
  - Truthiness in boolean context: Python's bool(x) calls __bool__, falling back
   to __len__. Does Sharpy synthesize this fallback? The spec doesn't say.

  Missing: What happens if someone defines __len__ but not __bool__? In Python,
  bool([1,2,3]) returns True because len([1,2,3]) != 0. Does Sharpy
  auto-generate operator true/false from __len__? If not, that's a Python
  semantics divergence.

  ---
  5. The Iterator Protocol Impedance Mismatch

  Issue: __iter__ ‚Üí IEnumerable<T>.GetEnumerator(), __next__ ‚Üí MoveNext() +
  Current.

  Pitfalls:
  - Python's StopIteration doesn't exist: Python's __next__ raises
  StopIteration. C#'s MoveNext() returns false. This is a fundamental semantic
  difference that the compiler must bridge.
  - Return type mismatch: Python's __iter__ returns self (the iterator). C#'s
  GetEnumerator() returns IEnumerator<T>. If you define both __iter__ and
  __next__ on the same class (common in Python), what happens?
  - Two-class vs one-class iterators: In Python, it's idiomatic to have one
  class be both iterable and iterator (def __iter__(self): return self). In C#,
  you typically have separate IEnumerable<T> and IEnumerator<T> implementations.
   The spec's Iterator<T> interface tries to bridge this but the mapping is
  subtle.

  Recommendation: The spec should explicitly show how a Python-style
  "self-iterating" class maps to C#, including the state machine that MoveNext()
   implies.

  ---
  6. Cross-Dunder Synthesis Complexity

  Issue: The spec allows self.__lt__(other) or self.__eq__(other) inside __le__.

  Pitfalls:
  - Compiler complexity explosion: You need flow analysis to determine:
    - Is this call site inside a dunder?
    - Is the receiver self or super()?
    - Is it an immediate call (not captured)?
  This is context-sensitive and error-prone to implement.
  - No operator-based alternative: Inside __le__, can you write self < other or
  self == other? If yes, that's cleaner. If no, why not? The spec should
  clarify.
  - Infinite recursion risk: What stops __le__ from calling __le__ on super,
  which calls __le__ on super, etc.? The answer is "don't write that," but the
  compiler can't prevent it.

  ---
  7. The @override Requirement for Object Methods

  Issue: The spec requires @override for __str__, __eq__, __hash__ because they
  override System.Object.

  Pitfalls:
  - Python developers won't expect this: In Python, you just define __str__ and
  it works. Requiring @override is a C#-ism that violates Axiom 2.
  - First-definition vs override is context-dependent: If a base class already
  defines __eq__, you need @override. If not (raw class inheriting from object),
   you still need @override because object.Equals() exists. This is confusing.
  - Inconsistency with operators: You don't need @override for __add__ because
  operators are static and not inherited. But __str__ maps to an instance
  method. The asymmetry is confusing.

  Recommendation: Consider making @override implicit for dunders that map to
  System.Object methods. This is a place where Axiom 2 should arguably win ‚ÄĒ the
   Pythonic expectation is "just define __str__."

  ---
  8. Missing Dunders That Will Cause Friction

  Major gaps:
  - __call__: Not supported because "C# has no callable object protocol." But
  Python developers use this constantly. The workaround "use explicit Invoke()"
  breaks Pythonic idioms. Consider supporting __call__ via a generated Invoke()
  method that the compiler recognizes for obj() syntax.
  - __enter__/__exit__: "Use IDisposable instead" loses the context manager's
  exception handling semantics. Python's __exit__ receives exception info and
  can suppress exceptions by returning True. IDisposable.Dispose() cannot.
  - __repr__: Not supported. But __str__ and __repr__ have different purposes in
   Python ‚ÄĒ __str__ for users, __repr__ for developers. Collapsing them loses
  expressiveness.

  ---
  9. The __getitem__ / __setitem__ Indexer Problem

  Issue: These map to C# indexers.

  Pitfalls:
  - Slice notation: Python's x[1:3] calls __getitem__ with a slice object. C#
  indexers don't support this natively (until C# 8's ranges). How does Sharpy
  handle x[1:3]? The spec doesn't say whether __getitem__(self, key: slice) ->
  ... is supported.
  - Multi-dimensional indexing: Python allows x[1, 2] which passes a tuple. C#
  allows x[1, 2] as multi-dimensional indexer. These have different semantics.
  Which does Sharpy use?
  - Property vs indexer: __len__ maps to Count property. __getitem__ maps to
  indexer. But what if someone wants a read-only indexer? The spec shows { get;
  } and { set; } separately, implying you can define one without the other.

  ---
  10. Implicit Interface Implementation Risk

  Issue: Several dunders synthesize interface implementations (IEnumerable<T>,
  IEquatable<T>, etc.).

  Pitfalls:
  - Surprise interface compliance: A Sharpy developer defines __iter__ for
  foreach support. They now implicitly implement IEnumerable<T>. Their type is
  now assignable to IEnumerable<T>, which may not be intended.
  - Interface method visibility: Explicit interface implementations in C# are
  private by interface. If __iter__ generates IEnumerable<T>.GetEnumerator(),
  you can't call obj.GetEnumerator() directly ‚ÄĒ only
  ((IEnumerable<T>)obj).GetEnumerator().
  - Conflicting inheritance: What if a Sharpy class inherits from a .NET class
  that already implements IEnumerable<T>? Does __iter__ override, shadow, or
  conflict?

  ---
  Long-Term Language Direction Concerns

  Extensibility of Built-in Dispatch

  The spec says len(x) dispatches to __len__ for Sharpy types but .Count for
  .NET types. This hard-coded dispatch logic will grow unwieldy as you add more
  built-ins. Consider a protocol-based dispatch system where the compiler checks
   for interface compliance rather than name-based dunders.

  Operator Precedence vs Python

  The spec says C# static resolution rules apply. But C# and Python have
  different operator precedence for some cases (e.g., ** vs ^ for exponentiation
   ‚ÄĒ though ** isn't supported). More importantly, C# doesn't have Python's "try
   __radd__ if __add__ fails" semantics. Document this clearly because it will
  bite Python developers.

  Async/Generator Dunders

  The spec lists __aenter__, __aexit__, __aiter__, __anext__, __await__ as "not
  supported yet." These are critical for async Python idioms. The implementation
   will be complex because C# async is fundamentally different (state machines,
  Task<T>, IAsyncEnumerable<T>). Plan this carefully ‚ÄĒ retrofitting async
  dunders later may require semantic model changes.

  ---
  Summary: Top 5 Actionable Concerns

  1. __eq__ without __hash__ must be an error ‚ÄĒ .NET contract requires it
  2. Reflected operators need explicit documentation that they're NOT Python's
  fallback semantics
  3. @override for __str__/__eq__/__hash__ is un-Pythonic ‚ÄĒ consider making it
  implicit
  4. Cross-dunder synthesis context rules add significant compiler complexity ‚ÄĒ
  document the exact rules exhaustively
  5. __call__ omission will frustrate Python developers ‚ÄĒ consider supporting it
   via Invoke() synthesis

  The specification is generally well-thought-out and correctly prioritizes
  Axiom 1 (.NET) where necessary. The main risks are around Pythonic developers'
   expectations not matching C# semantics, particularly for reflected operators,
   equality contracts, and iterator protocols.

  ---
  On __eq__ overload synthesis

  The "synthesize a dispatcher Equals(object other)" approach is sound but has
  subtle ordering issues:

  class Foo:
      def __eq__(self, other: Foo) -> bool: ...      # First
      def __eq__(self, other: Bar) -> bool: ...      # Second
      def __eq__(self, other: IComparable) -> bool: ... # Third

  Generated dispatcher:
  public override bool Equals(object other)
  {
      if (other is Foo f) return Equals(f);      // First
      if (other is Bar b) return Equals(b);      // Second
      if (other is IComparable c) return Equals(c); // Third
      return false;
  }

  Concern: If Bar : Foo, then declaration order matters critically ‚ÄĒ a Bar
  instance would match the Foo branch first. Python developers might expect
  "most specific type wins" but you're proposing "declaration order wins."

  This is defensible (explicit > implicit), but worth documenting clearly. You
  might also want a compiler warning if a more-derived type appears after a
  less-derived type in the overload list.

  Also: The return false fallback diverges from Python where x == y with
  incompatible types tries y.__eq__(x) (reflected). In Sharpy, false is final.
  That's fine for Axiom 1, just needs documentation.

  ---
  On str being System.String

  That simplifies things considerably. The __str__ ‚Üí ToString() mapping becomes
  clean:

  - __str__(self) -> str ‚Üí public override string ToString()
  - str(x) built-in ‚Üí calls ToString() on any type

  No explicit conversion operator needed. One dunder, one generated method. Much
   more Pythonic.
