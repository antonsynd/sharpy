# Implementation Plan: R-0.1.5.2 - Add Keyword Argument Validation

## Summary

This task adds missing validations for keyword arguments in function calls:
1. Detect positional arguments following keyword arguments
2. Detect duplicate keyword arguments (same name specified twice)
3. Consider C# keyword name mangling for parameter names

## Current State Analysis

### Parser (`Parser.cs:1994-2054`)
The parser currently:
- Correctly parses both positional and keyword arguments
- Stores them in separate lists (`Arguments` and `KeywordArguments`)
- **Issue**: Does NOT enforce ordering - allows `foo(x=1, 2)` to parse successfully with `2` going to the `Arguments` list

### Type Checker (`TypeChecker.cs:1400-1601`)
Current validations:
- Unknown keyword argument name → Error ✅
- Parameter already provided positionally → Error ✅
- Type mismatch → Error ✅

**Missing validations**:
1. Positional after keyword → No error (should error)
2. Duplicate keyword argument names → Wrong error (dictionary overwrites, counts as 1 arg)

### Test Evidence
```
foo(a=1, 2) → "Argument 'a' was already provided positionally" (WRONG - should be "positional after keyword")
foo(a=1, a=2) → "Function expects 2 arguments but got 1" (WRONG - should be "duplicate keyword argument")
```

## Implementation Plan

### Step 1: Add Duplicate Keyword Argument Detection (TypeChecker.cs)

**Location**: `CheckFunctionCall()` at line ~1413, before populating `kwargTypes` dictionary

**Change**: Add a HashSet to track seen keyword argument names and report duplicates.

```csharp
// Check keyword arguments and collect their types
var kwargTypes = new Dictionary<string, SemanticType>();
var seenKwargNames = new HashSet<string>();  // NEW
foreach (var kwarg in call.KeywordArguments)
{
    // NEW: Check for duplicate keyword arguments
    if (!seenKwargNames.Add(kwarg.Name))
    {
        AddError($"Keyword argument '{kwarg.Name}' specified multiple times",
            kwarg.LineStart, kwarg.ColumnStart);
        continue;  // Skip adding to dict to avoid overwriting
    }
    kwargTypes[kwarg.Name] = CheckExpression(kwarg.Value);
}
```

### Step 2: Add Positional-After-Keyword Detection (Parser.cs)

**Location**: `ParsePostfix()` at line ~2026-2028, in the argument parsing loop

**Change**: Track whether we've seen a keyword argument, and error if a positional arg follows.

```csharp
var args = new List<Expression>();
var kwargs = new List<KeywordArgument>();
bool seenKeywordArg = false;  // NEW

if (Current.Type != TokenType.RightParen)
{
    do
    {
        // Check for keyword argument
        if (Current.Type == TokenType.Identifier && Peek().Type == TokenType.Assign)
        {
            seenKeywordArg = true;  // NEW
            // ... existing keyword arg parsing ...
        }
        else
        {
            // NEW: Check for positional after keyword
            if (seenKeywordArg)
            {
                AddError("Positional argument cannot follow keyword argument");
            }
            args.Add(ParseExpression());
        }
        // ... rest of loop ...
    } while (true);
}
```

**Note**: Need to verify `AddError` method signature in Parser.cs.

### Step 3: Apply Keyword Name Mangling to Kwargs (CodeGen/RoslynEmitter.cs)

**Location**: `GenerateFunctionCall()` around line 2009-2024

**Current code**:
```csharp
var keywordArgs = call.KeywordArguments.Select(kwarg =>
    Argument(GenerateExpression(kwarg.Value))
        .WithNameColon(NameColon(IdentifierName(kwarg.Name))));
```

**Change**: Apply `NameMangler.EscapeKeywordIfNeeded()` to keyword argument names:

```csharp
var keywordArgs = call.KeywordArguments.Select(kwarg =>
    Argument(GenerateExpression(kwarg.Value))
        .WithNameColon(NameColon(IdentifierName(
            NameMangler.EscapeKeywordIfNeeded(kwarg.Name)))));
```

This ensures that if a Sharpy function has a parameter named `class`, and it's called with `foo(class=42)`, the generated C# uses `@class`.

**Note**: Currently Sharpy doesn't allow C# keywords as parameter names (lexer rejects them). This may need coordination with lexer changes if we want to support this in the future.

### Step 4: Similar Changes for Pipe-Forward Calls (TypeChecker.cs)

**Location**: `CheckPipeForwardWithFunctionCall()` around line 1199-1222

Apply the same duplicate detection logic for keyword arguments in pipe-forward calls.

## Risks and Questions

### Questions for Clarification

1. **Error location for parser errors**: Should the "positional after keyword" error point to:
   - The positional argument that's out of place? (preferred)
   - The entire function call?

2. **Should parser continue after positional-after-keyword?**:
   - Option A: Stop parsing arguments (safer, prevents cascade errors)
   - Option B: Continue parsing but skip adding the invalid arg
   - Recommendation: Option B to allow reporting multiple errors

3. **C# keyword support in parameter names**:
   - Currently, the lexer rejects C# keywords like `class` as identifiers
   - Is supporting `def foo(class: int)` in scope for this task?
   - Recommendation: Out of scope - note in task for future work

### Potential Risks

1. **Parser error handling**: Parser may not have same error infrastructure as TypeChecker. Need to verify `AddError` method exists and its signature.

2. **Error message consistency**: New error messages should match existing style/format.

3. **Test regression**: The existing `Error_PositionalAfterKeyword_ReportsError` test passes currently (compilation fails). After the fix, we need to ensure it still fails but with the correct error.

## Implementation Order

1. **Start with TypeChecker changes** (Step 1 & 4) - lowest risk, self-contained

4. **Verify existing test** `Error_PositionalAfterKeyword_ReportsError` produces correct message
