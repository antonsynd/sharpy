# Sharpy Compiler Health Audit — 2026-02-14

## Executive Summary

The Sharpy compiler is in **strong health** overall. All 6,889 tests pass with zero failures, the .NET compliance is excellent (clean C# 9.0, netstandard2.0/2.1 builds with zero warnings), and the codebase demonstrates mature engineering practices: well-structured partial class splits for large components, centralized dunder name constants, comprehensive diagnostic codes, and deliberate thread-safety annotations for future parallelism.

The most significant architectural concern is a **bidirectional dependency between Semantic and CodeGen** (4 files in Semantic import CodeGen types), which should be resolved by extracting shared utilities (`NameMangler`, `DunderMapping`) into a neutral namespace. The **parser lacks a recursion depth guard**, which could cause stack overflow on malicious input. Two multi-file compilation bugs (package re-export and import alias) are documented in `.skip` files but need GitHub issues.

Testing is comprehensive with 5,654+ test methods and 537 file-based fixtures, though Sharpy.Core's 8 module directories (2,105 lines: Datetime, Itertools, Math, Operator, Random, Collections, Sys) have **zero dedicated tests**. The codebase is well-positioned for future tooling (LSP, parallel builds, debugging) with several components already designed for it, though REPL support would require significant architectural changes.

## Critical Findings

1. **Bidirectional Dependency: Semantic <-> CodeGen** — 4 Semantic files import `Sharpy.Compiler.CodeGen` (for `NameMangler`, `DunderMapping`), while all 12 CodeGen files import Semantic types. This creates circular coupling in what should be a unidirectional pipeline. Extract shared utilities to a neutral namespace.

2. **Parser Stack Overflow Risk** — No recursion depth guard. 500 nested parentheses or 200 nested list literals cause `StackOverflowException`. Two tests are skipped for this reason. Malicious input could crash the compiler.

3. **Fragile Diagnostic Deduplication** — TypeChecker deduplicates diagnostics by exact `(Line, Column, Message)` triple. Any wording difference produces duplicates. Should use diagnostic code + span identity instead.

## Warnings

1. **Discovery/TypeMapper missing `Sharpy.Dict` handling** — Handles `Sharpy.List<T>` and `Sharpy.Set<T>` but not `Sharpy.Dict<K,V>`, creating an asymmetry.
2. **CLAUDE.md type mapping documentation is stale** — States `list[T]` maps to `System.Collections.Generic.List<T>` but actual mapping is `Sharpy.List<T>`.
3. **Lexer.cs is 1,941 lines** — The only major component not using the partial class pattern.
4. **12 methods exceed 200 lines** — `CheckFunctionCall()` (528 lines), `ParsePrimary()` (524 lines), `Compile()` (445 lines) are the worst.
5. **Repeated `AddError` pattern** — 6+ classes independently define their own `AddError` private method.
6. **`Compiler.Compile()` constructs `CompilationResult` 10 times** — Error-prone; adding a field requires 10 updates.
7. **81 hardcoded strings in CodeGen/** — `"self"`, `"cls"`, decorator names should be constants.
8. **2 multi-file compilation bugs** — Package re-export and import alias bugs documented in `.skip` files but lack GitHub issues.
9. **2 parser TODO comments lack GitHub issues** — Stack overflow bugs should be tracked.
10. **Sharpy.Core modules (2,105 lines) have zero dedicated tests** — Datetime, Itertools, Math, Operator, Random, Collections, Sys are untested.
11. **4 validators lack dedicated unit tests** — DecoratorValidator, DunderInvocationValidator, InterfaceConflictValidator, PropertyValidator.

## Opportunities

1. **Split `TypeChecker.Expressions.cs` (2,276 lines)** and **`RoslynEmitter.Expressions.cs` (2,115 lines)** into sub-concerns.
2. **Rebalance `Str.cs` partial split** — Main file has 85% of code (1,404 of 1,652 lines).
3. **Extract `Semantic/Registry/` sub-namespace** — 6 registry files could reduce the 60-file Semantic directory.
4. **Remove or document `StringExtensions` dual API** with `Str` type.
5. **Reclaim or document SPY0265** — Gap in diagnostic code numbering.
6. **Add position-to-node index** for LSP readiness.
7. **Consider `gofmt`-style formatter** — Current architecture supports AST-based formatting but not roundtrip formatting.

## Detailed Findings

### Architecture & Modularity

**Critical:**
- **C1. Bidirectional Dependency: Semantic <-> CodeGen** — `CodeGenInfoComputer.cs`, `ExecutionOrderAnalyzer.cs`, `NamingConventionValidator.cs`, and `DunderInvocationValidator.cs` in Semantic import CodeGen types (`NameMangler`, `DunderMapping`). Extract to shared namespace.
- **C2. Fragile Diagnostic Deduplication** — TypeChecker deduplicates by exact `(Line, Column, Message)` triple instead of diagnostic identity.

**Warning:**
- **W1. Large Monolithic Files** — Lexer.cs (1,941), ProjectCompiler.cs (1,758), Compiler.cs (938), NameResolver.cs (998), ImportResolver.cs (877), TypeInferenceService.cs (841).
- **W2. Monster Methods** — DumpNode (690), CheckFunctionCall (528), ParsePrimary (524), Compile (445), ParseSetOrDictTypeShorthand (357), CheckFunction (333), NextToken (281).
- **W3. Repeated `AddError` Pattern** — 6+ independent implementations across semantic pass classes.
- **W4. `CompilationResult` constructed 10 times** in `Compiler.Compile()`.
- **W5. Imbalanced Str.cs partial split** — 85% of code in main file.
- **W6. Semantic/ directory has 60 files** — Consider sub-namespaces for registries.

**Opportunity:**
- O1. `StringExtensions` vs `Str` parallel APIs create dual maintenance burden.
- O2-O3. `RoslynEmitter.Expressions.cs` (2,115) and `TypeChecker.Expressions.cs` (2,276) could be split further.
- O4. Parser/ is cleanly layered (no back-references).
- O5. `AstDumper.DumpNode()` (690 lines) could use visitor pattern.

**Test Results: 6,889 total | 6,876 passed | 0 failed | 13 skipped**

**Warning:**
1. **Parser stack overflow** — No recursion depth guard. 500 nested parens / 200 nested lists crash.

3. **SPY0265 gap** — Missing diagnostic code in sequence.

**SemanticInfo Integrity:** Sound. ReferenceEqualityComparer used correctly for record-based AST nodes. Thread-safety limitations documented.
