# Sharpy Compiler Health Audit — 2026-05-26

## Executive Summary

The Sharpy compiler is in excellent health. All **14,666 tests pass** with zero failures across all four test suites (Compiler, Core, Stdlib, LSP). Architecture is clean with no circular dependencies between compiler phases. .NET compliance is spotless — zero C# 9.0 or netstandard2.1 violations. TODO/FIXME compliance remains at 100% (all reference GitHub issues). The codebase has grown to ~120K+ lines across 920 source files with 1,883 integration test fixtures.

The only critical finding is that **Sharpy.Cli has zero unit tests** (12 source files, 0 test files). Key warnings include 10 skipped formatting fixtures (formatter is incomplete), the SymbolTable blocking parallel compilation (#610), and a missing `textDocument/documentLink` LSP handler. The compiler's future tooling readiness is strong — REPL, formatter infrastructure, and debugger source mapping are all implemented and ready.

Overall trajectory: healthy, well-maintained, and well-tested. The action items are incremental improvements, not structural repairs.

## Critical Findings

1. **CLI has zero tests** — `Sharpy.Cli` (12 source files) has no corresponding test project. The CLI is the primary user entry point (`run`, `emit`, `project` commands, argument parsing). End-to-end CLI behavior is completely untested at the unit level.

## Warnings

1. **10 Formatting fixtures skipped** — All `.skip` files in the repo are Formatting tests (7 in Compiler.Tests, 3 in Stdlib.Tests). The formatter feature is broadly disabled/incomplete.
2. **SymbolTable not thread-safe** — Uses `Stack<Scope>` and `Dictionary<string, Scope>` (non-concurrent). Blocks parallel compilation. Tracked as issue #610 with a design doc at `docs/design/parallel-compilation.md`.
3. **2 property/stress tests permanently skipped** — `HighFuelPrograms` and `DeepNestingPrograms` lexer/parser crash resistance tests are disabled. Worth confirming these are intentionally skipped for perf/time rather than masking a known hang.
4. **`textDocument/documentLink` not implemented** — Only missing standard LSP feature. Import/module paths in `.spy` files are not clickable.
5. **Oversized files trending toward 2K lines** — `TypeChecker.Expressions.Access.Calls.cs` (2,032), `DiagnosticExplanations.cs` (2,027), `TypeChecker.Definitions.cs` (1,911), `Parser.Statements.cs` (1,818). Already partial-split but individual partials growing large.
6. **DecoratorValidator (1,391 lines)** is the only large validator not partial-split.
7. **PositionConverterTests has only 4 cases** — Thin coverage for foundational code every LSP handler depends on.
8. **Stdlib test ratio lowest (0.71x)** — 31 modules with NuGet-backed complexity (numpy/MathNet, sqlite3) have the thinnest test coverage relative to surface area.

## Opportunities

1. **Formatter infrastructure is ready** — Trivia preservation fully implemented (Lexer, AST nodes), just needs the formatter tool built on top.
2. **REPL infrastructure is ready** — `ReplSession` with statement classification, partial input handling, and async evaluation is complete.
3. **Debugger source mapping is ready** — Enhanced `#line` directives with column ranges, `TextSpan` preservation through codegen.
4. **Parallel type checking is immediately possible** — `SemanticInfo` and `SemanticBinding` use `ConcurrentDictionary` throughout. Only `SymbolTable` blocks full parallel compilation.
5. **DiagnosticExplanations.cs (2,027 lines)** could be externalized to a resource file.
6. **Consider shared AST dispatch table** — 80+ switch statements in RoslynEmitter, 33 in TypeChecker suggest repeated node-kind branching.
