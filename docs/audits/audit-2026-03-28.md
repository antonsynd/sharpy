## Critical Findings

### C-1: `ContainsReturnStatement` Reimplements `StatementWalker`

**File:** `src/Sharpy.Compiler/CodeGen/RoslynEmitter.Statements.cs:1907–1955`

A hand-rolled recursive AST walker in codegen that is structurally identical to `StatementWalker.Any()` in `Shared/StatementWalker.cs`. The semantic layer already uses `StatementWalker.Any(stmts, s => s is YieldStatement)` for the analogous `ContainsYield`. The codegen copy skips `WithStatement` bodies and will silently produce incorrect results when new control-flow constructs are added.

**Fix:** Replace with `StatementWalker.Any(statements, s => s is ReturnStatement)`.

### C-2: Narrowing Logic Duplicated Between TypeChecker and RoslynEmitter

**Files:**
- `TypeChecker.Utilities.cs:34–173` — `ExtractNarrowedTypes()`, `CollectNarrowings()`
- `RoslynEmitter.Statements.cs:1138–1280` — `DetectOptionalNarrowings()`, `DetectIsInstanceNarrowings()`

Both implementations pattern-match `is None`, `is not None`, and `isinstance()` in conditions. `SemanticInfo` stores narrowed types but NOT the codegen-time decisions (which branch to emit the cast in). The emitter must re-analyze the condition tree, duplicating structural analysis TypeChecker already did. Already diverging: TypeChecker handles `not (x is None)` via negation inversion; the emitter only handles direct `is None`/`is not None`.

**Fix:** Store narrowing decisions in `SemanticInfo` during TypeChecker — `(variableName, targetType, narrowInThenBranch)` tuples per `IfStatement`/`WhileStatement`. Emitter reads from `SemanticInfo` instead of re-analyzing.
