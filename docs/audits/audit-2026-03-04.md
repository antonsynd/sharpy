# Sharpy Compiler Health Audit — 2026-03-04

**Focus**: Hard-coded type logic generalization + Dunder method spec adherence

## Executive Summary

The compiler is **well-architected** with excellent registry-based design (`PrimitiveCatalog`, `ProtocolRegistry`, `OperatorRegistry`, `BuiltinMethodDefinitions`). Spec-implementation alignment is 95%+. The main findings fall into two categories:

1. **Hard-coded type logic**: Codegen has 4 critical string-comparison patterns for collection types that should use a registry. Semantic analysis is cleaner but has scattered collection-type checks that could be consolidated.
2. **Dunder spec adherence**: 19 of 46 spec'd dunders are supported (27 explicitly unsupported per spec). One real spec violation: `__getitem__`/`__setitem__` emit regular methods instead of C# indexers for user-defined types.

## Critical Findings

### 1. `__getitem__`/`__setitem__` Emit Methods Instead of Indexers (Spec Violation)
- **Spec**: `__getitem__` → `this[K key] { get; }`, `__setitem__` → `this[K key] { set; }`
- **Implementation**: Built-in types get indexers, but user-defined types emit `GetItem()`/`SetItem()` methods
- **Files**: `RoslynEmitter.ClassMembers.cs:189`, `RoslynEmitter.Expressions.Access.cs:880-883`, `RoslynEmitter.Statements.cs:398-401`
- **Impact**: Functional (x[key] still works), but violates spec and breaks .NET interop expectations (consumers of the class can't use indexer syntax from C#)

### 2. Collection Type String Comparisons in Codegen
- **File**: `RoslynEmitter.Expressions.Literals.cs` (lines 34, 41, 78, 85, 129, 135, 194, 231, 268, 333, 353)
- **Issue**: Hard-coded `_targetTypeContext.Name == "list"` / `"dict"` / `"set"` throughout literal/comprehension generation
- **Impact**: Adding any new collection type requires modifying multiple locations
- **Recommendation**: Create a `CollectionTypeRegistry` with metadata (C# type name, spread method, add method)

### 3. Hard-Coded Collection Method Names in Codegen
- **File**: `RoslynEmitter.Expressions.Literals.cs` (lines 59, 152, 387)
- **Issue**: `"Extend"`, `"UnionWith"`, `"Add"` are magic strings
- **Recommendation**: Store in registry keyed by collection type

### 4. Dunder Dispatch If-Chain in Codegen
- **File**: `RoslynEmitter.ClassMembers.cs` (lines 108-192)
- **Issue**: Long if/else chain for dunder-to-codegen mapping. Each new dunder requires modifying this chain.
- **Recommendation**: Registry of `DunderCodeGenHandler` delegates

## Warnings

### 5. Collection Constructor Type Inference (Semantic)
- **File**: `TypeChecker.Expressions.Access.cs:870-875`
- **Issue**: Inline if/else for list/set/dict constructor type arg inference
- **Recommendation**: Extract to `CollectionTypeInference` helper

### 6. Duplicate Iterator Element Type Extraction (Semantic)
- **Files**: `TypeChecker.Utilities.cs:416-447` and `TypeInferenceService.cs:691-760`
- **Issue**: Same list/set/dict/tuple/string element extraction logic in two places
- **Recommendation**: Consolidate into `TypeUtils`; have `ExtractElementType()` delegate to `InferIterableElementType()`

### 7. String Type Comparisons in Validators (Semantic)
- **Files**: `SignatureValidator.cs:297,302`, `ModuleLoader.cs:544-556`, `VarianceValidator.cs:183`
- **Issue**: Hardcoded `"bool"`, `"None"`, `"function"`, `"Tuple"` etc. instead of semantic type checks
- **Recommendation**: Use `TypeUtils` helpers

### 8. Dict View Type Extraction in Codegen
- **File**: `RoslynEmitter.Expressions.Literals.cs:622-631`
- **Issue**: Switch on `gt.Name` for `DictItemsView`, `DictValuesView`, `DictKeyView`
- **Recommendation**: Data-driven element type extractor

### 9. `list.count()`/`list.index()` Return `uint` Instead of `int`
- **File**: `Sharpy.Core/Partial.List/List.Methods.cs:99,121`
- **Issue**: Spec says `int` return type; implementation returns `uint`
- **Recommendation**: Change return type to `int`

## Opportunities

### 10. Empty Literal Handling Consolidation (Semantic)
- **File**: `TypeChecker.Expressions.Literals.cs:13-28, 76-92, 143-159`
- Repeated pattern checking `_expectedType is GenericType expected && expected.Name == BuiltinNames.[Type]`
- Extract to `TryGetExpectedCollectionType()` helper

### 11. Mutable Default Check Data-Driven (Semantic)
- **File**: `DefaultParameterValidator.cs:154-160`
- Three pattern match arms for list/dict/set → replace with `MutableCollectionConstructors` array

### 12. Missing `int.parse()`/`float.parse()` Methods
- Spec defines `int.parse(x)` and `float.parse(x)` returning `Result[T, ValueError]`
- Not yet implemented in `Sharpy.Core`
