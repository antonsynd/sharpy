# Sharpy Compiler Health Audit — 2026-03-04

## Executive Summary

The Sharpy compiler demonstrates **strong architectural foundations** with clear phase separation, immutable AST, type-safe Roslyn-based code generation, and comprehensive test infrastructure. All 7,783 of 7,784 tests pass (99.99%), with the single skip being an acknowledged design limitation. The compiler pipeline (`Lexer → Parser → Semantic → Validation → CodeGen`) is well-organized with principled use of partial classes for large components.

**Key risks** center on: (1) a Parser → Semantic coupling violation that breaks the unidirectional pipeline, (2) ~193 TODO/FIXME comments lacking GitHub issue references (violating project rules), (3) critical test coverage gaps in the Project compilation and incremental build systems, and (4) several files exceeding 2,000 lines that impact maintainability.

**Top priorities**: Remove the Parser → Semantic dependency, create GitHub issues for untracked TODOs, and expand test coverage for multi-file/incremental compilation. The .NET compliance audit found **zero violations** — Sharpy.Core is fully C# 9.0 / netstandard2.0+2.1 compliant, and generated code is verified C# 9.0 compatible.

## Critical Findings

### C1: Parser → Semantic Coupling Violation
**File**: `src/Sharpy.Compiler/Parser/Parser.Types.cs` line 7
**Impact**: Breaks unidirectional pipeline architecture (`Lexer → Parser → Semantic → CodeGen`)
**Details**: `using Sharpy.Compiler.Semantic;` in Parser.Types.cs creates a backward dependency. Investigation shows the import appears unused. Remove immediately or refactor if semantic types are actually needed.
**Effort**: 5 minutes (remove unused using) to 2 hours (if refactoring needed)

### C2: 193+ TODOs Without GitHub Issue References
**Impact**: Violates project rule that "TODO/BUG/FIXME comments must have GitHub issues"
**Details**: Only 7 of 200+ TODO/FIXME/BUG comments follow the `TODO(#123)` format. The remaining ~193 represent undiscoverable deferred work buried in code.
**Effort**: 2-3 hours (batch `gh issue create`)

### C3: Project Compilation System — Critical Test Gap
**Impact**: 4,292 LOC with only ~15 programmatic tests (1:286 ratio)
**Details**: Incremental compilation, cache invalidation, transitive dependency tracking, and symbol serialization have near-zero dedicated tests. This is a user-facing feature with significant correctness risk.
**Effort**: 2-3 weeks for comprehensive coverage

## Warnings

### W1: RoslynEmitter.ClassMembers.cs — 2,448 Lines
Handles constructors, methods, fields, properties, events, and indexers in one file. Should split into ~4 focused partials (Constructors, Methods, Fields, Properties).

### W2: TypeChecker.Expressions.Access.cs — 2,312 Lines
Handles member access, calls, generics, and tagged unions. Extract generic type inference and tagged union handling to separate partials.

### W3: NameResolver.cs Not Partial — 1,347 Lines
Only major semantic component not using partial classes. Split into core/inheritance/utilities.

### W4: Type Discovery System Under-Tested
1,967 LOC with only ~60 tests. Missing: generic type parameter extraction, cache invalidation, assembly loading, nullable vs optional discrimination.

### W5: Sparse Feature Test Fixtures
Single .spy fixture for: lambdas, modules, optional types. Only 2 for context managers. These need expansion to 10+ fixtures each.

### W6: Magic Values in CodeGen/Semantic
200+ instances of hardcoded strings (`"Add"`, `"ToString"`, `"var"`, `"__spy_pm_"`) and numbers (`100`, `200`, `4`) that should be centralized constants.

### W7: Unused Diagnostic Code SPY0521
`TypeReExportNotSupported` defined but never emitted anywhere. Either implement or remove.

## Opportunities

### O1: Parallel Build Execution
`GetParallelizableGroups()` API exists in ProjectModel but ProjectCompiler processes files sequentially. Wiring the parallel executor is a small effort.

### O2: LSP Foundation
SemanticInfo queries, AstPositionService, and ISemanticQuery are ready. Missing cross-file symbol resolution for production LSP.

### O3: Trivia Pipeline Completion
Lexer captures trivia (comments, whitespace) but Parser ignores it. Propagating trivia to AST enables formatter and roundtrip tooling.

### O4: Extract TypeNarrowingAnalyzer
Type narrowing logic in TypeChecker.Utilities.cs could be a standalone testable class.

### O5: DependencyGraph.ToDot() for Visualization
Developer tooling improvement for multi-file project debugging.

#### Coupling Assessment
- **Parser → Semantic**: ❌ Violation detected (`using Sharpy.Compiler.Semantic;` in Parser.Types.cs — appears unused)
- **CodeGen → Semantic + Parser**: ✅ Correct
- **Semantic → Parser**: ✅ Correct

#### Separation of Concerns
- **TypeChecker vs ValidationPipeline**: ✅ Well-separated. TypeChecker handles inference-dependent checks; validators handle post-inference analysis.
- **RoslynEmitter partial boundaries**: ✅ Logically organized with clear responsibility per file.
- **Parser partial boundaries**: ✅ Balanced distribution (Statements, Definitions, Expressions, Primaries, Types).

#### Architecture Strengths
1. **Immutable AST**: All nodes are `record` types — thread-safe, easy to reason about
2. **SemanticInfo Separation**: Annotations stored separately from AST using `ReferenceEqualityComparer`
3. **Validation Pipeline**: Pluggable validators with explicit ordering (`Order` property)
4. **No String Templating**: RoslynEmitter uses `SyntaxFactory` exclusively
5. **Symbol Materialization**: Phased materialization from `SemanticBinding` at phase boundaries
6. **Incremental Compilation**: Cache with SHA-256 content hashing and transitive dependency tracking
