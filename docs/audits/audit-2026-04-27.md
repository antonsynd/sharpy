# Sharpy Compiler Health Audit — 2026-04-27

## Executive Summary

The Sharpy compiler is in **excellent health**. All 11,705 tests pass (1 flaky perf test aside), the codebase has zero C# 9.0 compliance issues, zero coupling violations, 100% TODO/FIXME compliance, and a strong 136% test-to-production line ratio. The architecture is clean — dependency flow (Parser → Semantic → CodeGen) has no back-references, and the partial class decompositions are well-bounded.

The main areas for attention are: (1) a large number of diagnostic codes marked "Active" that are pre-allocated but have explanations written before the emitting code — this is by design but worth documenting, (2) debugger support via `#line` directives is implemented but not yet validated in real debuggers, and (3) parallel compilation is architecturally blocked by the shared SymbolTable.

Overall, this is a mature, well-organized compiler codebase with strong test coverage across all components.

## Critical Findings

**None.** No critical issues found across any audit dimension.

## Warnings

### W1. Flaky Performance Test
- `CachedDiscoveryPerformanceTests.CachedDiscovery_CachedLoad_CompletesWithinTime` failed with 109ms (threshold: 100ms)
- This is a timing-sensitive test that flakes under load; consider raising the threshold or marking it as a known flaky test

### W2. Shared SymbolTable Blocks Parallel Compilation
- `ProjectCompiler` maintains a single `_symbolTable` instance shared across ALL files during semantic analysis
- Files cannot be compiled in parallel during type collection, import resolution, or type checking phases
- Not a bug, but a known architectural limitation for future scaling

### W3. No Statement-Level Parsing API (REPL Blocker)
- Parser only exposes `ParseModule()` — no `ParseStatement()` or `ParseExpression()` entry points
- Blocks future REPL implementation

## Opportunities

### O1. Debugger Validation
- `#line` directives are already emitted by RoslynEmitter — source mapping infrastructure is complete
- Needs validation in real debuggers (VS Code, Rider) to confirm step-into/breakpoint behavior

### O2. Formatter Groundwork
- Lexer captures trivia (`_preserveTrivia`, `TriviaKind.Comment`) but Parser discards it
- Attaching trivia to AST nodes would enable a source formatter (~1 week spike)

### O3. Document Diagnostic Code Allocation Strategy
- 283 codes defined, 275 marked Active, 36 marked Reserved
- All Active codes are properly referenced by constant name — zero orphans
- The allocation includes many pre-allocated codes with explanations written ahead of implementation

**Strengths**:
- Zero mutable static fields across entire compiler
- DiagnosticBag is thread-safe (lock-based)
- SemanticInfo uses ConcurrentDictionary throughout
- CompilerServices is instance-based with "Thread-safe for future parallel compilation" comment
- All AST nodes have TextSpan + line/column positions

**Blockers**:
- Single SymbolTable instance shared across all files (blocks parallel semantic analysis)
- Parser only exposes `ParseModule()` (blocks REPL)
- Trivia discarded during parsing (blocks formatter roundtrip)
