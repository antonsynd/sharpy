# Sharpy Compiler Health Audit — 2026-03-22

## Executive Summary

The Sharpy compiler is in **excellent health**. All 9,758 tests pass (1 intentional skip), the architecture is clean with no circular dependencies, .NET compliance is fully met, and the LSP server is production-ready. The codebase demonstrates strong engineering discipline: 100% TODO/FIXME compliance with GitHub issues, all 253 diagnostic codes actively used, and robust invariant enforcement at every phase boundary.

Key strengths: comprehensive error recovery with transitive propagation tracking, 6 always-on compiler invariants, thread-safe LSP with proper cancellation, and complete debugger/source-map infrastructure. The main areas for improvement are minor: four duplicated "is integer" predicates, a few dead code artifacts, and low test coverage in utility components (Shared, Services, Logging).

Future readiness is strong for debugger support (95% ready) and source maps (100% complete). Parallel builds need a small thread-safety fix in SemanticInfo. REPL and formatter require more substantial work (Q3-Q4 2026).

## Detailed Findings

### Architecture & Modularity

**Overall: Sound.** Pipeline layering is clean with no circular dependencies.

**Layer Boundary Assessment:**

| Boundary | Status |
|----------|--------|
| Parser -> Semantic | Clean — no Semantic imports in Parser/Ast |
| Semantic -> CodeGen | Clean — no CodeGen imports in Semantic |
| CodeGen -> Parser | Expected — reads AST nodes for traversal |
| CodeGen -> Semantic | Controlled — reads SemanticInfo/SemanticBinding, never writes |
| ValidationPipeline -> TypeChecker | Clean — validators don't call TypeChecker |
| LSP -> Compiler | Clean — uses only public API + Services |

**Warnings:**
- **W1**: Three validation methods in `TypeChecker.Utilities.cs` (ValidateConstructorOverloads, ValidateStructRules, ValidateEnumRules) conceptually overlap with ValidationPipeline validators
- **W2**: Four independent "is integer" implementations — `TypeUtils.IsInteger` (broadest, includes CLR types), `PrimitiveCatalog` (via NumericKind), `TypeChecker.Utilities.IsIntType` (only int/long), `RoslynEmitter.Operators.IsIntegerSemanticType` (only int/long). Divergence risk if new integer types added.
- **W3**: Dead `ToPascalCase` in `Compiler.cs:672` duplicates `NameMangler.ToPascalCase`
- **W4**: Dead `_logger` field in `SemanticBinding.cs:48-50` with pragma suppression
