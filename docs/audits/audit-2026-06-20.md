# Sharpy Compiler Health Audit — 2026-06-20

## Executive Summary

The Sharpy compiler is in **excellent health**. All 16,390 tests pass with zero failures. The architecture is clean with no circular dependencies between major subsystems (Parser, Semantic, CodeGen). .NET compliance is perfect — Sharpy.Core fully satisfies C# 9.0/netstandard2.1 constraints, and generated code is compliant. The LSP server scores 9/10 with 29 of 31 handlers implemented and tested, strong thread safety, and correct position conversion.

The main areas for improvement are: (1) testing gaps in imports, cross-module inheritance, and type narrowing; (2) low warning/snapshot test coverage (4.2% and 5.2%); (3) a few large files that could benefit from further splitting; and (4) future readiness work for parallel builds (SymbolTable thread safety) and debugger source mapping.

**Overall Grade: A-** — Production-quality compiler with strong test coverage, clean architecture, and excellent .NET compliance. Deductions for testing gaps in key features and partial future readiness.

## Critical Findings

**None.** No critical issues were identified across any of the seven audit dimensions. This is a strong indicator of codebase health.

## Detailed Findings

### Architecture & Modularity

**Overall: Clean.** No circular dependencies. Pipeline direction (Parser -> Semantic -> CodeGen) is strictly maintained.

**75 files exceed 500 lines** in `src/Sharpy.Compiler/`. The top concern is `TypeChecker.Expressions.Access.Calls.cs` at 2,891 lines — the largest non-test file. It mixes overload resolution (270 lines), constructor resolution, lambda calls, and IIFE calls. Natural split points exist.

**Coupling summary:**

| Direction | Status |
|-----------|--------|
| Parser -> Semantic | None (correct) |
| Parser -> CodeGen | None (correct) |
| Semantic -> Parser.Ast | Expected (traverses AST) |
| Semantic -> CodeGen | None (correct) |
| CodeGen -> Semantic | Expected (reads SemanticInfo) |
| Diagnostics -> Semantic/CodeGen | W6: CompilerInvariants.cs breaks this |

**Key warnings:** Diagnostic overlap between TypeChecker and OperatorValidator (W3), four parallel `AddError` implementations (W4), `UnwrapNullable` duplication (W5).

### .NET Compliance

**Overall: Full compliance.** No issues detected.

- **C# 9.0 compliance in Sharpy.Core:** Zero file-scoped namespaces, zero global usings, zero record structs, zero `required` keywords, zero raw string literals across all 168 .cs files
- **Generated code compliance:** Block-scoped namespaces, explicit usings, no C# 10+ features, `{ get; private set; } = default!;` pattern (C# 9.0)
- **Type mapping completeness:** TypeSyntaxMapper maps all 18 SemanticType variants exhaustively. ClrTypeMapper provides bidirectional CLR-to-SemanticType mapping with thread-safe caching
- **netstandard2.1 compatibility:** No unguarded .NET 6+ APIs in Sharpy.Core. `Microsoft.Bcl.AsyncInterfaces` correctly conditioned for netstandard2.1
- **PrimitiveCatalog:** All 27 primitive type registrations map correctly to C# keywords

### LSP Health

**Overall: 9/10.** Production-ready with excellent thread safety.

**Handler coverage:** 29 of 31 handlers implemented and tested. Missing: `DeclarationHandler` (optional), `willSave/willSaveWaitUntil` (optional). All 7 refactoring providers tested.

**Thread safety:** Proper `SemaphoreSlim`, `Volatile.Read/Write`, `ConcurrentDictionary` usage throughout. `CancellableAnalysisScope` uses CAS loop for atomic swap. Double-checked locking in `GetOrRunAnalysisAsync`. No race conditions identified.

**Position conversion:** Correct 0-based (LSP) to 1-based (compiler) conversion with clamping. Round-trip tested.

**Diagnostic publishing:** Correct severity mapping, generated source rerouting, transition hint filtering.

**Key limitation:** Project reanalysis is full (not incremental) on every file change — acceptable for current scale but will need addressing for large projects.

### Future Readiness

| Dimension | Rating | Key Evidence |
