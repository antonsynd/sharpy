# Sharpy Compiler Health Audit â€” 2026-02-27

## Focus: Type-Specific Hardcoded Logic That Should Be Generalized

## Executive Summary

The Sharpy compiler has significant amounts of type-specific hardcoded logic scattered across the TypeChecker, RoslynEmitter, validators, and type inference service. The core problem is the **absence of a method registry for built-in collection types** (dict, list, set, str). While the compiler has registries for operators (`OperatorRegistry`), protocols (`ProtocolRegistry`), and builtin functions (`BuiltinRegistry`), there is no equivalent registry that describes the methods available on built-in types, their signatures, overloads, or return types.

This gap forces type-specific logic to be hardcoded throughout the compiler pipeline â€” from the TypeChecker manually constructing return types for `dict.get()`, `dict.items()`, `dict.keys()`, `dict.values()`, to the OperatorValidator and ProtocolValidator maintaining parallel switch statements that enumerate which dunders each built-in type supports, to the RoslynEmitter checking type names when generating collection literals.

The most impactful fix would be introducing a **BuiltinMethodRegistry** that describes method signatures, overloads, and return type computation for built-in types. This would centralize scattered logic, make the compiler more maintainable, and make it straightforward to add new built-in type methods.

## Critical Findings

### C1. No Method Registry for Built-in Types

**The root cause of most findings in this audit.** There is no centralized registry describing the methods available on `dict`, `list`, `set`, `str`, and other built-in types. The `BuiltinRegistry` only covers top-level builtin *functions* (like `len()`, `print()`), not methods on types.

**Impact:** Every time the compiler needs to know about a built-in type method (its signature, return type, arity), the logic must be hardcoded at the point of use. This is the pattern the user identified with `dict.get()`.

**Current scattered locations:**
- `TypeChecker.Expressions.Access.cs:293-355` â€” `ResolveDictMethodType()` and `ResolveDictMethodCall()` hardcode dict method signatures
- `TypeInferenceService.cs:596-604` â€” hardcodes dict view type element inference
- `TypeInferenceService.cs:698-709` â€” hardcodes dict index access return type
- `OperatorValidator.cs:420-425` â€” hardcodes which operators each collection supports
- `ProtocolValidator.cs:325-337` â€” hardcodes which protocols each collection supports
- `TypeChecker.Utilities.cs:188` â€” hardcodes covariance only for `list` and `set`

### C2. Dict-Specific Return Type Resolution in TypeChecker

**Files:** `TypeChecker.Expressions.Access.cs:293-355`

Two dedicated methods â€” `ResolveDictMethodType()` and `ResolveDictMethodCall()` â€” exist solely to compute return types for dict methods. This is exactly the user's observed issue:

```csharp
// Lines 293-329: ResolveDictMethodType - hardcodes items(), keys(), values() return types
private static SemanticType? ResolveDictMethodType(GenericType dictType, string methodName)
{
    return methodName switch
    {
        BuiltinNames.DictMethodItems => new FunctionType { ... },
        BuiltinNames.DictMethodKeys => new FunctionType { ... },
        BuiltinNames.DictMethodValues => new FunctionType { ... },
        _ => null
    };
}

// Lines 339-355: ResolveDictMethodCall - hardcodes get() arity-dependent return type
private static SemanticType? ResolveDictMethodCall(
    GenericType dictType, string methodName, int argCount)
{
    if (methodName == BuiltinNames.DictMethodGet)
    {
        return argCount switch
        {
            1 => new OptionalType { UnderlyingType = valueType },
            2 => valueType,
            _ => null
        };
    }
    return null;
}
```

**Why this should be generalized:** Any type with overloaded methods could benefit from arity-dependent return type resolution. This pattern will need to be replicated for `list.pop()` (0 or 1 args), `str.split()` (0 to 2 args), `dict.pop()` (1 or 2 args), etc. A general mechanism should handle this.

## Warnings

### W1. Covariance Hardcoded to list and set Only

**File:** `TypeChecker.Utilities.cs:188`

```csharp
if (sourceGeneric.Name == "list" || sourceGeneric.Name == "set")
{
    return IsAssignable(sourceGeneric.TypeArguments[0], targetGeneric.TypeArguments[0]);
}
```

Covariance is hardcoded for only `list` and `set`. This uses raw string literals (`"list"`, `"set"`) rather than `BuiltinNames` constants (inconsistent with the rest of the file). More importantly, it doesn't consider:
- User-defined generic types that might be covariant
- Other collection-like generics (e.g., `Iterator`, future types)
- The fact that dict should NOT be covariant in its key type (correct to omit, but the reasoning isn't documented)

**Severity:** Warning â€” maintenance burden and incomplete generalization.

### W2. Parallel Hardcoded Switches in OperatorValidator and ProtocolValidator

**Files:** `OperatorValidator.cs:420-425`, `ProtocolValidator.cs:325-337`

Both validators maintain separate switch statements that enumerate which dunders each built-in generic type supports:

```csharp
// OperatorValidator.cs:420-425
generic.Name switch
{
    BuiltinNames.List => dunderName is DunderNames.Add or DunderNames.Mul or ...,
    BuiltinNames.Tuple => dunderName is DunderNames.Add or DunderNames.Mul or ...,
    BuiltinNames.Set => dunderName is DunderNames.Or or DunderNames.And or ...,
    BuiltinNames.Dict => dunderName is DunderNames.Or or DunderNames.Eq or ...,
    _ => ...
};

// ProtocolValidator.cs:325-337
generic.Name switch
{
    BuiltinNames.List => dunderName is DunderNames.Len or DunderNames.Iter or ...,
    BuiltinNames.Dict => dunderName is DunderNames.Len or DunderNames.Iter or ...,
    BuiltinNames.Set => dunderName is DunderNames.Len or DunderNames.Iter or ...,
    BuiltinNames.Tuple => dunderName is DunderNames.Len or DunderNames.Iter or ...,
    _ => false
};
```

These are parallel data tables masquerading as code. If a new built-in type is added, or a new operator/protocol is added to an existing type, both validators must be updated independently â€” a classic maintenance hazard.

**Proposed fix:** These should be data-driven â€” either part of a `BuiltinMethodRegistry` or defined as metadata on the `TypeSymbol` registered in `BuiltinRegistry`.

### W3. Builtin Function Return Type Inference Hardcoded in TypeChecker

**File:** `TypeChecker.Expressions.Access.cs:725-774`

The TypeChecker has a chain of `if` statements for special-casing the return types of builtin functions:

```csharp
if (id.Name == BuiltinNames.Len && argTypes.Count == 1) { ... }
if (id.Name == BuiltinNames.Hash && argTypes.Count == 1) { ... }
if (id.Name == BuiltinNames.Reversed && argTypes.Count == 1) { ... }
if (id.Name == BuiltinNames.Sorted && argTypes.Count >= 1) { ... }
if (id.Name == BuiltinNames.Min && argTypes.Count == 1) { ... }
if (id.Name == BuiltinNames.Max && argTypes.Count == 1) { ... }
```

Each builtin function gets its own `if` block with custom return type inference. The comment at line 71 of `BuiltinRegistry.cs` acknowledges this: *"Type inference for their return types is handled by TypeChecker special cases."*

**Why this should be generalized:** This is a classic case of behavior that should be declarative. Each builtin could declare a return type inference strategy (e.g., "returns the element type of the first iterable argument") rather than requiring imperative code in the TypeChecker.

### W4. Element Type Extraction Duplicated Across Files

**Files:** `TypeChecker.Utilities.cs:314-340`, `TypeInferenceService.cs:596-657`, `TypeUtils.cs:126-136`

Element type extraction from collection types is implemented in three separate places with slightly different logic:

1. **TypeChecker.Utilities.cs:314-340** â€” `ExtractElementType()`: Uses raw string `"list"`, `"set"`, `"dict"` comparisons
2. **TypeInferenceService.cs:596-657** â€” `InferIterableElementType()`: Uses `BuiltinNames` constants, handles dict views, CLR types, user-defined __iter__
3. **TypeUtils.cs:126-136** â€” `GetElementType()`: Uses `BuiltinNames` constants, returns value type for dict

These have subtly different semantics (dict returns key in one, value in another) and different levels of completeness.

### W5. Constructor Type Arg Inference Hardcoded Per Collection Type

**File:** `TypeChecker.Expressions.Access.cs:840-850`

When inferring type arguments for generic constructors with a single iterable argument, the code special-cases list, set, and dict:

```csharp
if (typeSymbol.Name is BuiltinNames.List or BuiltinNames.Set
    && typeSymbol.TypeParameters.Count == 1)
{
    typeArgs = new List<SemanticType> { elementType };
}
else if (typeSymbol.Name == BuiltinNames.Dict
         && typeSymbol.TypeParameters.Count == 2
         && elementType is TupleType tt && tt.ElementTypes.Count == 2)
{
    typeArgs = new List<SemanticType> { tt.ElementTypes[0], tt.ElementTypes[1] };
}
```

This should work for any generic type that accepts an iterable in its constructor, not just the three built-in collections.

### W6. BuiltinNames Has Dict-Specific Method Constants But Not For Other Types

**File:** `BuiltinNames.cs:66-75`

`BuiltinNames` has constants for dict methods (`DictMethodItems`, `DictMethodKeys`, `DictMethodValues`, `DictMethodGet`) and dict view types, but no equivalent constants for list methods (`append`, `extend`, `insert`, `pop`, `sort`, `reverse`) or set methods (`add`, `discard`, `union`, `intersection`, `difference`).

This asymmetry means dict is partially centralized while list and set method names are either hardcoded as raw strings elsewhere or handled entirely by CLR discovery at codegen time. The inconsistency makes it unclear what the intended architecture is.

### W7. Type-Specific Literal Generation in RoslynEmitter

**File:** `RoslynEmitter.Expressions.Literals.cs:28-155`

`GenerateListLiteral()`, `GenerateDictLiteral()`, and `GenerateSetLiteral()` each have nearly identical structure for resolving element types:
1. Check `_targetTypeContext.Name == "list"/"dict"/"set"` (raw string comparison against AST type annotation name)
2. Fall back to SemanticInfo generic type check
3. Fall back to element inference

The type resolution pattern is duplicated three times with only the collection kind varying. This could be a single method parameterized by the expected generic type name and the C# target type.

### W8. Imperative Comprehension Generation Parameterized by String

**File:** `RoslynEmitter.Expressions.Literals.cs:321-389`

`GenerateImperativeComprehension()` takes a `string collectionKind` parameter ("list", "set", or "dict") and branches on it with `if (collectionKind == "dict")` and `collectionKind == "list"`. This is a stringly-typed API within the compiler itself.

## Opportunities

### O1. DefaultParameterValidator Uses Raw Type Name Strings

**File:** `DefaultParameterValidator.cs:154-160`

```csharp
FunctionCall call when call.Function is Identifier id && id.Name == "set" => true,
FunctionCall call when call.Function is Identifier id && id.Name == "list" => true,
FunctionCall call when call.Function is Identifier id && id.Name == "dict" => true,
```

Uses raw strings instead of `BuiltinNames` constants. Minor inconsistency.

### O2. TypeUtils Provides Good Centralization But Is Underused

**File:** `TypeUtils.cs`

`TypeUtils` has well-defined helpers (`IsList()`, `IsDict()`, `IsSet()`, `IsCollection()`, `GetElementType()`, `GetKeyType()`) that centralize type checks using `BuiltinNames` constants. However, many call sites in TypeChecker and TypeInferenceService do inline `is GenericType { Name: BuiltinNames.Dict }` checks instead of using `TypeUtils.IsDict()`.

### O3. BuiltinNames Uses Inconsistent Naming Conventions

**File:** `BuiltinNames.cs`

Some constants use Sharpy names (`List = "list"`, `Dict = "dict"`), while view types use C# PascalCase names (`DictItemsView = "DictItemsView"`). The view types are synthetic names that don't exist in Python â€” they're internal compiler artifacts that should perhaps be in a separate constants class.
