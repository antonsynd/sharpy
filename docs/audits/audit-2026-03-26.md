# Sharpy Compiler Health Audit — 2026-03-26

## Executive Summary

The Sharpy compiler is in **excellent health** across all seven audited dimensions. All 8,923 tests pass (0 failures, 1 planned skip). The codebase demonstrates strong architectural discipline: clean pipeline layering (Parser → Semantic → CodeGen) with no circular dependencies, strict C# 9.0 compliance for both Sharpy.Core and generated code, comprehensive diagnostic coverage (247/248 codes active), and a production-ready LSP server with 26/26 handlers implemented.

Key strengths include: 100% TODO/FIXME issue tracking compliance (all 12 deferred items reference GitHub issues), robust concurrency patterns in the LSP server, full .NET runtime compatibility with proper polyfills, and a balanced test suite spanning 5,273 unit tests, 1,559 file-based integration fixtures, 55 C# snapshots, and 1,808 stdlib tests.

The main areas for improvement are minor: validation logic leaking into TypeChecker.Utilities (should migrate to ValidationPipeline), `AddError` method duplicated across 6 semantic classes, and the LSP lacking incremental project-level recompilation. The compiler is ready for parallel per-file compilation but not yet for REPL or formatter tooling (due to architectural choices around trivia preservation and module-bound parsing).

## Critical Findings

**None.** No critical issues identified across any audit dimension.

## Warnings

1. **Validation logic in TypeChecker.Utilities** — `ValidateConstructorOverloads`, `ValidateStructRules`, `ValidateEnumRules` are self-contained structural checks that don't require active inference state, matching the profile of ValidationPipeline validators. Blurs the TypeChecker/ValidationPipeline responsibility boundary.

2. **`AddError` duplicated in 6 semantic classes** — ImportResolver, TypeResolver, NameResolver, ModuleLoader, ModuleRegistry, and TypeChecker each define nearly identical private `AddError` methods. SemanticValidatorBase solved this for validators but the main semantic classes lack a shared base.

3. **`TypeChecker.Expressions.Access.Calls.cs` at 1,602 lines is a growth risk** — Handles call resolution, overload resolution, lambda calls, argument validation, builtin overloads, module function resolution, and IIFE lambdas. A future `OverloadResolver` extraction would improve testability.

4. **Backward-compatibility forwarding in Compiler.cs** — Lines 572–614 contain 5 `internal static` methods that only delegate to `CompilerInvariants`. If no external tests reference them directly, they can be removed.

5. **`SelfType` undocumented** — A 16th concrete `SemanticType` subtype (12 usages) not listed in CLAUDE.md or architecture docs. Either a placeholder to remove or docs need updating.

6. **LSP project reanalysis latency** — `OnDocumentChangedAsync` re-runs full ProjectCompiler with no incremental support. For large projects, could cause handler blocking. Mitigated by background Task and cached fallback.

## Opportunities

1. **Split `RoslynEmitter.Expressions.Literals.cs`** (1,405 lines) — Comprehensions (~477 lines) could be extracted to a dedicated partial file.

2. **Consolidate `SymbolSerializer` type encoding** — Maintains parallel `SerializeType`/`ResolveTypeFromId` representation; new SemanticType subtypes require updates in two places.

3. **Expand comprehension test fixtures** — Only 3 fixtures; recommend 8–10 for edge case coverage.

4. **LSP caching opportunities** — Semantic tokens and CodeLens could cache by AST fingerprint / document version.

5. **Rename `CodeGen/TypeMapper.cs`** to `SemanticTypeMapper.cs` or `SyntaxTypeMapper.cs` to disambiguate from `Discovery/ClrTypeMapper.cs`.
