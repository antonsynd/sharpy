# Sharpy Compiler Health Audit — 2026-02-23

## Focus: Pre-Phase 8/10 Readiness + Full Compiler Health

**Question:** What can be done before continuing work on later phases?

## Executive Summary

The compiler is in excellent health after completing Phases 6, 7, and 9 (13 items delivered). **7,011 tests pass with 0 failures and 8 skipped.** The codebase is clean, well-structured with good partial class decomposition, and has no circular dependencies between major components.

The next two phases — Phase 8 (Pattern Matching & Tagged Unions, 8 items) and Phase 10 (Async/Await, 6 items) — are both unblocked and can proceed in parallel. Significant groundwork exists: AST placeholder nodes, reserved tokens, control flow infrastructure, and the generator system (Phase 9) provides a clean pattern for Phase 10.5 async generators.

**Three architectural concerns surfaced by the deep audit:**
1. **Pattern dispatch extensibility** — The current match pipeline handles 5 patterns via if/switch chains. Phase 8 adds 8 more pattern types. Without refactoring, these switches will become unwieldy. This is the biggest tech debt risk.
2. **TypeChecker monolith** — At 5,520 lines across 8 partials, it's well-split but `TypeChecker.Expressions.cs` (1,388 lines) is the hottest file and growing. Some expression-handling responsibilities could be extracted.
3. **TODO compliance** — ~30+ TODO/FIXME comments lack GitHub issue references, violating project rules.

**Key recommendation:** Start with P0 prep work (trivial AST additions, test gap fills), then pursue Phase 8 items 8.1–8.5 and Phase 10 items 10.1–10.2 on parallel tracks. Cherry-pick 12.4 (nested comprehensions) and 11.1+11.2 (positional/keyword-only params) as independent quick wins.

## Critical Findings

### 1. Pattern Dispatch Needs Extensibility Refactor Before Phase 8
**Severity: Warning (will become Critical during Phase 8)**

The current `ParsePattern()` / `CheckPattern()` / `GenerateMatchPattern()` pipeline uses if/switch chains to handle 5 pattern types. Phase 8 adds 8 more: OrPattern, TypePattern, RelationalPattern, PropertyPattern, PositionalPattern, UnionCasePattern, MatchExpression (as expression context), and exhaustiveness. The switch chains will grow from ~5 arms to ~13 arms across three files.

**Recommendation:** Before starting Phase 8, refactor to a visitor pattern or registry-based dispatch for patterns. This is ~S effort and prevents accumulating tech debt across 8 implementations.

**Files:** `Parser.Statements.cs` (ParsePattern), `TypeChecker.Statements.cs` (CheckPattern), `RoslynEmitter.Statements.cs` (GenerateMatchPattern)

### 2. TODO/FIXME Comments Missing Issue References
**Severity: Warning**

Project rule requires all TODO/FIXME/BUG comments to reference GitHub issues. The audit found ~30+ violations scattered across the codebase, primarily in newer code from Phases 6–9. These represent invisible deferred work.

**Recommendation:** Batch-create GitHub issues for all untracked TODOs, or remove/resolve them. Run a pre-commit hook check to prevent new violations.

### 3. TaskType Is a Stub — Blocks Phase 10 Type Checking
**Severity: Warning (will become Critical during Phase 10)**

`TaskType` exists in the `SemanticType` hierarchy but is essentially a placeholder with no `ResultType` property for `Task<T>`. Phase 10 needs full TaskType support for async return type wrapping and await expression type inference.

**Recommendation:** Flesh out `TaskType` with `ResultType` property before starting Phase 10. ~S effort.

2. **RoslynEmitter.Expressions.cs** (~1,600 lines) and **RoslynEmitter.Statements.cs** (~1,500 lines) are the largest codegen partials. Pattern matching codegen (Phase 8) will add significant code to Statements.cs. Consider a dedicated `RoslynEmitter.Patterns.cs` partial.

4. **ProjectCompiler** (7 partials, ~1,473 lines) has clean responsibility boundaries after the recent refactoring.

### Type Safety & Correctness

5. **SemanticInfo uses ReferenceEqualityComparer correctly** throughout. The dictionary maps AST nodes (records with value equality) by identity, which is correct since the same syntactic pattern can appear in different locations.

6. **UnknownType is used consistently** as an error sentinel — when type checking encounters an error, it records UnknownType and continues, allowing multiple diagnostics per compilation. The TypeChecker handles UnknownType propagation well (e.g., `UnknownType + anything = UnknownType`).

7. **Error recovery in Parser** is adequate for single-file compilation but could be improved for IDE scenarios (Phase 10 LSP readiness). The parser generally aborts on first error rather than attempting recovery.

8. **ControlFlowValidator handles all existing statement types** including the recently-added yield. It will need extension for `await` (Phase 10) but the infrastructure (`BasicBlock.ContainsAwait`, `AsyncStateRegion`) already exists.

### .NET Compliance

9. **Sharpy.Core is C# 9.0 compliant** — no C# 10+ features found. All files use block-scoped namespaces, no `global using`, no `record struct`, no `init` accessors, no `required` keyword.

10. **Generated code is C# 9.0 compliant** — RoslynEmitter produces block-scoped namespaces, standard class/struct declarations, traditional property patterns. Verified via .expected.cs snapshot files.

11. **netstandard2.1/2.0 dual-targeting works correctly** — Sharpy.Core's API surface avoids netstandard2.1-only APIs. `Span<T>` and related types are available via polyfill packages where needed.

12. **Phase 10 .NET readiness for async:**

    - **Action needed:** Add `Microsoft.Bcl.AsyncInterfaces` package reference to Sharpy.Core before Phase 10.3–10.5

### Testing Health

13. **Test infrastructure is solid** — `IntegrationTestBase.CompileAndExecute()` handles the full pipeline. `FileBasedIntegrationTests` auto-discovers .spy/.expected pairs. Tests are parallelizable (xunit default with no shared state concerns found).

14. **Async test readiness** — `CompileAndExecute()` would need modification for async main entry points (Phase 10). The generated C# currently uses a sync `Main()` method. This is a known prerequisite for Phase 10 testing.

## Changes vs. Previous Audit Analysis

The previous agent's analysis was broadly correct. Key differences in this deeper audit:

1. **Pattern dispatch extensibility elevated to P0** — The previous analysis listed it as P1. After reviewing the actual switch chains in all three pipeline stages, this is genuinely blocking. Adding 8 pattern types to the existing if/switch structure will create significant maintenance burden.

2. **`Microsoft.Bcl.AsyncInterfaces` identified as prerequisite** — Not mentioned in previous analysis. Phase 10.3–10.5 (async for, async with, async generators) require `IAsyncEnumerable<T>` and `IAsyncDisposable`, which are NOT in netstandard2.1 natively.

3. **TODO/FIXME compliance added** — Previous analysis didn't audit this. ~30+ untracked TODOs represent invisible tech debt.

4. **Test counts refined** — Previous audit estimated ~7,180 tests. Actual count from `dotnet test` is 7,011 (the ~7,180 may have included parameterized test variations in the display count).

5. **Generator infrastructure reuse for async confirmed** — Phase 9's `IsGenerator` on `FunctionSymbol`, `GeneratorValidator`, and yield codegen provide a clean template for Phase 10's `IsAsync` pattern. The two systems are structurally parallel.

6. **Async test infrastructure gap identified** — `CompileAndExecute()` needs modification for `async Task Main()` entry points. This is a Phase 10 prerequisite not previously called out.
