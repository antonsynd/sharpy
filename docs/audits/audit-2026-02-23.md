# Sharpy Compiler Health Audit — 2026-02-23

## Focus: Pre-Phase 8/10 Readiness Assessment

**Question:** What can be done before continuing work on later phases?

## Executive Summary

The compiler is in excellent health after completing Phases 6, 7, and 9 (13 items delivered). **7,180 tests pass with 0 failures and 8 skipped.** The next two phases — Phase 8 (Pattern Matching & Tagged Unions, 8 items) and Phase 10 (Async/Await, 6 items) — are both unblocked and can proceed in parallel on independent tracks.

Significant groundwork already exists for both phases: AST placeholder nodes, reserved tokens, and control flow infrastructure. However, the existing match statement implementation is minimal (5 basic patterns only), and the pattern dispatch system will need extensibility improvements before the 8 new pattern types can be added cleanly.

**Key recommendation:** Start with Phase 8 items 8.1–8.5 (pattern completions, independent of tagged unions) and Phase 10 items 10.1–10.2 (async def + await, the foundation). Several small Phase 11/12 items can be cherry-picked in parallel as they're fully independent.

### Current Match Implementation Assessment

The existing match statement works with a `ParsePattern()` → `CheckPattern()` → `GenerateMatchPattern()` pipeline across Parser, TypeChecker, and RoslynEmitter respectively. This pipeline supports:
- **Literal patterns**: `case 1:`, `case "hello":`, `case True:`
- **Wildcard pattern**: `case _:`
- **Binding pattern**: `case x:` (captures value)
- **Tuple patterns**: `case (1, 2):`
- **MemberAccess patterns**: `case Color.Red:`
- **Guard clauses**: `case x if x > 0:`

### Preparatory Work for Phase 8

1. **Make pattern dispatch extensible** — The current `ParsePattern()`, `CheckPattern()`, `GenerateMatchPattern()` functions handle 5 cases with if/switch chains. Before adding 8 more pattern types, consider a visitor or registry pattern to keep the switch from becoming unwieldy.

2. **Add `RelationalPattern` AST node** — This is the only missing AST node for items 8.1–8.5. Adding it early (with stub parser/semantic/codegen) creates a clean surface for implementation.

3. **Add `PropertyPattern` / `PositionalPattern` AST nodes** — Needed for 8.5. Could be a single `StructuredPattern` record with named/positional variants.

4. **Study C# switch expression emission** — Match expression (8.1) lowers to C# switch expressions, which is a different Roslyn surface than the existing match statement (which lowers to C# `switch` statement). Understanding the `SyntaxFactory.SwitchExpression()` API early is valuable.

### Preparatory Work for Phase 10

1. **Add `IsAsync` property to `FunctionDef`** — This is the single most impactful prep: a boolean flag on the AST record. Without it, nothing else in Phase 10 can start. This is trivial to add.

2. **Study `Task<T>` / `ValueTask<T>` type mapping** — The TypeMapper needs to know about `Task<T>` and `IAsyncEnumerable<T>`. Planning this mapping early prevents type system surprises.

3. **Verify control flow async infrastructure** — `AsyncStateRegion` and `IdentifyAsyncRegions()` exist but are disconnected. A quick review of whether they're usable or need redesign would save time during Phase 10.

4. **Add `TaskType` support to type system** — `TaskType` exists in the `SemanticType` hierarchy as a placeholder. Fleshing it out (with `ResultType` property for `Task<T>`) enables type checking work.
