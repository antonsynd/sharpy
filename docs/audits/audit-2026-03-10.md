## Executive Summary

The Sharpy compiler is in **excellent health**. All 8,746 of 8,747 tests pass (1 intentional skip), .NET compliance is 100%, and the codebase maintains strong discipline: zero unused diagnostic codes, 100% TODO-to-issue compliance, and clean layering at the namespace level.

The most significant architectural risk is **duplicated compilation pipeline logic** between `Compiler.cs` (single-file) and `ProjectCompiler.Phases.cs` (multi-file), creating a dual-maintenance burden where pipeline changes must be applied in two places. A non-deterministic validator ordering at `Order 500` (ProtocolValidator/OperatorValidator) should also be fixed promptly.

Testing is strong overall (4,117 logical tests, 69% positive / 25% negative), but codegen snapshot tests (55) and warning tests (47) are sparse relative to the codebase's maturity. Future tooling readiness is mixed: formatter and debugger support can ship immediately, LSP and parallel build need 2-3 weeks of thread-safety work, and REPL is deferred to v0.3.

## Critical Findings

### 1. Duplicated Compilation Pipeline (Architecture)
Both `Compiler.cs` and `ProjectCompiler.Phases.cs` independently instantiate the full semantic pipeline (NameResolver, TypeResolver, TypeChecker). Any pipeline change must be applied in two places. **Fix**: Extract a shared `FileCompilationPipeline` that both orchestrators delegate to.

### 2. Non-Deterministic Validator Ordering (Architecture)
`ProtocolValidator` and `OperatorValidator` both declare `Order = 500`. `List<T>.Sort()` is not stable in .NET, so execution order is non-deterministic across runs/platforms. **Fix**: Give one `Order = 501`.

### 3. `SemanticAnalysisException` Misplaced (Architecture)
Defined at the bottom of `TypeChecker.Utilities.cs` but used across `Compiler.cs` and `ProjectCompiler.Phases.cs`. **Fix**: Move to `Diagnostics/` or its own file.

## Detailed Findings

### Architecture & Modularity

**Critical (Must Fix)**

1. **Duplicated Compilation Pipeline** — `Compiler.cs` and `ProjectCompiler.Phases.cs` both independently instantiate NameResolver, TypeResolver, TypeChecker. `ProjectCompiler.Phases.cs:254` even has a comment noting it mirrors "single-file Compiler.Compile() behavior." Extract a shared `FileCompilationPipeline`.

2. **Non-Deterministic Validator Ordering** — `ProtocolValidator` and `OperatorValidator` both at `Order = 500`. `List<T>.Sort()` is unstable. Additionally, `ProtocolValidator.cs:308` contains self-documented duplication: *"This duplicates logic from ProtocolValidator for post-pass validation."*

3. **`SemanticAnalysisException` Misplaced** — Defined at bottom of `TypeChecker.Utilities.cs:1180`, used in both `Compiler.cs:317` and `ProjectCompiler.Phases.cs:375`.

**Warning (Should Fix)**

4. **Bidirectional `Discovery` <-> `Semantic` Coupling** — `Semantic.Registry` uses `Discovery` while `Discovery` uses `Semantic` + `Semantic.Registry`. Makes it impossible to extract `Discovery` into its own assembly.

5. **`AstDumper` Does Not Use `AstVisitor`** — `AstDumper.cs` (900 lines) uses a large `switch` statement, duplicating node-type enumeration that `AstVisitor.cs` (815 lines) already provides.

6. **`CompilationResultBuilder` Nested in `Compiler.cs`** — ~200 lines embedded as nested class at line 746. Should be extracted to `Model/`.

7. **`ImportResolver.cs` Overloaded** — 1,051 lines handling loading, caching, resolution, re-export symbol synthesis, and .NET namespace resolution. Boundary with `ModuleLoader.cs` (681 lines) is unclear.

**Opportunity (Nice to Have)**

8. **`StringExtensions.cs`** (1,565 lines) — Could split into partial files by category (search, case, padding, predicates), consistent with `Partial.*` pattern.

9. **`DiagnosticExplanations.cs`** (1,467 lines, 244 `Add()` calls) — Effectively a data file; consider embedded JSON resource.

10. **`TypeChecker.Utilities.cs`** (1,183 lines) — Catch-all containing type compatibility, validation, error recovery, and event resolution. Could split into 4 partial files.
