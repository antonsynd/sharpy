# Sharpy Compiler Health Audit — 2026-02-15

## Executive Summary

The Sharpy compiler is in **excellent health**. All **6,978 tests pass** (0 failures) across 6,083 compiler tests and 906 Sharpy.Core tests, with only 11 programmatic tests skipped. The codebase demonstrates strong engineering discipline: all 5 TODO/FIXME comments reference GitHub issues (100% compliance), the test-to-code ratio is healthy across all components, and the architecture cleanly separates concerns across lexer, parser, semantic analysis, validation, and code generation phases.

Key risks center on two correctness bugs tracked in GitHub issues (#168/#165 for `ContainsReturnStatement` shallow check, #167/#166 for cross-module comprehension reference identity) and one architectural violation (Semantic -> CodeGen reverse dependency via `CodeGenInfoComputer`). The most significant code health concern is that decorator name strings are scattered as raw literals across 4+ files with no constants class — a typo would be a silent bug.

The 9 open GitHub issues (excluding #108) are well-documented and properly triaged. The compiler has a strong foundation for continued development with 578 file-based test fixtures, 203 negative error tests, 30 warning tests, and 48 C# snapshot tests.

## Critical Findings

### 1. Semantic -> CodeGen Reverse Dependency
`CodeGenInfoComputer` (in `Semantic/`) imports `Sharpy.Compiler.CodeGen` to use `DunderMapping.ResolveCSharpName()`. This creates a reverse dependency from the semantic layer up into the code generation layer, violating the intended `Lexer -> Parser -> Semantic -> CodeGen` dependency flow.

**Fix:** Move `DunderMapping` (or its `ResolveCSharpName` method) to `Shared/` alongside `NameMangler` and `DunderNames`. It's a static lookup table, not tied to Roslyn emission.

### 2. Duplicate GitHub Issues: #168 and #165
Issues #168 and #165 describe the same bug (`ContainsReturnStatement` only checks top-level statements in try/else). One should be closed as a duplicate.

### 3. Cross-Module SemanticInfo Reference Identity Issue (#167/#166)
The `SemanticInfo` dictionary uses `ReferenceEqualityComparer` for AST nodes (which are records with value equality). When comprehension elements are resolved in cross-module contexts, the AST node identity may not match what was stored, causing `GetExpressionSemanticType()` to return `null`. The current workaround is correct but duplicated 3 times across list/set/dict comprehension generators.

### 4. `ContainsReturnStatement` Shallow Check (#168)
`ContainsReturnStatement()` in `RoslynEmitter.Statements.cs:1691` only checks immediate top-level statements, missing nested returns inside `if`/`else` within try/else bodies. This can cause CS0161 (not all code paths return a value) in the generated C#.

## Warnings

### 1. No `DecoratorNames` Constants Class (20+ raw string occurrences)
Decorator names like `"abstract"`, `"static"`, `"virtual"`, `"override"`, `"final"` are used as raw string literals across `NameResolver.cs` (8), `ModuleLoader.cs` (5), `TypeChecker.Definitions.cs` (3), `RoslynEmitter.ClassMembers.cs` (6). A typo in any of these strings would be a silent bug.

### 2. Incomplete `BuiltinNames` Constants Class
`BuiltinNames` only defines `Bool`, `Int`, `Float`, `Str`. Missing: `Long`, `Double`, `Float32`, `List`, `Dict`, `Set`, `Tuple`, `None`, `Void`, `Object`. Raw type name strings appear in `TypeResolver.cs`, `ModuleLoader.cs`, `TypeInferenceService.cs`, and `RoslynEmitter.Expressions.Literals.cs`.

### 3. Duplicated Compilation Pipeline Orchestration
Both `Compiler.cs` and `ProjectCompiler.cs` independently set up the same pipeline: `NameResolver -> ImportResolver -> InheritanceResolver -> TypeResolver -> ValidationPipelineFactory -> TypeChecker -> RoslynEmitter`. Bug fixes must be applied in two places.

### 4. `ProjectCompiler.cs` Is a 1,759-Line Monolith
Handles 5+ distinct responsibilities: parsing, incremental cache management, symbol restoration/validation, semantic analysis, code generation, and assembly compilation. Consider splitting into partial classes.

### 5. Duplicate Type Resolution Switch
`TypeResolver.cs` and `ModuleLoader.cs` have duplicate switch expressions for primitive type name -> SemanticType mapping.

### 6. `InterfaceConflictValidator.CollectExplicitGenericInterfaces` Is a Stub (#122)
The method body is empty. Low impact since Sharpy code rarely explicitly implements .NET generic interfaces.

### 7. 11 Skipped Programmatic Tests
- 6x `ParserEdgeCaseTests` (nested list comprehension, decorator args, kwargs, dict comprehension, lambda defaults, all parameter types)
- 3x `ImportResolverDependencyTests` (parallelizable groups, transitive imports, affected files)
- 1x `SemanticAnalyzerNegativeTests` (yield outside function)
- 1x `Phase0110IntegrationTests` (import same name from different packages)
- 1x file-based skip (`dunder_eq_iequatable_explicit.skip`)

### 8. Async Control Flow Analysis Is Stubbed (#105)
`ControlFlowAnalysis.cs` has placeholder code. `IdentifyAsyncRegions()` creates one region per await block without proper state splitting. Not a correctness risk since async is not yet user-facing.

### 9. 11 Diagnostic Codes Defined But Never Emitted
Reserved codes for future use or consolidation decisions. Includes: SPY0207 (DuplicateConstant), SPY0208 (DuplicateTypeAlias), SPY0221 (IncompatibleTypes), SPY0238 (InvalidComprehension), SPY0245 (InvalidSomeConstructor), SPY0246 (InvalidOkErrConstructor), SPY0249 (MissingParameterAnnotation), SPY0261 (MissingReturnType), SPY0286 (DuplicateClass), SPY0303 (ImportPrivateSymbol), SPY0500 (EmitError).

### 10. 27 Sharpy.Core Files With No Dedicated Tests
Key untested files include: `Slice.cs`, `Slice.Sized.cs` (critical for list/string ops), `Abs.cs`, `Bytes.cs`, `Chr.cs`, `Ord.cs`, `Sum.cs`, `Next.cs` (builtin functions), `IndexError.cs`, `KeyError.cs`, `StopIteration.cs` (exception types), `DictItemsView.cs`, `DictKeyView.cs`, `DictValuesView.cs` (dict views).

### 11. Two TypeMapper Classes With Same Name
`CodeGen/TypeMapper.cs` (829 lines, forward mapping) and `Discovery/TypeMapper.cs` (189 lines, reverse mapping) share the same class name in different namespaces. Consider renaming Discovery's to `ClrTypeMapper`.

## Opportunities

### 1. Generic Type Semantic Analysis (#112)
Type parameters work at the parsing and codegen level but aren't fully understood by the semantic analyzer.

### 2. Imperative Comprehension Codegen (#100)
Complex/nested comprehensions could benefit from imperative loop-based code generation.

### 3. Entry Point Enforcement (#116)
The `main()` function requirement for executable modules is documented in the spec but not enforced.

### 4. Extract Narrowing/Walrus State From RoslynEmitter
RoslynEmitter has 25+ mutable fields across 9,524 lines (11 partials). Consider extracting narrowing state into a `NarrowingTracker` and walrus state into a `WalrusHoistingContext`.

### 5. Split Large RoslynEmitter Partials
`Statements.cs` (1,696 lines) could split into `Statements.ControlFlow.cs`. `ClassMembers.cs` (1,672 lines) could split iterator protocol into `ClassMembers.Iterators.cs`.

### 6. TypeChecker.Utilities.cs Is a Grab-Bag (1,081 lines)
Contains 36 methods spanning type assignability, struct/enum validation, super handling, interface validation, LCA computation, and error recovery. Consider splitting into focused partials.
