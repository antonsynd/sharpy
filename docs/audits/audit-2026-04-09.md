# Sharpy Compiler Health Audit — 2026-04-09

## Executive Summary

The Sharpy compiler is in **strong health** across all audited dimensions. All 10,165 tests pass (7 skipped), the architecture is well-modularized with clean separation of concerns, and .NET compliance is solid. The codebase has grown to **104,453 lines** across **465 source files** (237 Compiler, 228 Core) with 1,662 test fixtures — a mature, well-tested project.

The main areas for attention are: (1) 2 files exceeding 2,000 lines (RoslynEmitter.Statements.cs at 2,369, TypeChecker.Statements.cs at 1,998) that are candidates for splitting, (2) ~25 diagnostic codes allocated for planned features but not yet emitted, (3) 11 skipped test fixtures, and (4) future tooling readiness (REPL and formatter) requires significant architectural investment. No critical production issues were found. TODO/FIXME compliance is at 100% — all 11 deferred-work comments reference GitHub issues.

The LSP server has excellent handler coverage (17 handlers, 7 refactoring providers) with comprehensive E2E tests. .NET compliance is strong — Sharpy.Core correctly targets netstandard2.0/2.1 with C# 9.0 language version, and no violations were found in generated code patterns.

## Critical Findings

**1. `Compiler.CompileInternal` is a 430-line god method** (`src/Sharpy.Compiler/Compiler.cs:131`)
Orchestrates all four compilation phases (Lex → Parse → Semantic → CodeGen) inline, including error propagation, cancellation checks, and metric gathering. `ProjectCompiler` already splits this into partial files (`.Phases.cs`, `.Initialization.cs`, etc.) — `Compiler.cs` has no equivalent decomposition. When the single-file path diverges from the project path, bugs go undetected. Should be extracted into a `FileCompilationOrchestrator` or aligned with `ProjectCompiler.Phases.cs`.

**2. `ImportResolver` uses setter injection for required state** (`src/Sharpy.Compiler/Semantic/ImportResolver.cs:51–84`)
`SetSemanticBinding()`, `SetCurrentModule()`, `SetDependencyRecorder()`, and `SetCancellationToken()` are public mutators called from three different call sites. This is temporal coupling: callers must remember to call setters in the right order before `ResolveAllImports`. Missing a setter causes silent misbehavior, not a compile-time error. Should be constructor parameters or passed directly to `ResolveAllImports`.

**3. Type-parameter substitution is duplicated across `TypeChecker.Utilities` and `GenericTypeInferenceService`** (`TypeChecker.Utilities.cs:386–525` vs `GenericTypeInferenceService.cs:182–340`)
`TypeChecker` has its own `SubstituteTypeParameters` (139 lines) while `GenericTypeInferenceService` has a parallel `Unify` hierarchy (158 lines). The TODO at `TypeChecker.Utilities.cs:448` acknowledges this (tracked in #414 but unresolved). Two diverging implementations create correctness risk: a fix in one path may not be applied to the other.

## Warnings

1. **2 files exceeding 2,000 lines** — `RoslynEmitter.Statements.cs` (2,369 lines) and `TypeChecker.Statements.cs` (1,998 lines) are the largest partials and candidates for splitting by statement kind.

2. **11 skipped test fixtures** — Test fixtures with `.skip` files represent known gaps. These should be periodically reviewed and re-enabled as features are implemented.

3. **~25 allocated but unused diagnostic codes** — Codes allocated for planned features (e.g., SPY0019, SPY0112, SPY0131–0133, SPY0221, SPY0238, SPY0245–0246, SPY0521) are defined but not yet emitted. Only SPY0521 is explicitly marked as reserved. The rest should be tracked against planned features.

## Opportunities

1. **LSP handler gaps** — CallHierarchy, TypeHierarchy, CodeLens, WorkspaceSymbol, LinkedEditingRange, and SelectionRange are not yet implemented.

2. **REPL/Formatter readiness** — The AST doesn't preserve whitespace/trivia, making formatter support a significant future investment. Statement-level compilation for REPL is not yet supported.

3. **Dead code cleanup** — Several potentially unused functions were identified that could be cleaned up.

#### Critical: Architectural Issues

1. **`Compiler.CompileInternal` god method** (430 lines at `Compiler.cs:131`) — Orchestrates all 4 phases inline. `ProjectCompiler` already has phase decomposition; single-file path doesn't. Risk: divergence between the two compilation paths.

2. **`ImportResolver` setter injection** (`ImportResolver.cs:51–84`) — 4 public `Set*()` mutators must be called in correct order from 3 call sites. Temporal coupling creates silent misbehavior risk. Should use constructor params or method params.

3. **Duplicated type substitution** (`TypeChecker.Utilities.cs:386–525` vs `GenericTypeInferenceService.cs:182–340`) — Two parallel implementations of type parameter substitution/unification (~300 lines total). Tracked in #414 but unresolved. Correctness risk from divergence.

#### Coupling Analysis

- **Parser → Semantic**: Clean one-way dependency via AST nodes
- **Semantic → CodeGen**: Clean via `CodeGenInfo` materialization on `Symbol`
- **CodeGen → Semantic**: Appropriate read-only access to `SemanticInfo` and `SymbolTable`
- No circular dependencies detected between major subsystems
- `RoslynEmitter` correctly uses `SyntaxFactory` exclusively — zero `Parse*` calls found

#### Separation of Concerns

- **TypeChecker vs ValidationPipeline**: Well-separated. TypeChecker handles type inference and checking; validators handle self-contained analyses. The 20+ validators each have a clear, focused responsibility with explicit ordering.
- **RoslynEmitter partials**: Boundaries are logical — Expressions, Statements, TypeDeclarations, ClassMembers (further split into Methods, Constructors, Properties, Dataclass, Iterators), Operators, Patterns, Comprehensions, Literals.

#### Warning: Files Needing Splitting

- `RoslynEmitter.Statements.cs` (2,369 lines) — split by statement kind (loops, control flow, definitions). Goal: <1,500 lines/file
- `TypeChecker.Statements.cs` (1,998 lines) — consider extracting validation logic
- `RoslynEmitter.Expressions.Access.cs` (1,911 lines) — split into sub-expression categories
- `TypeChecker.Expressions.Access.Calls.cs` (1,671 lines) — call resolution + overload handling in one file
- `ImportResolver.cs` (1,193 lines) — single class, not partial, with setter injection issues

#### Warning: Minor Issues

- `ValidationPipeline.AddValidator` re-sorts the list on every add (`ValidationPipeline.cs:33`) — O(N²) at startup. Should sort once after all validators added.
- `TypeSyntaxMapper` has two parallel mapping paths (`MapSemanticType` vs `MapType`) with cross-fallbacks. Boundary conditions documented only in comments.

#### Opportunity: Clean Layering Confirmed

Dependency graph is correctly layered: `Lexer → Parser → Semantic → CodeGen`. No circular dependencies. `Analysis.ControlFlow` consumed only by `ControlFlowValidator` (correct).

### Future Readiness

| Dimension | Rating | Summary |
|-----------|--------|---------|
| Parallel Build | **Partial** | No mutable statics, but core data structures use non-concurrent collections |
| REPL | **Not Ready** | No statement-level compilation; parser requires full module |
| Formatter | **Not Ready** | AST doesn't preserve whitespace/trivia |
| Debugger | **Partial** | TextSpan locations preserved in AST; no source-map generation |

#### Parallel Build Readiness — Partial

**Positive**:
- `CompilerServices` uses builder pattern — can instantiate multiple independent instances
- No mutable static fields found in the compiler
- `ProjectCompiler` supports incremental compilation with file-level granularity

**Blockers**:
- `SymbolTable` uses `Dictionary<>` (not `ConcurrentDictionary`)
- `SemanticInfo` uses `Dictionary<>` with `ReferenceEqualityComparer`
- These would need to be thread-safe for true parallel file compilation
- The sequential pipeline (Name Resolution → Import Resolution → Type Resolution → Type Checking) has inherent ordering constraints

#### REPL Readiness — Not Ready

- Parser requires a complete module structure (module-level statements)
- No support for evaluating individual expressions or statements
- No incremental semantic analysis at the statement level
- `ScopedTypeChecker` in LSP does partial re-analysis but is document-scoped, not statement-scoped

#### Formatter Readiness — Not Ready

- AST nodes are semantic records — they don't preserve original whitespace, comments, or formatting
- Parser discards trivia during token → AST conversion
- No roundtrip capability (parse → unparse produces semantically equivalent but not identical source)
- A formatter would need either trivia-preserving AST or a separate CST layer

#### Debugger Readiness — Partial

**Exists**:
- `TextSpan` on AST nodes preserves source locations (line, column, offset)
- Source positions flow through semantic analysis
- `ILocatable` interface provides uniform location access

**Missing**:
- No source-map generation in RoslynEmitter (Sharpy line → C# line mapping)
- No debug info emission (PDB mapping from generated C# back to .spy source)
- Would need `#line` directives or similar mechanism in generated C#
