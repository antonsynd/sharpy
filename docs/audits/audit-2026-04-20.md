# Sharpy Compiler Health Audit — 2026-04-20

## Executive Summary

The Sharpy compiler is in **excellent health** across all seven audit dimensions. The codebase demonstrates strong architectural discipline: no circular dependencies, clean pipeline boundaries (Lexer → Parser → Semantic → CodeGen), and 100% TODO/FIXME compliance with GitHub issue tracking.

**Test suite**: 10,351 tests pass (99.99% pass rate), with only 1 timing-sensitive performance test failing marginally (+9.5% over threshold). The 1,690 file-based integration fixtures provide extensive feature coverage across 61 categories.

**Key risks**: The primary concern is a 548-line `CheckFunctionCall` method in the type checker — the single largest complexity hotspot. Eight validators lack dedicated unit tests (notably ExhaustivenessValidator). CodeGen test coverage trails production code by ~25%. These are manageable issues, not blockers.

**.NET compliance is flawless**: C# 9.0 compliance in Sharpy.Core and generated code is verified across all 230 source files and 62 snapshot tests. No prohibited API usage detected. The LSP server is well-architected with 342 tests, sound thread safety, and 26 implemented handlers covering all major IDE features.

## Critical Findings

### CRITICAL-1: `CheckFunctionCall` — 548-line Method
**File:** `Semantic/TypeChecker.Expressions.Access.Calls.cs:14`

Single method handling tagged union constructors, IIFE lambdas, `type()` built-in, null-conditional calls, event invoke, `super().__init__()`, generic instantiation, overload resolution, constructor dispatch, and keyword arguments. 5 levels of nesting, effectively untestable as a unit.

**Suggested split:** Extract ~6 focused methods (`CheckTaggedUnionCall`, `CheckEventInvoke`, `CheckSuperInitCall`, etc.), reducing the dispatcher to ~100 lines.

### CRITICAL-2: 8 Validators Without Dedicated Unit Tests
Missing dedicated tests for: BodylessSyntaxValidator, ConstructorOverloadValidator, EnumRulesValidator, EventValidator, **ExhaustivenessValidator** (match statement exhaustiveness — correctness-critical), GeneratorValidator, InterfaceImplementationValidator, StructRulesValidator.

These validators may have implicit coverage via file-based integration tests, but lack targeted unit tests for edge cases.

## Warnings

### WARNING-1: CodeGen Under-Tested (0.88:1 ratio)
RoslynEmitter (15,256 lines across 21 partial files) has only 15,220 lines of test code. Missing coverage for expression codegen edge cases (lambdas, comprehensions), generic type codegen, operator overload emission, and protocol interface synthesis.

### WARNING-2: Duplicate Decorator Switch Pattern
**File:** `RoslynEmitter.TypeDeclarations.cs:185` and `:1002`

Two nearly-identical methods (`GenerateModifiersFromDecorators` and `GenerateTypeModifiersFromDecorators`) share 70+ duplicated lines of access-modifier switch logic.

### WARNING-3: `ImportResolver.cs` — Mixed Responsibilities (1,193 lines)
Handles five distinct concerns: spy module loading, .NET assembly discovery, package resolution, re-export symbol creation, and `import *` filtering. The .NET interop resolution (~200 lines) is a candidate for extraction into `NetModuleResolver`.

### WARNING-4: `TypeChecker.Definitions.cs` — Broad Scope (1,670 lines)
Covers class, struct, interface, enum, union, delegate, and function definitions with 27 diagnostic emissions. Could be split into `.Definitions.TypeDeclarations.cs` and `.Definitions.FunctionDeclarations.cs`.

### WARNING-5: Performance Test Threshold
`CachedDiscovery_FirstLoad_BuildsCacheWithinTime` — Expected <1500ms, actual 1642ms (+9.5%). Consider adjusting threshold for ARM64 Mac CI environment.

### WARNING-6: Project/Incremental Compilation Under-Tested (0.71:1 ratio)
IncrementalCompilationCache logic, dependency graph tracking, and cache invalidation scenarios have sparse test coverage.

### WARNING-7: 7 Unimplemented Diagnostic Codes
SPY0112, SPY0132, SPY0133, SPY0221, SPY0238, SPY0245, SPY0246 — all allocated placeholders for planned v0.2.x features. Not critical, but should be tracked.

### WARNING-8: LSP Position Converter Defensive Guards
`Math.Max(0, compilerLine - 1)` silently clamps negative line numbers instead of failing loudly. Could mask compiler bugs.

## Opportunities

### OPP-1: `DiagnosticExplanations.cs` — 1,589 Lines of Inline Data
Single method initializing ~180 dictionary entries. Could be split into per-range partials following DiagnosticCodes.cs structure.

### OPP-2: Events Mixed into Properties Partial File
`RoslynEmitter.ClassMembers.Properties.cs` (1,160 lines) handles both properties and events. Event generation (~200 lines) could move to `RoslynEmitter.ClassMembers.Events.cs`.

### OPP-3: `SymbolSerializer.cs` Hand-Rolled Type Parser
Contains a mini-lexer for type annotation strings (lines 663–723) that could drift from the real Lexer. Consider extracting into `TypeAnnotationStringParser`.

### OPP-4: `AstDumper.cs` — Source Generator Candidate
At 1,175 lines, this debug tool handles every AST node type manually. Could be auto-generated from AST record definitions.

### OPP-5: LSP Handler Direct Unit Tests
Handlers are thin wrappers tested indirectly via service-level and E2E tests. Adding direct unit tests would improve edge case coverage (null analysis, cancellation, malformed positions).

**Parallel build:** Infrastructure exists (`DependencyGraph.GetParallelizableGroups()`, `ThreadSafeAttribute`, thread-safe `DiagnosticBag` and `SemanticInfo`). Blocked by sequential semantic analysis (single shared SymbolTable/SemanticInfo).

**REPL:** No statement/expression-level parsing exposed. Parser only has `ParseModule()` as public API. Would require parser refactoring and per-statement compilation infrastructure.

**Formatter:** Re-indentation works via LSP. No roundtrip parsing (AST discards whitespace/comments). Full formatter would need trivia preservation.

**Debugger:** `AttachLineDirective()` support exists for `#line` directives. Missing DAP server and comprehensive line directive coverage audit.

**Incremental compilation:** Fully implemented with SHA-256 content hashing, dependency graph tracking, symbol serialization, and automatic cache invalidation. File-level granularity is appropriate.
