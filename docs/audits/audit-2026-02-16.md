# Sharpy Compiler Health Audit — 2026-02-16

## Executive Summary

The Sharpy compiler is in **excellent health**. All 7,008 non-skipped tests pass (100% pass rate), zero TODO/FIXME comments exist in production code, all 127 diagnostic codes are actively used, and .NET compliance is fully verified for C# 9.0 and netstandard2.0/2.1. The architecture is cleanly layered with zero circular dependencies.

The primary areas for improvement are: (1) duplicated compilation pipeline logic between `Compiler.cs` and `ProjectCompiler.cs`, with `ProjectCompiler` missing `CompilerInvariants` assertions; (2) C# reserved keywords list duplicated in 3 locations; (3) sparse unit-level test isolation for RoslynEmitter and TypeChecker partial files (mitigated by strong integration test coverage); and (4) Discovery and Project components having the lowest test-to-code ratios.

For future tooling, the compiler has solid LSP foundations (ISemanticQuery, SourceText/TextSpan) and partial parallel-build readiness (SemanticBinding is thread-safe, no mutable statics), but is not ready for REPL usage and needs work for formatter roundtripping.

## Critical Findings

**None.** No critical architectural, correctness, or compliance issues were found.

**Key concern: Pipeline duplication.** `Compiler.cs` (977 lines, single-file) and `ProjectCompiler.cs` (1,759 lines, multi-file) independently implement the same compilation pipeline. `Compiler.cs` calls `CompilerInvariants` assertions at each phase boundary; `ProjectCompiler.cs` does not. Extracting a shared `PipelineExecutor` would eliminate duplication and ensure uniform invariant checking.

**Dependency direction** is perfectly clean — no coupling from Parser->Semantic, Semantic->CodeGen, or any reverse direction. The Semantic/ directory is the largest at 66 files but is well-organized with Validation/ as a proper sub-directory.

### Type Safety & Correctness

**Overall: Sound.** All tests pass. Error recovery uses a disciplined three-layer defense (MarkErrorRecovery -> automatic propagation -> CompilerInvariants.WarnIfUnknownTypes). All diagnostic codes are actively used.

**Test results:** 7,008 passed, 0 failed, 11 skipped.

**Error recovery architecture:**
- `MarkErrorRecovery(expr)` tags intentional Unknown types
- Automatic propagation in `CheckExpression()` cascades error marks from sub-expressions
- `CompilerInvariants.WarnIfUnknownTypes()` catches any Unknown type not marked as error recovery (compiler bug detector)
- 65+ `return SemanticType.Unknown` sites all follow the pattern of emitting a diagnostic first

**SemanticInfo integrity:** Type recording is centralized through a single `SetExpressionType` call in `CheckExpression()`. All 6 dictionaries correctly use `ReferenceEqualityComparer.Instance`.

**11 skipped tests** represent deferred functionality: kwargs parsing, dict comprehension, nested list comprehension, decorator arguments, lambda default args, parallel import dependency tracking, cross-package imports, yield-outside-function validation.

### Future Tooling Readiness

| Dimension | Rating | Key Strength | Key Gap |
|-----------|--------|-------------|---------|
| **LSP** | Partial | `ISemanticQuery` interface, `SourceText`/`TextSpan`, `GetEffectiveType` | No position-to-node lookup, symbols lack `DeclarationSpan` |
| **Parallel Build** | Partial | `SemanticBinding` thread-safe, no mutable statics, per-file TypeChecker/Emitter | `SymbolTable`/`Scope` not thread-safe, no batch scheduler |
| **REPL** | Not Ready | N/A | No public statement parsing, no incremental semantic context |

| **Debugger** | Partial | `#line` directives on statements, `EmitLineDirectives` flag | No column mapping, no PDB/source-link config |

**Highest-ROI recommendations:**
1. Add `DeclarationSpan` and `FilePath` to base `Symbol` for LSP go-to-definition
2. Make `Scope._symbols` use `ConcurrentDictionary` for parallel builds
3. Expose `ParseStatement()`/`ParseExpression()` as public for REPL
4. Extend `TriviaKind` to include `Whitespace` for formatter
